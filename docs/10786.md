# 客户端路由的过去和未来

> 原文:[https://dev . to/react ive conf/客户端路由的过去和未来](https://dev.to/reactiveconf/past-and-future-of-client-side-routing)

我有机会在下个月的 [ReactiveConf](https://reactiveconf.com/) 上发言，届时我将谈论客户端应用程序路由。我已经尝试和研究路由近两年了，这一切都是从我开始慢慢从 Angular 过渡到 React 开始的。去年 6 月，第一届 React Europe 会议在巴黎举行。我在网上看过，在他的主题演讲中，克里斯托弗·切多(又名 Vjeux)谈到 React 中的路由器时说:

> 我们只见过一个玩家。有什么创新吗？（...)创新在几年前就已经发生了。这件事发生在 Ember 社区。

当时，我正在开发第一版的 [router5](https://router5.github.io) ，这是一个框架和客户端无关的路由解决方案。尝试理解直到最近是什么影响了路由并塑造了我们的“心智模型”，以及为什么我们认为路由在很大程度上是一个已解决的问题，这是一件有趣的事情。

我们在客户端应用程序中进行路由的方式受到了服务器端路由的影响。在服务器上，路由是某种无状态的模式匹配。对于给定的请求，在执行控制器方法或回调之后，匹配一个路由并返回一个页面。我们通过让路由器负责在单页面应用程序中呈现路由级别的组件:“route => view”或“route => component”，将这种“route => page”方法转化为客户端。客户端应用程序不是无状态的，并且处理状态转换(包括路由转换)，但是服务器处理路由的方式让我们隐藏了状态。

如果这个简短的介绍让您想了解更多关于路由的信息以及实现路由的新方法，那么不要忘记下个月的 [ReactiveConf](https://reactiveconf.com/) ！
