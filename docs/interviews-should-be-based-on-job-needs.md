# 面试应该基于工作需要

> 原文：<https://dev.to/pbeekums/interviews-should-be-based-on-job-needs>

我写过关于[为开发者进行采访的文章。人们太关注开发人员提高他们的“面试技巧”，而对面试官自己做得更好关注不够。面试官的一个问题是他们要求应聘者解决经典的计算机科学问题。](https://blog.professorbeekums.com/2016/03/better-interviews.html)

[![](img/85d844559ead2b14c6d1e97312532670.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--Hx5tTh5I--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://blog.professorbeekums.com/img/2017/3/binary.jpg)

我刚开始面试的时候就是这么做的。这很有意义，因为其他人都是这么做的。还有如果一个人的工作是写软件，那就要擅长算法！

在一家公司，我们雇佣了这位以这个标准来看非常出色的开发人员。他对我们问题的解决方案让所有其他候选人都大吃一惊。这甚至比我们自己提出的解决方案还要好得多。他之前在 3D 图形方面的工作也令人印象深刻。

你认为他的工作表现如何？

他并不可怕，但他的表现远低于我们的预期。他擅长算法，但是我们做的工作中很少需要复杂的算法。我们有小团队与十倍于我们规模的公司竞争。我们需要快速构建东西，并以更容易快速构建东西的方式设计我们的软件。我们需要我们的基础设施扩展到数百万用户。我们需要有广博知识的人:前端、后端、数据库、网络等等。

我们没有问任何关于我们需要的东西的面试问题。

这位开发人员在许多方面都是一位伟大的开发人员，但他的最佳技能没有被我们充分利用，我们没有从他身上获得出色的工作。这个经历告诉我，面试的问题必须反映出这个人实际要做的工作。这听起来应该是显而易见的，但大多数面试官都会犯和我们一样的错误。不同公司的开发人员职位的职责范围差别很大。

每当我加入一家新公司的招聘程序时，我都会根据这家公司正在做的工作提出问题。公司解决的任何功能或问题都是公平的游戏。我不必泄露任何专有信息。我只是展示用户在应用程序中看到的一个特性，并询问候选人他们将如何实现它。

因为这个问题是公司已经解决了的，任何能想出一个像样的解决方案的候选人都将能够做公司要求的工作。还有一个额外的好处是，与候选人讨论问题反映了我将如何与他们合作。这些讨论将类似于当我们最初不得不解决这个问题时，如果候选人是公司的一部分，我们将进行的讨论。如果应聘者在面试中很适合共事，那么他们在工作中也很适合共事。(注意:自从采用这个策略以来，我已经做了一次错误的判断)

在某些情况下，你可能会发现自己想要雇用一个知道一些你不知道的事情的人。如果你自己没有能力，你如何测试他们在这方面的能力？

雇用这个人的需要将源于你知道你将来必须解决的一个问题。在这一点上，你至少应该有某种解决方案，即使是一个糟糕的方案，万一你不能及时招聘，你也可以采用。在这种情况下，这个问题就成了完美的面试问题。如果一个候选人拿不出比你更好的解决方案，那么你就不需要他们。如果他们不能用你能理解的方式解释解决方案，那么你就不能和他们一起工作。在我的职业生涯中，我只在三个职位上这样做过，但这三次招聘都非常成功。

无论如何，这种面试策略并不完美。知道一个人是否适合的唯一完美方法是雇用他们，然后看看几个月后会发生什么。我们进行采访，因为那太贵了，不可行。但是，根据你需要应聘者实际做的工作来提问，会比根据你理论上的需求来提问得到更好的结果。
