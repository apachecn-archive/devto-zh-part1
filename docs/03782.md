不久前，甚至在 Leonid Suschev 和我在 mkdev.me 上发布专门化之前，我们还在讨论它们可能的内容。这些讨论令人筋疲力尽，充满渴望，甚至会让整个行业看起来令人厌恶。将它归类为知识是一件相当吃力不讨好的工作。对一些人来说，结果可能不令人满意，有些人甚至会因此感到愤怒。

以下是我们遇到的一些问题:

*   我们应该将 Ruby on Rails 开发区分为一个独立的专门化吗？
*   我们是否应该将一些特定的 JavaScript 框架区分为单独的专门化？
*   是否应该对厨师和木偶进行单独的专精？

就这样一直持续下去。在这里你可以找到所有讨论的结果:[mkdev.me/en/specializations](https://mkdev.me/en/specializations)。

有趣的是，Leonid 对编程和所有相关主题没有清晰的理解。这很好，就像这里一样，在 mkdev 上，我们这样教导人们:几乎没有任何知识，那些不知道做什么和选择什么，但非常想掌握诀窍的人。所以我不得不偶尔解释一些细节，全球方面和发展的魔力。

有一次，话题转到了 Django 和 Ruby on Rails 上。Leonid 试图理解两种 web 框架之间的根本区别，当你有特定的技术规范时，这种区别允许在一个和另一个之间进行选择。谈话突然转向:

Python 和 Ruby 的实际区别是什么？他们是截然不同的。完全不同的编程语言。

——这意味着 Rails 和 Django 拥有完全不同的应用创建能力。

——不，一样。

—那为什么 Yandex 是用 Python 写的，而不是 Ruby？

——它是用几十种语言编写的，包括 Ruby 和 Python。为什么不只用一种语言呢？为什么有的东西是用 Ruby 写的，有的东西是用 Python 写的？

说真的，为什么？为了回答这个问题，我可能会喋喋不休地说 Python 是数据科学任务的完美之选，而 Ruby 是小型脚本的最佳解决方案，这就是为什么您需要使用这个和那个。我还可以补充说，有时这些特定语言的速度不够快，所以 Yandex 可能也会使用 Go 来处理速度要求高的组件。对于任何其他联合创始人来说，这将是一个准确而令人满意的回答，或许对于任何其他精通数字的对话者来说也是如此。毕竟，这也回答了另一个 Leonid 提出的关于技术基本差异的问题:

—为什么这栋房子的墙有些是木头做的，有些是石头做的，有些是玻璃做的？为什么我们不把它们都做成木头的呢？

事实上，对它一无所知的人或刚入门的初级开发人员会认为选择一种编程语言或在两种框架之间选择取决于一些技术方面。当你只需要敲一个钉子的时候，这看起来就像你选择了一把锤子或者一把螺丝刀。当然需要锤子啦！但是有一天螺丝刀也可能会派上用场！这就是为什么典型的 Yandex 使用 Ruby、Python、C++和 C 以及许多其他东西。是这样吗？好吧。大概是吧。但这不是全部真相。

事实上，当你选择技术时，你可能会在旗舰 iPhone 和 Android 手机之间做出选择。至少，你选择这个或那个的原因与你选择技术的原因非常接近。

约翰在 2010 年买了他的第一部 iPhone，从那以后他就离不开它了。他用它打电话、拍照、发短信、上网都很方便。他没有理由改变它，因为苹果的事情进展顺利。几乎所有约翰的朋友都有 iPhones，除了一个固执的詹姆斯。他也在 2010 年买了第一部安卓手机。他用谷歌 Pixel 打电话、拍照、发短信、上网都很方便。

詹姆斯总是试图鼓励他的苹果粉丝朋友换成安卓，但缺乏论据。因为他的旗舰产品 Pixel 和 iPhone 7 没有本质区别。

詹姆斯不像他的邻居鲍勃，他也有一个机器人。鲍勃总是称 iPhone 用户是愚蠢的笨蛋，并热切地试图证明 Android 更好。没有人敢反驳他，但他的论点只对那些有类似心态的人来说就够了。

曾经的约翰(有 iPhone 的那个，记得吗？)在一家叫 WagnerCom 的公司找到了一份工作。在那里，他收到了一部公司的安卓手机。所以他必须用它来处理业务问题，用他的 iPhone 来处理个人事务。一段时间后，他习惯了 Android。现在他用 iPhone 和安卓手机一样方便。他同意詹姆斯的观点，认为 Android 在某些方面更好，但也知道在其他一些方面他更喜欢 iPhone。

约翰有一个儿子迈克尔，他迷上了与电子产品相关的网站。读了其中一些，他发现了这款手机使用了一个全新的实验平台 NonameOs。所以迈克尔破解了他的 iPhone 并安装了 NonameOS。它工作起来很糟糕，应用程序很少，但它基本上是一个新平台。他和其他 100 个人在留言板上混在一起，每个人都全心全意地使用 NonameOS。他们相信，正确平台的正确概念总有一天会胜出，NonameOS 会胜出，与 iPhone 和 Android 处于同一水平，然后甚至会超过它们。然后所有的人都可以用没有名字的手机打电话、拍照、发短信和上网。

别忘了詹姆斯的同事威廉。威廉急需一部双卡手机。这很重要，因为他买不起两部手机，但他肯定需要两个 sim 卡。他别无选择，只能买一些罕见的双 SIM 卡安卓手机。他没有选择，因为没有双卡 iPhones。

所以现在我们有约翰、詹姆斯、鲍勃、威廉以及成千上万甚至上百万的人，他们中的每一个人都倾向于某种特定的手机。这种选择可能是由社交圈(同事或朋友)和消费者营销强加的；或者你买它仅仅是因为你想鹤立鸡群。如此循环往复。但结果是我们每个人都可以用手机打电话、拍照、发短信、上网。实际上用什么来做没有太大的区别。

使用什么来编写 web 应用程序没有什么特别的区别。你可以使用 Django，你可以使用 Rails 甚至 Node.js。使用什么来自动化服务器也没有区别。厨师和木偶同时出现，用相似的方式解决相似的问题。如果你的新雇主突然使用 PostgreSQL，你可以用 MySQL 代替 PostgreSQL。当你重写应用的一些慢的部分时，你用的是比 Ruby 更快的东西，比如 Go、Scala 或者 Erlang。

技术栈的选择不是一个严肃的工程项目。这不是根据技术要求选择最好的仪器。这不是锤子和螺丝刀之间的选择。这是一个基于规格、历史原因、就业市场情况、个人偏好、专业知识、开发团队的习惯和兴趣、营销材料的质量、对营销的敏感性以及公司工作人员的宣传的选择。

脸书使用 PHP，因为很久以前一个年轻的学生扎克伯格知道它，而且用 PHP 写脸书对他来说更快。后来完全重写它就没有任何意义了。

另一个事实是，脸书使用厨师，因为有人从公司听到它比木偶。他们也使用 Ruby，就像他们使用 Chef 一样。

肯定还有 Go 隐藏在某个地方，因为一些 SRE 感到无聊，在他的下班时间，他试图使用它，并用它重写了一些内部服务。然后，他说服他的同事，这是一个伟大的想法。为什么要去？它被夸大了，它有许多新的工具，如果他想退出，这可能是一个很好的简历条目。

当然，当您出于某些特殊原因选择技术时，会出现一些明显的无法选择的情况。当我们创建完整的 iOS 游戏时，我们必须使用 Swift，就像 William 必须使用双卡手机一样。这样的限制在某种程度上启发了。你不必在两把相似的锤子中做出选择，因为有一大堆乐器。

在大多数情况下，没有什么可以阻止你在任何事情上使用同一种语言。我们使用 Ruby，因为它对我们来说既舒适又有趣。然后我们用 Go 重写了应用的一些部分，因为 Ruby 不够快，也因为 Go 太棒了。我们还可以包括用 Scala 编写的 Spark 应用。然后，让我们试着向一个非程序员解释，为什么我们不能首先使用 Scala，然后使用 Scala 编写整个应用程序及其所有服务。或者为什么我们不能把 Ruby 用在所有事情上，并投资服务器，让它们工作得更快。

—这太愚蠢了。那你为什么要用不同的语言编写应用程序的不同部分呢？
