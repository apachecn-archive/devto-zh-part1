# 软件系统会失败

> 原文:[https://dev.to/pbeekums/software-systems-will-fail](https://dev.to/pbeekums/software-systems-will-fail)

Gitlab 在二月份有一次非常公开的停机。大多数公司在服务中断时都会提供某种解释。这些通常是经过消毒的(或看起来经过消毒的),以使事情看起来比实际情况更好。Gitlab 反而提供了一份极其详细的事件报告，以及他们知道自己可以做得更好的所有事情。

我的一些朋友对这份报告深感不安。他们认为 Gitlab 已经表现出极度的无能，不能被他们的客户所信任。Gitlab 应该关闭商店，帮助他们的客户迁移到不同的服务。

我不同意这个立场。虽然 Gitlab 确实描述了许多导致其停机的糟糕实践，但仍有一些原谅的余地。中断有很多先决条件，但最终导致开发人员在错误的机器上执行操作。

听起来很可怕。的确如此。但是让我们看看这个作品大概是什么样子的:

[![command line terminal with a number of commands.](../Images/7fec1178393979c159e8a1f90e030180.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--aq1739T3--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://blog.professorbeekums.com/img/2017/3/terminal.png)

开发商可能打开了几扇窗户。所有文本。没有图像。没有足够的视觉线索来确保正确的操作正在进行。

如果你认为开发人员不会犯这样的错误:

*   你有没有不小心在一封关于你老板的刻薄邮件上点击了回复所有人并发给了你的老板？
*   你有没有不小心在脸书上加了前任好友？
*   你曾经给你的父母发送过给你朋友的聊天记录吗？
*   你曾经从你的电脑里删除过错误的文件吗？

人类会犯错。我们是不完美的生物。

大多数工程组织都采取措施来解决人为错误。最关键的操作应该有安全措施，以确保人们不会意外地做一些他们不想做的事情。批评一个团队没有这些保障措施是很容易的。

但是，您如何知道应该采取什么保护措施呢？你可以读到它们。然而，你曾经通过阅读成为某方面的专家吗？大概不会。专业知识需要大多数职业的经验。管理软件系统也不例外。知道什么安全措施是值得的，以及如何正确地实现它们需要经验。

所有的软件系统都会在某个时候崩溃。即使拥有世界上最好的开发人员的世界上最大的公司也有问题。Gmail 去年秋天出现了一次宕机。在早期，Twitter 每隔一天就会宕机。就连许多公司依赖其托管网络应用程序的亚马逊也经历了多次下跌。如果你读过[对他们最近一次停机的事后分析](https://aws.amazon.com/message/41926/)，他们使用的是既定的程序，所以即使有安全措施，他们也会遇到问题。停电实际上是由一个打字错误引起的。

一个错别字。

一个打字错误导致依赖亚马逊的大量互联网服务瘫痪。即使是像亚马逊这样规模庞大、技术娴熟的公司，也没有找到正确的安全措施组合来完美地防范人为错误。

虽然拥有一个 100%时间都在运行的 web 应用程序是件好事，但这根本不现实。与保持 web 应用程序正常运行同样重要的是，当应用程序停止运行时，您如何处理所发生的事情，以及您从这一经历中学到了什么。Gitlab 没有很好地处理他们的中断，但他们似乎从这次事件中学到了很多。每个人都需要一个获得学习经验的机会。个人需要，组织也需要。
