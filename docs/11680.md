我看到很多人都在担心自己编程生涯的下一步。或者甚至在学习新的框架或语言时担心下一步。今天，让我通过描述我的方法来减轻一些恐惧。

# [](#what-button-is-jump)跳转是什么按钮？

编程是一项了不起的工作。这也可能会带来极大的压力。很少有行业像编程，尤其是 web 编程，发展得如此迅速和彻底。五年前、十年前、十五年前你还在用 Rails 吗？五年前，Ruby web 开发人员仍然必须在 merb 和 Rails 2 之间做出选择。十年前，Rails 甚至还没有公开发布。15 年前，我认为这甚至不是 DHH 眼中的闪光点:我们都在洞穴里编写 Java 和 mod_perl，挥舞着钝器。

步伐没有放慢。Rails 4 很快就要发布了，如果没有其他的东西，你也可以继续关注其他很多框架来挖掘实现思路。由于移动平台的出现，现在有更多的浏览器可以瞄准，而不是更少。

即使在 Rails 框架中，也有很多东西需要跟上。前沿领域建模的最新进展。两个 Rails 堆栈，使用普通的旧 Ruby 对象，服务层，测试测试测试。

对于有经验的和新的程序员来说，甚至不知道从哪里开始。新闻节目可能是压力和焦虑的来源。那一点也不好玩。我希望你玩得开心，所以以下是对我有效的方法。让我们从头开始:我是如何学习的。

# [](#embrace-failure-as-a-means-to-learn)拥抱失败作为学习的手段

电子游戏玩家认为失败是进步的必要条件。不管电子游戏有多简单，在到达终点之前，你总会失败。在电子游戏中，失败是学习的主要机制。学习一个游戏如何“工作”以及它期望什么。同样的方法可以用来学习新的编程语言、框架或技术。

当我想尝试一项新技术，或者我在 Railscasts 上看到一个新的宝石，或者我想探索一个我听到的编程想法时，我只需要创建一个空白目录。通常我甚至不会给它起一个合适的名字，因为命名很难。然后我把写糟糕的代码作为我的使命。我不只是忽视或无视错误，我的目标是*犯*个错误。一旦我做错了，我就可以螺旋上升，真正理解发生了什么。

想了解 Rails？打破它。把它打碎。删除所有的 application.rb，看看会发生什么。创建从文件而不是数据库中提取数据的控制器。创建初始化变量并包含大量业务逻辑的视图。创建构建 HTML 和 JavaScript 的模型。用胶带和咖啡脚本把它们绑在一起。什么都不测试，什么都测试。构建只包含一个字段的模型。构建一个包含应用程序可能需要的所有数据的模型。

我发现，如果我想太多如何最好地接近甚至一个简单的学习实验，我会开始过度分析每一件小事:这个目录/应用程序名称在六个月后有意义吗？我用最好的方式编码 X 了吗？等等。这是个陷阱！

阿克巴上将:他知道什么时候是陷阱。

这是一个陷阱，因为你会浪费所有的时间去担心永远不会犯错，而不是真正学到有用的东西。我们程序员有一个有趣又神奇的领域。我们是一群非常聪明的人，他们激烈地争论着如何让我们的工作变得更简单。这太棒了。但也可能令人生畏。哦不！你在用工厂女孩，那太过时了！别担心。

想象一个格斗游戏。街头霸王 2，真人快打，灵魂之剑，什么的。有没有一个角色是*正确的*？是的，答案是春丽。但是任何角色都有可行的策略。

许多关于语言、框架和/或过程的激烈争论没有认识到这一点。这一次，他们在尝试为一家初创公司扮演沃尔多时经历了一段糟糕的时间，所以他们已经从那次经历中清醒过来，知道沃尔多很糟糕，任何人都不应该扮演他/她/它。这同样适用于 minitest/rspec 或延迟作业/resque/beanstalkd。所有这些工具都是为解决特定的问题而设计的。如果你不喜欢一个，那么这并不意味着没有人喜欢或没有有用的工作可以出来。

好吗？好的。

我也抽时间练习基本功。也就是说，编写一个我已经编写了几十次的 Rails 应用程序。也许我会改变事情或采取稍微不同的方法，但通常不会。我喜欢说，连乔丹都练过罚球。

我还会定期浏览我们现有的所有介绍 Ruby 基础知识的酷网站。比如红宝石僧侣，真的很酷。Benjamin Fleischer 在 github repo 中汇编了一份学习 Ruby 和一般编程的资源列表: [bf4/learning](https://github.com/bf4/learning)

浏览一个 Rails 教程就像玩一个视频游戏。你取得了进步，但布瑟仍然逍遥法外。嗯，铁路上的布瑟是什么？也许把一个复杂的应用程序拆分成几个专注的 Rails 引擎？我应该多想想这些类比。

# [](#new-games-yay-new-games)新游戏？耶！新游戏！

至少对我来说，当我试图跟上新闻时，对基本面的扎实掌握对我的自信大有帮助。一旦你学会了一些框架或语言(或打败了一些视频游戏)，你就会兴奋地得知一个新的框架或语言即将出现或在 github 上发布。例如，有一天下午，我玩流星玩得很开心。

您如何知道应该尝试哪种工具或方法？没关系，挑一个自己感兴趣的就好。你并没有在你的实验中购买终身股份，你只是在做实验。

你可以选择引入一些严谨性，用一个基本的项目来接近这个框架，遵循他们给定的教程，或者只是瞎搞，猜测事情是如何工作的。如果这是一个完全陌生的框架或语言，你可能不得不坚持使用教程(例如，第一次玩一个新的视频游戏类型)，但如果你在熟悉的领域(例如，你的一种语言的新版本)，那么探索一会儿会很有趣。

# [](#have-fun)玩得开心！

如果你玩得不开心，而你的工作是编程，那么你需要重新思考事情或者找一个更好的地方工作。我害怕另一只鞋掉下来的时候，世界上的其他人意识到我们程序员整天都在玩，并把它叫做工作。:-)
