# 开发人员如何让他们的工作变得更加困难

> 原文：<https://dev.to/pbeekums/how-developers-make-their-jobs-harder-3k1f>

软件开发很难。错误和不准确的时间估计是生活的现实。如今软件太复杂了，事情不可能完全按照我们的意愿发展。然而，开发人员经常通过做一些不必要的事情来增加软件开发的难度。我知道我有。

其中一个趋势是把减少打字和节省时间联系起来。一开始听起来很有逻辑。少打代码，少花时间写代码。然而，大多数开发时间并没有花在打字上。如果超过 5%的时间花在打字上，我会很惊讶。大多数开发时间是理解现有代码、计划新代码、测试新代码和调试新代码。重用相关特性的代码很有意义。它确实节省了打字，并有助于完成上面列出的所有其他事情。

问题是当看起来相似的不相关特性的代码被重用时。感觉是一样的，因为重用代码就是重用代码，对吗？但是不相关的特性在某个时候会在功能上产生分歧。这意味着重用的代码对于每一个都必须有稍微不同的功能。在最好的情况下，每个接触代码的开发人员都会意识到这一点，并在代码中创建特例。额外的时间花费在测试两个特性上，而不是一个，额外的时间被新开发人员花费在不必要的复杂性上。

在最坏的情况下，有人不会意识到某个地方重用了一段代码，因为这没有逻辑意义。他们对一个功能进行修改，但是破坏了一个完全不相关的功能。现在，我们不仅有过于复杂的软件，而且还有充满错误的软件。

将可重用性提升到另一个层次，我们开始得到过度“通用”的整个系统。在许多情况下，这很有意义。亚马逊最初有一个卖书的系统。这是足够普遍的，他们可以出售任何东西。如果最初的意图是卖任何东西，而书只是一个起点，那么做一个仅仅卖书的系统是愚蠢的。

然而，软件系统最终可能过于通用。销售产品的代码和评论/评论这些产品的代码是不同的系统。它们在某些时候被联系在一起，但是产品是一个不同的概念，评论是一个不同的概念。一些系统将试图消除这种区别，并将它们归入某个虚构的保护伞下，比如说一切都是“节点”或一切都是“帖子”。很多内容管理系统都这样做。

如果你想知道“节点”或“发布”是什么意思，那么这个问题就是我的观点。他们是暧昧的。过度以至于增加混乱。它们太普通了，对于我们的评论或产品系统来说，它们真的没什么用。我们仍然需要围绕评论和产品建立大量的功能。然而，如果我们试图将这些不同的概念包装在某个通用的保护伞下，我们现在还必须处理该保护伞提供的额外的复杂性。这是补充，而不是代替。

结果是，我们没有通过重用代码来节省时间，而是花费更多的时间来处理代码的复杂性。

还有很多其他例子说明开发商让他们的日子更难过了。当事情不需要扩展时，我们试图让它们变得可扩展。我们试图考虑用户永远不会遇到的用例。我们试图使我们的代码高效，但最终为了节省无意义的纳秒而增加了太多的复杂性。

在所有这些方面，我们作为开发人员使我们的工作比他们需要的更困难。我们的工作已经够难的了。认识到我们的哪种倾向会导致我们增加不必要的工作量，可以让我们的生活变得更轻松。至少对我来说是这样。
