它是用 python 制作的，使用简单的功能。它使用 API 的。我在 Google Places API 中使用了 slimkrazy 的 python wrap around，在 Open Weather Map 中使用了 pyowm，在《英雄联盟》(League of Legends)中使用了 Riot 的 API(我怀疑没有人会认出这个游戏)。

聊天机器人是第一年第一学期的项目。6 个随机的人被放在一个小组里，他们的任务是“制作一个聊天机器人”虽然在那个时候，这似乎没有什么大不了的，我们有这些疯狂的想法，它必须有一个特定的个性，甚至应该有一些愤怒的问题。我们幻想着它会如何从网上给我们买东西，成为人类最好的朋友。
不要把画面想当然。它还可以进行一些简单的聊天，我只是把我自己专门做的功能推了出来。
我们不知道编码过程没有按计划进行。每个人或多或少都知道自己的任务，但最终没有办法将它们整合在一起。一个函数接受列表作为参数并返回字符串，另一个函数接受列表作为参数并返回字符串，这完全是一团糟。然后我们两个开始做同样的函数，当最好的一个爬到顶端时，另一个人只是在浪费时间。整体代码集成让我们欣喜若狂。

在这一部分，我将谈一谈调试，所以请随意跳到下一段，因为这个词从来不会带来好的情绪。尤其是一年级的时候。我们知道一个好的做法是用 try-catch 块包围我们的代码，这样我们的程序就不会随机崩溃，但是当最终某个东西不能按预期工作时，几乎不可能找到代码的哪个部分出错了。只是我们忘了把错误本身打印出来。如果某样东西在一台电脑上运行，并不意味着它在另一台电脑上也能运行。此外，虽然我的笔记本电脑上的代码只能在 32 位 python 上运行，但我的一位同事只能在 64 位 python 上运行相同的代码。单元测试是这里的一种方法，因为你创建一个函数来测试其他函数一次，但是每次你需要发生一些事情的时候运行它。我不知道你怎么想，但是我的讲师明确告诉我远离全局变量。现在我不知道该怎么感谢他们，因为有一个问题我们花了几个晚上试图解决，但最终一个全局变量解决了。

最终，聊天机器人不再像最初那样了。从字面上看，它唯一继承的是它的名字 Snek。但是即使一路上有很多障碍，我还是很高兴现在它把我带到了哪里。通过这个项目，我教会了自己比我想象中更多的东西。其中之一是如何作为一个团队工作，以及下次使用什么样的管理技巧。另一个——我渴望进一步探索 API 的整个世界。
