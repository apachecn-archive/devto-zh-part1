# 不要让 JavaScript 疲劳成为一个时髦词

> 原文:[https://dev . to/Ben/do-not-let-JavaScript-fatigue-being-a-buzzword](https://dev.to/ben/do-not-let-javascript-fatigue-become-a-buzzword)

如果您为 web 编写 JavaScript，或者在过去几周内关注过开源软件，您可能听说过“JavaScript 疲劳”这句话尽管 JavaScript 令人兴奋，但这种现象是真实的。这个短语指的是当前流行的 JavaScript 框架，即 [ReactJs](https://facebook.github.io/react/) ，需要围绕工具进行艰苦的设置，只是为了获得一个基本的“Hello World”应用程序，并且在项目得到良好的润滑和管理之前，很难实现生产率的提高。 [Eric Clemmons 在最近的一篇博客中很好地总结了 JavaScript 疲劳](https://medium.com/@ericclemmons/javascript-fatigue-48d4011b6fc4#.mhkrt7gvc),并提供了该社区主要挫折的简明摘要。

> 最终，问题是通过选择 React(和固有的 JSX)，您已经不知不觉地选择了一个混乱的构建工具、样板文件、linters 和时间接收器的巢，在您开始创建任何东西之前，您必须处理它们。
> 
> 埃里克·克莱蒙斯

## Rails 无法扩展

“Rails 不可伸缩”是人们常说的一句话，但意义不大。这是一个流行的词，似乎可以追溯到 2011 年 Twitter 脱离 Rails 作为其服务核心结构的新闻。人们使用这个短语，有时是为了快速解释软件选择，有时是为了听起来聪明，有时是因为它符合所说的上下文。Ruby on Rails 完全不符合 Twitter 正在做的事情，原因对于有经验的开发人员来说是显而易见的。但是有经验的开发人员也倾向于认识到 Ruby 语言的优势和 Rails 框架的好处。缺乏经验的开发人员和无知的决策者使用“Rails 不可伸缩”这个时髦词来达到过于复杂的系统，这些系统将第一个产品的发布时间延长了几个月或几年，原因无非是害怕“可伸缩”的恶魔。

类似地，短语“JavaScript 疲劳”是上下文特定的、时间特定的，最终是工具在其普及时的状态的简写。这并不意味着二手观察者现在和将来会把“JavaScript 疲劳”理解为什么。JavaScript 疲劳作为一个流行词，缺乏激励更大的编程社区投身其中，尝试一些令人兴奋的新想法，但这是一个令人兴奋的领域，值得无畏地探索。禁止性的流行语会让开发人员止步不前，并为制定技术决策提供有缺陷的启发。

## 开源软件的本质

开源软件是由不同风格、目标、文化和语言的人构建的。在一个缺乏合理共识就像死亡和税收一样确定无疑的世界里，软件开发中群体方法的智慧(和努力)有时似乎是不合适的。但是尽管一路走来有一些艰难困苦，还是有一些伟大的事情发生了。开源软件为世界上成千上万不同规模的公司提供动力，并为我们所知的大部分可用网络提供粘合剂。随着时间的推移，开源软件的副作用往往会暴露出来并得到管理。

由于开源软件开发的这种分布式本质，公共辩论是任何项目健康和稳定的关键因素。流行项目的健康和稳定最终对软件开发人员的长期生产力至关重要。因为稳定、经济的软件开发对全球经济实力至关重要，所以不能低估这些公开辩论的重要性。但我们应该认识到，当辩论过于简化，被归结为未经适当咀嚼就太容易消化的内容时。

## 那我们该怎么办？

JavaScript 疲劳已经成为 2016 年最热门的科技词汇之一。它似乎比它描述的更多。目前，社区已经使用它来描述 JavaScript 社区在它被创造出来时面临的工具问题，但是随着这个短语扩散到流行词领域，它很容易失去原来的上下文，人们很容易推断出他们自己对它的含义。

谨慎使用像“JavaScript 疲劳”这样的热门词汇。尽可能提供背景和细节。不要用它来提供对 JavaScript 的一般性批评，即使你已经成为一个铁杆的 Elm 布道者或者对其他一些假定的成功感到兴奋。JavaScript 是一种语言，也是一个生态系统，每个程序员都有一些经验，有它的优点和缺点，但它是一个非常令人兴奋和增长的领域，应该根据它的优点而不是时髦的词语来判断。

JavaScript 是一个真实的东西，但你是否也知道短语“disrupt”也指一种由克莱顿·克里斯滕森提出的[特定观察到的现象](http://www.claytonchristensen.com/key-concepts/)？因为它变成了一个科技流行语，被用在似乎任何商业概念上，所以它失去了所有的意义，尽管克里斯滕森[试图澄清](http://a16z.com/2015/10/28/what-disruption-theory-is-and-isnt/)。让我们从一些历史中吸取教训，尽量避免让“JavaScript 疲劳”和未来技术中值得讨论的短语重蹈“Rails 不可伸缩”和“颠覆”的覆辙。
