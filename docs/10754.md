# 反应式编程的概念和误解

> 原文:[https://dev . to/react ive conf/the-concepts-and-misconcepts-of-reactive-programming](https://dev.to/reactiveconf/the-concepts-and-misconceptions-of-reactive-programming)

反应式编程是一种强大的、有时会被误解的编程范式。关于这个话题，很少有人比即将在[举行的反应会议](https://reactiveconf.com/)上发言的[和](https://twitter.com/andrestaltz)更有发言权。我请 Andre 澄清一些概念，并就何时选择被动方法可能是个好主意提供建议。如果你对这些概念感兴趣，我希望你和我一样发现他的答案很有帮助。

## [](#interview-with-andr%C3%A3%C2%A9-staltz)采访安德朗·斯塔尔茨

**你是如何进入反应式编程的？**

在加入 [Futurice](http://futurice.com/) 之前，我从未听说过反应式编程。正是通过使用 [Bacon.js](https://baconjs.github.io/) 、 [RxJS](https://github.com/Reactive-Extensions/RxJS) 、 [RxJava](https://github.com/ReactiveX/RxJava) 、 [ReactiveCocoa](https://github.com/ReactiveCocoa/ReactiveCocoa) 的同事，我对反应式编程和函数式反应式编程产生了好奇。很快，我在一个 Android 项目中广泛使用了 RxJava，并不得不艰难地学习 Observables 和 operators:通过真正使用它来交付有期限的软件。我的学习过程是通过相信“一定有‘反应’的方式来做这件事”和在纸上画出几十个大理石图来指导的。

在学习和采用反应式编程时，开发人员需要克服哪些常见的心理障碍？

大多数接触过反应式编程的程序员，包括我在内，都习惯于“触发”、“更新”、“调度”、“启动”方法和过程。这些是与反应式编程相反的习惯用法和技术，在反应式编程中，我们试图声明什么是“触发的”或“更新的”。最大的挑战就是简单的经历从“A 改变 B”到“B 被 A 改变”的心理革命。之后，较小的挑战是学习 Rx 操作符，它们适用的地方，以及常用的技术。

你是否发现人们对 React、脸书库和反应式编程的命名区别感到困惑？

不幸的是，是的。反应式编程是古老的，尽管它已经通过 Rx 和 Akka 等变成了一种现代的方法。另一方面， [React](https://facebook.github.io/react/) 使用了一些反应技术，因此借用了这个名字。我喜欢把这两个比作“汽车”和“地毯”。你的车里面有地毯，但是汽车和地毯是完全不同的物体。React 中有一些反应式编程，但这两者是完全不同的东西。事实上，在普通的回调中甚至有一些反应式编程，所以我不认为反应式编程是 React 中值得注意的特性。

你见过人们选择 React/Redux 的例子吗，当他们使用纯反应式方法会更好？

是的，我确实看到了那些例子。然而，我认为情况并不总是如此。甚至有这样的情况，人们选择 React/ [Redux](https://github.com/reactjs/redux) (或任何单向数据流方法)，当他们更适合双向数据绑定时，就像在 [Vue.js](https://vuejs.org/) 中一样。每种方法都有其理想的用例。谈到完全反应式方法，我认为它最适合需要处理来自多个不同来源的传入数据的应用程序:WebSockets、Firebase、传感器数据、键盘、Kinect 等等。当浏览器中只有图形用户界面、来自点击的几个用户事件以及要与之通信的单个服务器时，React/Redux 工作得相当不错。

人们可以使用反应式编程来解决通常被忽视的问题吗？

在很多情况下，反应式编程会有所帮助。RxJS 和类似的库是目前最好的以最简单的方式处理各种取消的工具。承诺是不可取消的。这意味着您不能停止使用`fetch()`构建的进行中的请求。即使您自己用手动状态簿记和 Ajax 实现取消逻辑，所需的代码量也令人失望。事实证明，大多数前端开发人员忽略取消，因为他们认为硬件资源很充足。然而，使用 RxJS，取消是简单的、惯用的和半自动的，因此您只需添加一些操作符，然后您的代码就可以处理复杂的异步取消。不过，RxJS 不只是用来取消的。

在 Reactive Conf 上，你将谈论什么？

我将讨论如何超越 console.log 和单步调试程序，以及如何开发新的代码可视化工具来帮助我们理解和调试应用程序。特别是，我将指导人们使用 [Cycle.js](http://cycle.js.org/) DevTool，这是一个将应用程序中的全部代码可视化为数据流图的工具，以及如何在使用应用程序时实时观察事件的流动。

**什么是 Cycle.js，你为什么要创建它？**

Cycle.js 是一个用于构建用户界面的 JavaScript 框架。它允许您构建典型的前端应用程序，如您最喜爱的电子商务单页应用程序，但也有一个灵活的架构，知道如何以结构化的方式适应不同的用例，如游戏控制器、WebSockets、键盘、Kinect 的使用，并具有良好的关注点分离。我创建这个主要是因为我想解决“用户界面架构”。我希望有一个指导原则和必要的工具来处理我构建的任何类型的 UI 应用程序。我相信 JavaScript 将在未来驱动几乎所有的用户界面，并且必须有一个通用的方法来编程它们，关注代码的可读性和生产率。

对于 Cycle.js 项目的持续健康发展和成功，什么是重要的？

说实话，钱。不是为自己，是为贡献者。我们已经达到了这样的规模，作为一个人，我无法独自处理所有的问题和请求，拥有核心贡献者是至关重要的。Tylor Steinberger 和 T2 Frederik kraut wald 是我非常感激的两个人，他们作为志愿者帮助建立了这个框架。我们把它作为一个真正的草根开源项目，而不是一个属于像谷歌或脸书这样的大型科技公司的项目。最近，我们为我们的贡献者建立了一个[捐赠站点](https://opencollective.com/cyclejs)。
