# 玩追赶

> 原文:[https://dev.to/eclairereese/playing-catch-up](https://dev.to/eclairereese/playing-catch-up)

妨碍我的一种工程思维是，我永远不会“赶上”其他人已经知道的东西。在我 26 年的生命中，有 23 年是在没有代码的情况下度过的，我错误地认为我将一直在赤字中工作，并且永远不会与我的开发同行的智慧持平。对于学习艺术或外语，我从来没有过同样的失败主义想法，所以想知道“我已经落后了，为什么还要烦恼呢？”是一个我无法回答的全新问题。

是什么让工程界对初学者产生了这种宿命论？新手和专家都相信网络、游戏和硬件开发的天空是无限的，但是作为一个新人，当遇到一个又一个首字母缩写词或者一个特别固执己见的 GitHub 问题时，很难保持这种乐观的态度。

这里起作用的一个原因是，新手和专家的公共街道信用指标完全不同。初学者通过在 Codecademy 上获得徽章、在 Rails 中分享他们的第一个博客的链接或者将他们的第一个应用程序推送到 GitHub 来获得积分。

然而，随着时间的推移，这些交错的、基于共享的指标让位于另一种“成功”许多初学者，包括我刚开始的时候，认为有经验的开发人员拥有大量的 Twitter 追随者，在 GitHub 上有大量的贡献历史，或者对编程库有很多强烈的意见。

事实上，这些指标与实际工程无关。在 GitHub 上出现一个又一个绿色方块并不意味着某人是一个负责任的、聪明的或令人钦佩的开发人员，也不意味着他们发了 X 次推特。

我希望我们认识到，这些“精通”的标准太容易成为程序员超级明星的标志。脱离了渐进的、可见的学习的成功衡量标准，使得初学者不可能按照他们所崇拜的工程师来规划他们的课程。

在其他学科，初学者和专家掌握的标志是平行的。新科学家在学校写论文；专家科学家在期刊上发表论文。“论文”的概念保持不变。从新手到专家，这是评估科学家知识的一个可见且一致的标准。我们在编程中没有这样一个具体的、可接近的标准，我们必须对我们如何赞美专家更加负责。

如果工程界教导说“成功”是拥有一个密集的网络存在，并且能够对某些编程语言屈尊地窃笑，我们并没有证明学习和耐心是掌握的真正途径。任何人都可以对 XYZ 框架嗤之以鼻，但是需要一个真正令人钦佩的开发人员来承认他们不知道什么或者他们正在学习什么。

无论你在学习的哪个阶段，我鼓励你们都做类似的事情，展示你的工程旅程，而不是你的目的地。还有初学者，你也不落后！你已经在你应该在的地方了。我们都是。♥
