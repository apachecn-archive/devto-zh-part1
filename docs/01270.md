# 加利福利亚软件项目管理

> 原文:[https://dev . to/code mouse 92/galli freyan-software-project-management-29a 1](https://dev.to/codemouse92/gallifreyan-software-project-management-29a1)

注意:我有机会在第五集中更深入地讨论这篇文章。跳到 30:45 那段。

* * *

我第一次接触编程是在我高中的第三年——也许更好的理解是我的第二个大三，因为两年前的一次创伤性脑损伤[TBI]让我在学术上倒退了几光年，从 4.0 的大二和大学水平的阅读到不及格的预科材料。我爬着，用手抓着，回到了我曾经能够完成的课程的水平。在这个过程中，我失去了完全掌握我曾经热爱的生物科学的能力，但我获得了对数学的巨大理解。与此同时，我现在有能力在更高的维度上可视化算法。我把它比作一个超级英雄起源的故事，虽然没有有形的力量。

正是在高中“二年级”的时候，在我玩 Visual Basic 的时候，我买了一本斯科特·罗森伯格的《用代码做梦》。

在第一章，罗森博格描述了他所谓的“软件时代”...

> 在制作软件的过程中，时间确实表现得不同。当事情进展顺利时，在心理学家称之为“心流”的状态下，你可能会忘记过去的时间。当事情变得糟糕时，你被卡住，冻结在维度之间，无法移动或看不到前进的方向。无论哪种方式，你都已经远远落后了。你现在是软件时间。

我的大脑已经将时间视为一种流动的织物，而不是一条刚性的线，它立即引起了我的注意。在我短暂的几个月编程生涯中，我经历了这两个极端。我知道软件时间是什么感觉。

*用代码做梦*继续描述围绕软件开发及其项目管理出现的大量奇怪现象。

就在这时，我明白了:我不仅想成为一名程序员，还想成为一名软件项目经理。

* * *

一晃八年过去了，我在这里，经营着一家开源软件创业公司。我个人遇到过几十个罗森伯格、布鲁克斯、雷蒙德等黑客抄写员描述的离奇现象...我*喜欢它*。

我的员工会告诉你，作为一名首席开发人员，我有点像神秘博士:古怪，可能疯狂，从一些令人费解的角度运作，有时会给出奇怪的解决方案，要么不可能成功，要么失败得很惨。然而，这是可行的——我们实际上*的确*生产功能软件，在一个完全由远程的每周工作六小时的程序员组成的团队中(除了我自己)，他们都是实习生或毕业生。

我无法想象“加利弗里亚”式的项目管理方法是我独有的，但我观察到这种情况很少见。看起来大多数管理技术要么试图将软件开发的不稳定的时间性塞进僵化的结构中，要么根本没有计划。

我可能很疯狂(虽然这是一种为我们的项目工作的疯狂)，但是我希望我的一些技术在其他地方可能有用。有时很难将我的非线性思维转化为可表达的形式，但这是我最好的尝试。

世卫组织可能想拿出他们的声波螺丝刀。其他人，系好安全带，坚持住。这个话题在内部更大。

艾伦西。

# 通量:“这是一个时间的事情”

我认识到我不能向任何人解释整个项目路线图；每次尝试都以同样呆滞、略带惊慌的表情告终。基本上，我已经学会了保持沉默，因为没有人需要知道。

然而，到目前为止，在我计划的路线图和项目的实际过程之间很少出现巨大的偏差。这并不是因为我有超人的远见(我没有那么好)，而是因为我有能力理解和计划*和**的变化**。*

从*神秘博士*中进行类比，你知道时间中有一些永远不会改变的“固定点”,而其他一切都是一大堆“可能”吗？这就是我在项目管理中看到的。

[![Wibbly-Wobbly Timey-Wimey](img/b492d04b9361b039adcc8c73c02bdbbf.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--DzD9Ipqd--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/x9avprhpss98j661ky0k.gif)

换句话说，我明白了在软件中有三件事是永远不可预测的:

*   问题复杂性

*   生产时间

*   第三方软件的生存能力

在我的路线图中，这些都是巨大的变化领域。我只是不知道那里会发生什么。我在项目时间表中对它的想象有点像跨越鸿沟的黑桥。它可能只有几英寸或几百英尺长，但我们要到那里才能知道。值得注意的是，你可以相当准确地预测一条路径上有多少磁通桥。你只是不知道它们有多长。

基本上，关于通量本身，你唯一能预测的就是它是不可预测的。这是最让项目经理害怕的部分。如果我说它从来没有吓到我，那我是在撒谎，但它没那么可怕，因为我遵循两个简单的规则。

同样，很难把我的非线性思维用语言表达出来，所以让我们再次调用“通量桥”类比。

1.  在你放弃一条路径之前，对它可以有多少个磁通桥设置一个硬限制。

2.  在你放弃一条路径之前，为你沿着一个通量桥走多长时间设置一个硬限制。

换句话说，项目管理不像旧的最短路线算法问题。是 NP。没有办法最优解决，只能蛮干。从一条路径开始，直到它变得太长，然后尝试另一条路径。

# 相对论:修道院的钟声

和医生一起旅行是危险的。如果你想生存下去，你必须对时间和相对空间的基础有所了解，否则你就完了。

项目管理不仅仅是一个人的责任。在一个健康的团队中，每个人都有自己的角色。对我来说，这意味着我必须找到一种方法，让我的整个团队能够发现那些易变的领域。

在 MousePaw Media，我们有一个称为量化任务管理的标准，它将关于编程任务(问题、错误或功能)的基本信息归结为四个 0-5 级:严重性(重要性)、优先级(紧迫性)、摩擦(可用的帮助资源)和相关性。

最后一条可能是最重要的。相对论是一个问题成为黑洞的概率。对于你们这些卫星人来说，这有点像塔迪斯修道院的钟声，每当有重大的暂时危险时就会响起。

基本上，黑洞是一个没有端点的通量桥。你永远也到不了彼岸。会消耗无限的时间和资源。黑洞存在于软件中。没有一个项目将*从黑洞*中拉出来的记录案例。你*必须*回到起点，在一条不同的道路上重新开始。

[![Tardis in space](img/25055f3a3ba689447bcc42c2b5877a9c.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--ulzKmfig--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/u6pfi6e9f4jyjkxouqb0.gif)

由于问题的复杂性，任务有很多变化。直到你开始工作，你才真正知道一项任务需要多少工作量，或者完成起来有多难。

我们的**相对性**评级对任务成为黑洞的可能性进行排名:

*   `r0`:没有黑洞的机会。没有流量。
*   `r1`:黑洞概率低。可能是安全的。
*   `r2`:中等黑洞概率。一些通量，但看起来有可能完成。
*   `r3`:黑洞概率高。仍然有可能完成，但要小心。
*   几乎确定的黑洞。可能完成，但不太可能。
*   `r5`:塌陷。马上跳伞。

黑洞任务在*做梦代码*中描述(第。13-14).在书中，开发团队采用了一种不那么微妙的方法。每一只黑洞虫都被贴上了“可怕”的标签。

正如这本书所描述的，确定一项任务的相关性并不像人们想象的那么难...

> “在研究这个 bug 的第一个小时内，”Burgess 主动说道，“你就知道会是哪一个。”

# 时间:“我们永远在一起”

正如我前面提到的，我基本上从一开始就知道，当你处理软件开发时，线性时间就不存在了。

换句话说，软件有自己的时间线。你可以试着设定所有你想要的截止日期，但最终，那些只是愿望，没有任何可能的实质内容。只有两种方法可以在截止日期前完成:

1.  忘记“功能完整”这个东西，使用任何有效的东西，或者，

2.  把质量完全扔出窗外。

现在，如果你作为一个程序员有一点体面的话，你甚至不会把#2 作为一个选项。因此，你只能瞄准两件事中的一件:你的功能集或你的截止日期。不会两者兼得。

有时项目会给你惊喜，你会按时交付全部功能集，甚至更多！当这种事情发生的时候总是很棒，但是不要把它写在书上。

举个例子，2017 年对 MousePaw Media 来说是诡异的一年。我们完成了两个项目:PawLIB 1.0 和省略。

无论以何种标准来衡量，PawLIB 都应该被视为已逾期近一年。我们在开发过程中遇到了很多问题，但我们最终发布了第一个版本。最后，我们只是通过将几个主要特性标记为“实验性”并将其发布到 PawLIB 1.1(目前正在开发中)来实现这一点。我们可能实际上满足了 PawLIB 的理论期限，但是我们重视某些关键特性的发布。

另一方面，省略的出现比预期的要快得多，而且比最初计划的功能更多。然而，这个项目具有讽刺意味的是，由于所有地方的*包装*中的`r4`黑洞，发布被推迟了五个月(还在继续)。绕过黑洞需要用不同的 GUI 进行完全重写，但我们在一些外部帮助下完成了，现在有一个相当确定的 2018 年 2 月发布目标。

几年前，在一次臭名昭著的事件中，我学会了不要扰乱生产时间的流动。在 C++中，我们需要一个完全 Unicode 兼容的字符串类，并具有某些特性，我低估了所涉及的流量。我将它分配给我们的一个开发人员，并声明“这应该只需要一个周末。”

预计这个字符串类会很稳定，并在今年冬天晚些时候，也就是两年和三个开发者之后，准备好首次发布。这确实是一件了不起的事情，但是它有着惊人的流量。

# 第三方软件:“当我说运行...快跑！”

第三个，也可能是最大的，流量来源是第三方软件。任何时候你选择依赖一个工具，一门语言，一个库，或者一个框架，你都是在你的项目中引入一个巨大的未知量。很多事情都可能出错:

*   该软件对于开发者来说可能太复杂了，以致于无法学习或使用。

*   可能有一个你还没有意识到的错误或缺失的特性阻碍了你的进步。

*   在软件和你的开发栈的任何其他部分之间可能有一个独特的不兼容性。

[![Okay, I think we should run](img/76c94d135543e35d40b5562fc50c5a6f.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--cOZlnn7W--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/12gilap3r44yehk8po0h.gif)

正如我前面提到的，我们被阻止在 2017 年发布*遗漏*是因为包装上的黑洞。我们选择使用 Kivy，一个 Python GUI 库，来构建游戏。虽然这个库在代码中做了我们想要的一切，但它有一堆复杂的依赖项(和一个`setup.py` hack ),这使得它实际上不可能为我们的主要目标平台之一 Linux 打包。当 Kivy 的一个开发人员设法用 pyinstaller 编译省略时，我们才摆脱困境，最终结果不符合 Debian 打包标准(需要一些非标准的。deb 封装)。正如我之前所说的，我们可以绕过这个黑洞，但这需要使用不同的 GUI 库进行完全重写。

现在是另一个例子:在 2017 年早些时候，我们项目的主要部分被我们对一个名为 Xerces 的 C++ XML 库的依赖所阻碍。两名开发人员在 2016 年花了大部分时间学习如何使用该库，在各自的代码领域进展甚微。最后，我们作为一个团队决定停止使用 Xerces——它太难学，也太难用——而使用 pugixml。我们实际上不得不*抛弃*有问题的软件来绕过这种流动。

# 限制通量

到目前为止，我可能没有告诉你太多你不知道的事情。许多软件项目由于失去最后期限、无限深入的任务和难以使用的第三方依赖而被有形地延迟和陷入困境。

但是请记住，我们并不是完全没有反击的方法！回忆一下那些**磁通桥**。在我的精神项目路线图中，我通常可以找出大多数这些通量桥在哪里，并使用它们来计划！

[![Sonic Screwdriver](img/a290828216d902aba94c398e67a9bb7f.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--M3Hk9aGL--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/uk1az2979hczukv3qjho.gif)

让我们回顾一下这两条规则:

**1。在你放弃一条路径之前，对它可以有多少个磁通桥设置一个硬限制。**

我真的没有一个具体的数字可以给你。每个项目都有一点不同，随着时间的推移，你会感觉到有多少磁通桥太多了。

例如，我们最近在讨论如何在两个进程之间发送数据。一种方法是在一个宏中构建两个解决方案，在平台上切换:UNIX 系统将使用套接字，而 Windows 系统将使用本地方法。另一条路是使用谷歌协议缓冲区。

对于方法#1，我们知道我们在编写自己的协议方面有一些变化。方法#2 涉及第三方依赖，以及一个动态链接库。我们已经遇到了动态链接库的重大问题。因此，虽然第二种方法显然是可移植的，但我们有两个主要的变化点，而方法#1 只有一个。

明白这是怎么回事了吗？你算出每条可能的路径上有多少磁通桥，然后选择最少的一条。

**2。在你放弃一条路径之前，为你沿着一个通量桥走多长时间设置一个硬限制。**

一旦你开始走这条路，你意识到那里的流量比你能处理的还要多，*退出*并尝试另一条路。重新开始所花的时间几乎总是比继续进入黑洞所浪费的时间要少。

回顾之前的 Xerces 事件，我们认识到我们在学习库上花费了太多的时间。[学习是不可转移的](https://dev.to/codemouse92/your-project-isnt-done-yet)，这意味着项目中的每个新开发人员都必须走过同样的艰难的 Xerces 和我们对它的使用的通量桥。我们最终为图书馆设置了一个硬性限制:如果做一件有意义的事情需要超过 20 个小时，就把它扔掉。

我对通量还有一个考虑:

**3。当你试图让两条路径收敛时，保持即将到来的磁通桥的数量大致相等。**

让路径收敛是一个相当的帽子戏法。我们不得不做一次，当时我们需要为 PawLIB 1.0 的发布同时完成三个主要特性，而不耽误其他项目。

我们放弃了另一个尝试，试图同时编写两个依赖库，这样他们阻止的项目就可以由整个团队一起开始了。老实说，当你不能保持通量桥计数一致时，你就放弃尝试并重构。

# 最后的想法

老实说，我看起来比实际上更能控制局面。我对通量没有太多的控制。我试图尽可能地避开它，但有时我们还是会陷入黑洞。

这让我想到了 Gallifreyan 软件项目管理最重要的原则...不要假装什么都有！无论你是否对项目有时间观念，你都只能在问题出现时处理问题。

我的团队已经习惯了我。他们知道我在驾驶这个 TARDIS，他们通常对我很有信心，但有时事情会出错。那时我不得不承认我们正处于软件时代。

或者用第十一位医生的话说:“做我做的事。抓紧，假装这是个计划！”
