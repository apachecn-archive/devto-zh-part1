# Igor Minar 和 AngularJS 团队如何构建软件

> 原文:[https://dev . to/react ive conf/how-Igor-minar-and-the-angular js-team-build-software](https://dev.to/reactiveconf/how-igor-minar-and-the-angularjs-team-build-software)

软件开发可以是选择正确的技术，也可以是构建技术。像将应用程序的未来绑定到哪个前端 JavaScript 框架这样的决策是很困难的，因为需要考虑太多关于需求的现在和未来。这种情况已经转变了相当一段时间，你可能会发现完全不同的观点，这取决于你和谁交谈。选择 JavaScript 框架需要考虑很多因素，包括消息传递和数据流的技术细节、支持的内置工具和模块生态系统以及各种学习曲线。

然而，在核心上，决策可能不是关于 API 细节，而是关于项目的前景，了解团队如何操作和思考软件。Node、Rails 等开源 web 项目的成功与失败。取决于团队合作、领导力和远见等诸多因素。像 [Famous.js](https://techcrunch.com/2015/11/06/nopen-source/) 这样资金极其充足的项目已经失败，而由少数人在业余时间维护的项目有时会蓬勃发展。

下个月我将参加 [Reactive Conference](https://reactiveconf.com/) ，届时 Igor Minar 将讨论网络生态系统如何塑造其应用框架。Igor 是 Angular 项目的领导者，Angular 项目是由 Google 开发的流行的 JavaScript 框架。我找到他，从他的角度问了一些关于 Angular 团队如何运作的问题。随着最近 [Angular 2.0](http://angularjs.blogspot.com/2016/09/angular2-final.html) 的发布，有必要了解一下项目负责人是如何对待开发策略和社区的，供您在规划自己的应用程序的未来时考虑。

Igor 是一个忙碌的人，所以我很感激他能够回答几个问题。

## [](#questions-for-igor-minar)对伊戈尔·米纳尔的提问

团队如何做出决策并权衡来自更大社区的意见？

有两种主要的方法可以做到这一点。首先，所有的开发都是公开的，我们公开所有的设计文档和会议记录。这允许社区参与进来并指出缺失的需求或被忽略的设计缺陷。其次，我们每天都与社区中的许多成员保持联系，他们提供了宝贵的反馈，并强调了我们可能忽略的问题。

你如何平衡短期需求和项目的未来需求？

这取决于我们谈论的是公共 API 还是内部实现。对于公共 API，我们倾向于保守，因为修复这里的错误可能会非常昂贵，通常更有意义的是提供一个短期需求的解决方案，然后改变框架。当谈到框架的内部时，重点是可维护性和可延展性，因为这是允许我们发展框架的地方。

团队如何处理任何可能产生的技术债务，或者处理设计冲突？

在理想世界中，所有技术债务都可以避免，但在一个有期限的世界中，这是不可能的。对我来说最重要的事情是，每个为代码库做贡献的人都感受到了对它的所有权和责任感。这与我们的 CI 测试期间的各种检查相结合，有助于减少但不能防止技术债务的数量。当累积的债务开始干扰我们完成工作的能力时，或者当我们有一点休息时间时，我们就回去清理事情。

你对框架或库的编码与对应用程序的编码有何不同？

构建应用程序和构建框架的最大区别在于，应用程序与(相对)容易适应变化的人进行交互。如果你移动一个按钮几个像素或者改变它的颜色，人们很可能会像以前一样使用你的应用程序。如果你对一个框架的常用 api 做了一个小的改动，所有使用该 api 的应用程序都将停止工作。与框架相比，这使得在应用程序上迭代要容易得多。

给有兴趣为项目做贡献的人的建议。

从小处着手，坚持不懈。我们审查了大量的社区贡献，只有极少数可以被合并。许多人想为 Angular 做贡献，做出大的改变，但是这些改变经常被拒绝。不是因为它们不是来自核心团队，而是因为变更是如何实现的，以及它是如何与代码库的其余部分集成的。从小事做起，例如文档变更和错误修复，您可以了解我们的测试、文档和其他需求。
