想看另一场重量级比赛吗？找出谁在云提供商之战中胜出。

## 设置场景

曾几何时，我们使用 [Parse](http://parseplatform.org/) 作为应用程序中推送通知的主要提供者。效果很好。但是 [Parse 去世了](https://bits.blogs.nytimes.com/2016/01/28/facebook-to-shut-down-parse-its-platform-for-mobile-developers/)，剩下的任务就是寻找新的提供商。

进入舞台左边:多面手。Firebase 是谷歌移动应用开发的 Johannes 杂役。崩溃报告、云存储、托管、实时数据库、远程配置、身份验证...当你开发一个移动应用时，Firebase 会在合适的时间变成合适的工具。

我们已经使用了移动应用的瑞士军刀，并且知道他们有推送通知的工具。我们想知道:这个工具够锋利吗？

它必须锋利。我们管理的项目有一些非常复杂的需求。一些团队成员希望引入第二名玩家来探索所有选项。

进入舞台右侧:专家。OneSignal 比 Firebase 有一个直接的优势——它是免费的。它承诺通过推送传播，专注于高质量的用户参与。

第一幕时间到了。

## 第一幕:多面手

Firebase 是一个很棒的平台，可以帮助开发者快速构建高质量的应用。除了上面提到的所有东西，它还利用了使谷歌在分析等领域成为市场领导者的功能。这是一个你可以在其上构建的控制台；一站式商店，大摇大摆的装修套房。

在发送推送通知时，它也很慢，而且相对不稳定。我们对某些项目有一些特定的要求，当我们尝试一种变通方法时，API 并没有像预期的那样工作。

一把瑞士军刀有时会让你不堪重负。

另外，在撰写本文时，Firebase 的推送通知不能与 [Xamarin](https://www.xamarin.com/) 一起使用。

也许是时候让专家走上舞台中央了。

## 第二幕:专家

OneSignal 的即时优势？这是一种推送通知服务。正是我们所需要的——不多不少。我们从测试一个特定项目中的一个信号开始，从 Firebase 迁移过来。集成很容易，从开始到结束都非常完美。该 API 比 Firebase 的 API 更全面，它允许我们定制行为以满足我们的需求。

在两端(移动端和后端)进行集成既快速又极其简单。作为一名专家，它真正开始拥有舞台，推出了一些迷人的动作，包括:

*   跟踪推送通知。
*   A/B 测试。
*   过滤和分段(例如按位置)。
*   很棒的文档。
*   为网站推送通知。

然而，像任何专家一样，OneSignal 并不拥有大量成熟的功能，也不能像 Firebase 那样处理移动开发所需的所有服务。只有在推送通知方面，它才真正出类拔萃，就像熊猫只有在有大量竹子可供咀嚼时才会茁壮成长一样。

## 第三幕:高潮

Firebase 拥有一切。它非常适合从头开始创建应用程序，并且避免了使用后端服务的需要。理论上，它是 Parse 的完美替代品。它提供了所有相同的功能，甚至更多——用谷歌精心制作的产品的影响力来增加交易的吸引力。

此外，它还有推送通知——这是我们一直想要的，对吗？为什么不保持干净，把 Firebase 作为一切的枢纽，包括推送通知？

OneSignal 做得更好。

通过它的过滤器和查询(会话计数、花费金额、标签、语言、位置、电子邮件)，您可以从后端卸载相当数量的计算。

另外，它是免费的。有什么问题吗？我们还没找到。至少，我们还没有找到一个巨大的，打破交易的。免费计划包括无限用户和通知，可用于商业或私人应用程序。这项服务是利用通过你的免费账户的数据来赚钱的。需要优先支持？可以报名参加付费企业计划。你可以在这里找到所有的细节。

## 谢幕:结论

当推送(通知)到了紧要关头，我们 Mobile Jazz 会寻找最好的工具，而不是一刀切的解决方案。OneSignal 有一个高级用户界面，专门用于通过仪表板管理和调整通知。它的 SDK 定位于推送通知，比 Firebase 有更一致的标签惯例。我们决定使用 OneSignal 作为推送通知的主要提供商。

也就是说，Firebase 有大量有用的工具。如果你喜欢你的应用程序开发干净整洁，并且都在一个地方，不要再找了。它有推送通知以及其他服务，如远程存储、基于服务器的认证系统、分析等。当我们需要单个信号无法提供的解决方案时，我们会使用它。

通才和专家都有他们的优势。下次有人称你为“绝对的约翰内斯杂役”时，把它当成一种赞美。
