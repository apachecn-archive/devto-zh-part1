# 从工作申请到兼容性列表

> 原文:[https://dev . to/yechielk/from-job-applications-to-compatibility-lists](https://dev.to/yechielk/from-job-applications-to-compatibility-lists)

## [](#how-i-repurposed-my-bootcamp-project)我如何重新利用我的训练营项目

在我学习编码的整个过程中，我非常幸运地得到了我目前工作的经理的全力支持。他鼓励我提升自己的技能，甚至让我在休息时间上熨斗课程。

很自然地，当我毕业的时候，我很兴奋地让我的老板知道了这件事，不出所料，他也为我感到高兴，让我给他看我的最终项目。

当我向他展示我制作的[工作申请跟踪器](http://app-tracker-react.herokuapp.com/)时，他似乎真的很感兴趣，他问我是否可以对它进行一些调整，以帮助我们的集成部门跟踪哪些支付网关和购物车与我们公司的支付处理平台兼容。

到目前为止，兼容性请求是逐个处理的，结果在单独的 excel 表格中跟踪，没有任何中央数据库进行跟踪。如果我能稍微调整一下我的应用程序，它将跟踪支付网关，而不是工作申请。现在，集成团队中的任何人都可以在一个中心位置查看给定的网关或购物车是否已经被研究过，如果没有，记录当前研究的结果。

这听起来像一个有趣的任务。最大的挑战是在我运行 Windows 的工作电脑上建立一个 Rails 环境。最简单的答案是建立一个运行 Ubuntu 的虚拟机，尽管让它与我们公司的防火墙配合并不简单。

最后，我设置了我的环境。那时，只需要两天的停工时间(一天调整 Rails API，另一天更新 ReactJS 客户端),兼容性列表就可以演示了。

有些特征必须改变。例如，在我的工作应用程序跟踪器中，用户只能看到他们自己的工作申请，对于兼容性请求跟踪器，我希望公司中的任何人都能够看到所有的兼容性请求。这涉及到在我的 REST API 中摆弄路由。我的 API 是使用嵌套路由方案构建的，其中一个请求将被发送到`/users/56/applications`以获取属于用户编号 56 的所有工作申请的列表。在兼容性列表跟踪器中，我删除了嵌套路由，现在任何经过验证的用户都可以访问`/applications`来查看所有应用程序的列表。

另一个区别是关于认证。在我的工作应用跟踪器中，我不需要认证就可以查看用户的应用程序；我这样做是为了让求职者更容易与他们的职业教练或导师分享他们的申请。对于我正在处理敏感公司信息的兼容性请求应用程序，我必须收紧要求，以便只有公司内部的人才能查看兼容性请求。

出于同样的原因，我取消了传统的注册表单，实现了一个只有现有管理员用户才能注册新用户的方案。

总而言之，我非常感谢 Flatiron 教会我们编写干净、模块化和经过测试的代码的重要性。进行测试意味着确信底层应用程序不会在我改变特性时崩溃，而将我的代码放在谨慎的模块化组件中意味着我所做的任何改变都是局部的，不需要追踪代码库其余部分引入的 bug。

最重要的是，我要感谢我的老板(你好，莫蒂！)给我机会将我的新知识应用到现实世界的实践中！
