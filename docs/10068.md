# 你的项目还没有完成

> 原文:[https://dev.to/codemouse92/your-project-isnt-done-yet](https://dev.to/codemouse92/your-project-isnt-done-yet)

今天下午，我花了大部分时间和我的助理首席开发人员 Alex 和 Jarek 开了一个规划会议。我们为新的一年讨论了很多细节，尤其是采用库的基本规则。

你看，我们过去在学习图书馆上花了很多时间。我们的一个项目最终因为一个臭名昭著的 XML 解析库而寿终正寝(为了保护相关的有罪开发人员，该库将保持匿名)——我们项目的开发人员不得不投入大约 6 个月的工作来学习这个库。我们的另一个开发人员，从事一个完全不同的项目，不得不投入相同的时间来学习相同的库。

当你在经营一家依靠每周 6 小时实习生的软件创业公司时，那是你不能浪费的时间。

当我们在今天早些时候的会议上讲述这些事实时，我有了一个启示。“学习图书馆是一项不可转让的投资。”

“我们需要设定一个上限，限制我们可以在这种事情上花费多长时间，”Jarek 沉思道。

“我说一个月，”亚历克斯补充说。"对于一个典型的实习生的时间表，那是什么，20 个小时？"

我们都同意这是合理的，但这也带来了采用任何库的新要求:**它必须有写得很好的文档**。我们决定，API 自动文档不算。“这是最没用的文档形式，”亚历克斯说。“你通常需要知道如何做特定的事情，而不是一个函数需要多少个参数。”

我们开始仔细检查我们的图书馆列表，去掉几个，再选几个。会议转移到了其他话题，最后当我们谈到我们自己的文档标准时，又回到了原点。

“也许我们可以建立一个应该经常涵盖的主题列表，”亚历克斯建议道，点了点电脑屏幕上的一些现有文件。"比如，图书馆能做什么？"

“对于每一个‘什么’，我们都需要一个‘如何’，”我补充道。

就在那时，我们集体意识到了一些事情:一个完整的项目必须回答我们的老师过去在小学英语课上问我们的所有相同的问题:谁、什么、哪里、何时、为什么以及如何。

## 一个好的自述

当您下载一个库或查看一个存储库时，您首先要查找的是自述文件。找不到一个是相当令人不安的。我读过不少关于写好自述文件的指南，但我们设法将其归结为回答其中的四个问题:

你的自述文件是人们寻找你的项目描述的第一个地方。你想达到什么目标？是什么让你的项目如此特别？这个问题在其他地方有更深入的回答，但是自述文件应该简单地涵盖它。

谁:这不仅仅是为了获得工作上的荣誉！每个项目都需要一个最终负责发号施令的人或团队。回答这个问题包括提供联系负责人的信息。如何提交 bug 文件、变更和特性请求？

**Where:** 用户会以多种形式遇到你的项目，并不是所有的都非常有用。也许他们捡起了一个旧的源代码包，获得了一个编译好的二进制文件，或者发现了一个过时的存储库。自述文件应该描述找到源代码最新副本的官方位置，以及关于项目和/或其维护者的更多信息的链接。

**何时:**您的用户需要知道何时进行了更改，以及何时发布了不同的版本。作为最终用户，MousePaw Media 经常使用这些信息来确定我们可以使用的库、工具或应用程序的最低版本。这种类型的信息通常放在一个单独的 CHANGELOG 文件中，但我通常认为它是“README bundle”的一部分。

## 写文档！

我见过太多项目的全部文档都是自动生成的技术术语。新闻快讯:除了项目开发人员，API 文档几乎对任何人都没用！

正如我曾经向某人解释的那样，大多数 API 文档就像通过解释加热元件的电气规格来教某人如何使用烤面包机。文档是异步的，所以你不会看到用户在另一边融化:“我不关心愚蠢的加热元件的电压！我只想知道怎么烤面包！”

不要误解我的意思，API 文档有它的位置，但它不能代替真正的文档。

好的文档用简单的英语回答了两个问题:

**什么:**想象你站在用户旁边，描述你项目的所有特性。文档通常是用户看到特性列表的第一个地方。不要抑制你的热情。让他们兴奋起来！没有人喜欢枯燥的文档。

对于每一个 what，你都需要一个 How。如果你的库允许计算机站在它的 USB 端口上，做一个小软鞋，准确描述用户如何能做到这一点！

此时你可能会说“这就是教程的目的！”不完全是。教程描述了如何完成一个单一的，预先定义的目标。文档为使用每一项功能提供了明确、详细的说明，同时为用户提供了根据其特定需求定制说明所需的知识。

我见过许多项目不提供真正的文档，而是与勇敢面对未记录文档并创建教程的好心用户玩“推卸责任”的游戏。坦率地说，这是懒惰的，可悲的，完全不可原谅的。如果你写了代码，你就有义务提供使用说明！不写文档迫使每个用户花费两到三倍于你“节省”的时间，仅仅是为了使用你写的代码。

## 注释你的代码

这就剩下最后一个问题:“为什么？”我相信这个应该可以通过评论你的源代码来回答。

我听到的不写注释的主要理由是“任何人通过阅读就能知道代码做了什么！”你是对的，但是这忽略了评论的全部意义。当然，如果我对编码有所了解的话，我可以看出你在使用一个循环来遍历数组中的每个元素。我无法猜测的是你为什么要这么做。事实上，我可能会猜测你正在使用线性搜索算法(这很可能是一个坏主意)，但由于我无法读懂你的想法，我可能永远不会告诉你改用二分搜索法。

过一段时间你甚至会忘记自己的推理。除了那些拥有完美记忆力的令人惊叹的人，任何吹嘘自己在远离代码六个月后从未忘记自己意图的程序员都是在骗你，很可能是在骗他自己。

好的注释应该描述为什么你的代码是这样写的。您可以用“我们必须使用线性搜索，因为对象是无序的”来注释前面的循环不管是好是坏，你的假设现在已经很明显了。四个月后，其他人(或者你的未来版本)可能会遇到这种情况，并认为“哦！我们只需要重载几个操作符，这些对象实际上将是可排序的！”

正如我们在 MousePaw Media 所称的，注释显示意图也能让新开发人员很快熟悉代码。无论您从哪里进入源代码，您几乎可以立即知道代码应该做什么。除非有评论，否则开始为一个开源项目做贡献总是令人畏惧的，有时几乎是不可能的。

你可能认为不评论你的项目代码是在节省时间，但是你实际上是在混淆你未来的自我，隐藏错误，并阻止新的贡献者加入项目。毕竟不是很省时。

如果你想要有效地注释你的代码的实际步骤和例子，看看 MousePaw Media 的官方[注释显示意图标准](https://standards.mousepawmedia.com/en/latest/csi.html)。

## 你为什么要在乎

我明白了——维护一个好的自述文件、编写文档和注释代码并不光彩，对大多数程序员来说也不是特别愉快。毕竟，没有这些“额外的东西”，数百个开源软件项目已经存活了数年，甚至数十年，对吗？

也许是这样，但是我相信尽管存在这些问题，这些项目仍然存在，完全是基于自由和开源软件的思想。专注的程序员愿意忍受，甚至接受，乏味的文档和稀疏的注释，以获得拥有功能性自由软件的光荣利益！

对于那些项目来说，不幸的是，现在是 2017 年。来自其他承诺相同功能的开源库和应用程序的竞争似乎永无止境，许多相对较新的人在文档和注释领域更强。没有一个项目能够仅仅依靠它的特性集来生存。

我认为，一个成功的软件项目回答了我们在小学学到的六个基本问题:谁、什么、哪里、何时、为什么和如何。所以让我问你——如果你以前的英语老师现在站在你的桌子旁边，他或她会对你的项目说什么？
