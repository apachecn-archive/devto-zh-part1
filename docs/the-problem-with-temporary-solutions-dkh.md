# 临时解决方案的问题

> 原文：<https://dev.to/ben/the-problem-with-temporary-solutions-dkh>

在某种程度上，软件开发中的一切都是临时的解决方案。认为一个实现将永远存在是愚蠢的。但是一些临时的解决方案比其他的更暂时。在一个理想的世界里，我们有时间和精力让一切尽善尽美，但在现实中，这总是一种权衡。临时解决方案是非常自然的事情，如果我们不依赖它们，我们将一事无成。

临时的解决方案往往不是那么暂时的。这是一个问题，不应该如此令人震惊。我们继续前进，却从来没有找到时间或精力回到过去。克服这一点通常需要可靠的测试/监控，以给人们做出改变的信心，以及管理层接受“回溯”重要性的意愿。

但这不是我真正想谈论的问题。更困难的问题是临时解决方案通过模式匹配制造的病毒。在缺乏完美的沟通和领导的情况下，软件开发人员理所当然地依赖模式匹配来构建代码库。这在很多时候都很棒。这是人们如何自己找到帮助别人的方法。理想情况下，开发人员将逐渐对领域有更好的基本理解，但是如果这是贡献的必要前提，我们将一事无成。

当代码库中充斥着临时的修复和实现时，这些就会成为未来贡献的指南，而糟糕的方法就会像病毒一样传播。最初的实现者通常知道他们认为什么是理想的，即使当前的方法不符合标准。但是很容易原谅贡献者的模式匹配，并允许这类贡献带有“我们现在可以保持这种风格，最终我们会补救它”的感觉。这是一条危险的道路。

当贡献代码时，很难摆脱我们现在的心态，但是我们必须把我们的代码方向看作是我们不能完全控制的东西。假设任何合理数量的成功，未来的开发者将会是各种不同的人。经验水平和性格类型会有所不同，我们今天能做的最好的事情就是为自己的成功做好准备，并满怀希望。
