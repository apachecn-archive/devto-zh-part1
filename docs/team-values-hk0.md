# 团队价值观

> 原文：<https://dev.to/craigbuchek/team-values-hk0>

上周我和我的新团队举行了一次回顾会。该团队包括 2 名高级开发人员、2 名初级开发人员、一名产品负责人和一名产品分析师。我以工程经理的身份加入了团队，我认为这更像是一个头衔更高的团队领导。作为这个团体的新成员，我想要一种理解他们价值观的方式。他们的动机是什么？我们有哪些共同的价值观，可以利用这些价值观朝着同一个方向前进？

我以一个非常简单的问题开始:“你看重什么(关于我们正在构建的东西)；你愿意为什么而战？”我让他们每个人写下几个值，然后把它们放在黑板上，再看其他人的答案。我还要求每个人按照重要性排列他们的价值观。

原来我的问题有点模糊。一些人从最终结果的角度考虑问题，一些人从创建软件的过程的角度考虑问题。在某些方面，这是一个问题，因为不同的解释把人们引向不同的方向。但在其他方面，不把他们推向任何特定的方向会得到更多不同的答案，暴露他们对项目和产品的想法。

我的清单(按顺序)是效率(做正确的事情)、质量(第一次就把事情做对)、快乐、目标和团队合作。回想起来，我应该把幸福放在第一位。如果我在工作中不开心，我真的不想待在那里，需要继续前进。(有时候我可以用工作中的一些快乐来换取家里更多的快乐，但我绝对是一个需要工作快乐的人。)其他价值观有助于提高我的幸福，但幸福更重要。

我自己对我的价值观进行排名的问题被证明是这个排名的一个普遍问题。它们应该按照作为结果的价值的必要性的重要性排列，还是按照关注价值的必要性的重要性排列？我认为前者是我一直在寻找的，但当我开始练习时，我也不清楚这一点。

在每个人把他们的价值观写在黑板上后，我们把它们读出来。然后，我要求团队选择几个我们作为一个团队共享的价值观。我们把他们从个人成员列表中抽出来，放在团队列表中。然后我让每个人上来给这些值排序，然后解释为什么他们会这样排序，特别是当他们的排序与上一个人明显不同时。我希望随着时间的推移，等级能够有所收敛，这样我们就可以按照等级顺序记录我们的价值。那没有发生。但是讨论对我和团队中的每个人都有启发。

我认为缺乏融合最有趣的部分是开发人员和产品人员之间的差异。产品人员更看重结果，而不是过程。这是有道理的——他们并没有密切参与产品的构建过程。

不过，我们能够在首要问题上达成一致:用户会购买产品吗？这是我们融合在一起的几个不同价值观的组合。这包括最终用户体验以及确保团队继续存在的理由。我们留下的其他价值是无序的:团队合作(凝聚力)、数据驱动的决策、团队所有权、简单性、有效性、可维护性/可支持性、质量、自动化和性能。我认为这是一份相当不错的清单。

正如一些队友指出的，这些值可能部分反映了当前的时间点。如果我在其他时间，在不同的条件和团队动态下问同样的问题，答案可能会有一些变化。如果再问一次，我们可能会给出其他答案，只是因为我们思考这些事情的方式是随机的。

但是我不认为我会和团队一起做第二次这样的活动。这实际上是关于理解我们的动机——包括我们自己的动机和我们队友的动机。我发现这样做很有效，也有助于团队思考我们的文化，以及我们如何努力塑造它，以帮助我们朝着同一个方向前进。

有一些警告。当我询问对这个练习的反馈时，一位队友指出，如果人们不诚实地对待他们的价值观，这个练习就行不通，他们用他们认为管理层或他们的队友想听到的话来回答。我不认为这是这个小组的问题，但这是要记住的事情。

我要做的另一件重要的事情是事先澄清，我并没有从这个活动中寻找任何行动项目；更多的是了解对方，了解自己。下一次，我将阐明如何对这些值进行排序。

我会为一个新的团队推荐这个活动，或者当团队的组成发生重大变化时。我希望有一种方法可以帮助一个团队在他们的价值观排名上达成一致，但是我想我应该很高兴在一系列重要的价值观上达成一致。想法和观点的多样性可能是我应该拥抱的一种祝福——我们的想法越多，我们可以想象的解决方案的种类就越多。
