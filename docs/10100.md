# 我个人对 TDD 的看法

> 原文:[https://dev.to/grahamcox82/my-personal-take-on-tdd](https://dev.to/grahamcox82/my-personal-take-on-tdd)

我要开始说一些会让人不高兴的话。我个人并不认为 TDD 对单元测试很有效。

现在，不要误解我的意思，TDD 对某些人来说非常有效。我个人认为它在更大范围内非常有效。但是当涉及到编写单元测试时，我发现首先编写测试并不奏效。

这可能是我的 Java 背景的副作用(静态类型，编译语言意味着你不能写单元测试，直到你有东西要测试)

然而，当谈到验收测试时，我认为 TDD 是一个很好的工具。

那么，有什么区别呢？

在单元测试层面，你通常不会一口气写一个单元的代码。你会经常发现你正在写几段不同的代码，它们都可以一起工作，而且在你开发它们的时候，这些不同代码的设计是相当灵活的。对我来说，为这些代码编写测试是违反直觉的。

另一方面，在验收测试阶段，你有希望对你所写的东西以及它将如何工作有一个体面的想法。无论这是 UI/UX，还是 REST API，或者所有这些的外部接口是什么，在编写测试之前，您应该知道这部分是如何工作的。因此，在开发之前，你应该知道你正在测试什么，以及它将如何工作，这意味着你可以先为它编写测试。

所以，对我个人来说，TDD 不太适合单元测试。
