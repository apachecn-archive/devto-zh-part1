# 测试与检查——那又怎样？

> 原文:[https://dev.to/roesslerj/testing-vs-checkingso-what](https://dev.to/roesslerj/testing-vs-checkingso-what)

如果你对“测试与检查”的讨论感到好奇或困惑，这篇文章是为你准备的。我读了詹姆斯·巴赫和迈克·伯顿写的关于测试和检查的文章 。一开始我想“那又怎样？”。然后我想这是二元论的一个例子。现在我更清楚了。这甚至是有意义的。

如果您在测试社区，您可能知道关于测试与检查的讨论。粗略地说,“测试与检查”背后的思想是，检查是一项纯机械的任务，相反，测试只能由人来执行。自从这个区别被詹姆斯·巴赫和 T2·迈克·伯顿传播开来，它在测试界变得非常有名。

当我第一次读到这种区别时，我无法理解。我想“[那又怎么样，我们还是要两个都做](https://www.techwell.com/techwell-insights/2017/06/debate-over-testing-versus-checking)”。比如，除了吹毛求疵，这种区别还有什么好处。它给讨论增加了什么价值？用不同的方式称呼事物并不能改变现实世界中的事实，所以为什么会有人为此烦恼呢？我不是唯一这样想的人:

然后我意识到这种区别是关于一台*机器*做不到而且永远也做不到的事情。嗯，我知道一些关于人工智能的事情，而且我不认为有什么事情是机器永远做不到的*。所以我想，这是[二元论](https://en.wikipedia.org/wiki/Mind%E2%80%93body_dualism)的一个奇怪例子。再说一次，我不是第一个有这种想法的人。也不是[第二个](https://iodegradable.wordpress.com/2014/01/11/testing-vs-checking/)。*

这是我失去兴趣的一点，并把它作为一些古怪的测试人员古怪的事情来看待。但是当我在 test bash 的时候，我尊敬的人指出了这种区别。于是我回到那篇文章，想“我错过了什么”？

## 讨论的核心是什么

结果是我确实错过了一些东西。

在我去过的大多数公司，测试被认为是“低价值”的。测试人员大多是学生，或者更糟糕的是，作为开发人员失败的人。他们“点击应用程序”，通常遵循给定的脚本，并手动检查他们被告知要检查的内容。大多数公司执行这项任务的方式没有什么挑战性或可敬之处。这通常也反映了那些公司在软件质量问题上的立场:软件质量不是那么重要，如果时间和预算允许，可以在以后添加。这些“测试人员”被(错误地)用作“点击机器”。这是一种“可以轻松外包的手动自动化——而且经常如此。

但是有些人和公司把软件的质量放在他们工作的中心。有一整群以测试为荣的人。把测试当成一门手艺的人。如果你做好测试，这是一个原创的，创造性的，具有挑战性的任务。一项试图理解产品的*目的*并试图找到产品可能被误用或改进的方法的任务。这些人离“点击机器”已经很远了。然而，它们被混为一谈。作为测试员。做测试的。就像，你知道，低价值的点击和检查。

我花了一些时间去理解，但是在这个“测试与检查”讨论的核心是真正测试的尊重。以及以贡献高质量最终产品为荣的人的自尊。因此，如果你曾经遇到有人对测试和检查的区别很情绪化*，这个人可能会把自己看作一个真正的测试者，一个尊重这种技术的人。通过区分来表达对他的尊重。*

 *但更重要的是:它还与营销有关。如果你想“销售测试并推广它”,你需要做出区分。(这个术语是由顾问创造的，这绝非巧合。)打个比方:目前的情况是，测试的概念是一个装满沙子和钻石的袋子。当人们需要“销售测试(例如，向管理层或客户)，他们必须销售测试本身。当然，没有人会支付钻石的全价，如果其中有很多沙子的话。所以如果你想卖掉钻石，把它们和沙子分开是很重要的。通过区分，你清楚地表明测试是重要的和有价值的，并且需要专业知识。

本质上，这就是这种区别的意义所在。测试是钻石——一项有价值且重要的任务。另一方面，检查仅仅是一个低价值的机械任务，你可以[更好地自动化](https://dev.to/roesslerj/why-there-is-no-way-around-test-automation-except-one)。

检查只是沙子。
