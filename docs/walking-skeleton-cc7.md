# 骨瘦如柴的人

> 原文：<https://dev.to/brightdevs/walking-skeleton-cc7>

## 什么让你的客户满意？

你知道什么能真正让你的客户满意吗？大量的单元测试？使用 Docker 在每台机器上运行高级集成测试？成千上万行格式良好的干净代码？或者是每个程序员都在谈论的新的开源库？其实都不是。

## 胜负

客户真正关心的是结果。他们珍视他们所能看到和触摸到的东西，他们欣赏看得见的进步，当他们梦想已久的想法实现时，他们会赞叹不已。

这是否意味着所有那些单元和集成测试、持续集成和交付过程、设计模式和干净代码实践都不如一个可见的过程重要？不尽然，每一个成熟的客户都明白，如果你想更长时间有效地为项目工作，这些实践是必要的，但是即使是最有经验的**客户也会更看重结果**。

## 行走的骷髅

当您设置好一切，配置一个健壮的环境，集成高级崩溃跟踪、日志记录、报告和分析系统时，不要从 Sprint 0 开始一个项目，而是从一个执行小型端到端功能的**微小实现开始。**

这种技术被称为**行走骨骼**。Vic Basili 于 1975 年在他的论文[“迭代增强:软件开发的实用技术”](http://www.cs.umd.edu/~basili/publications/journals/J04.pdf)中首次提到了它。行走骨架不会使用最终的架构，但它应该将主要的架构组件链接在一起。

[![image1](img/658111e7ef1470aad3f2a432714a1869.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--tRxV-4YZ--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/767qq1ksysbooat9vec2.jpg)

行走的骨架并不完整，它没有任何“血肉”。它缺少功能，而这些功能是随着时间的推移与基础架构一起逐步构建的。你可以把它看作是你的概念的一个可执行和可运输的轮廓。

行走的骨骼不同于长钉，尽管这种差别很微妙。Spike 要么是功能的一个特定组件，要么是一个系统的粗略草图，以验证这个想法，它旨在回答“我们正在朝着一个好的方向前进吗？”，而且事后就扔了。另一方面，行走的骨架是用生产习惯构建的永久代码，伴随着测试，旨在与系统一起成长。

## 穿上你客户的鞋子走一走

下一次，在运行一个新项目时，试着站在你客户的立场上走一走。永远记住，你的客户把钱花在一个软件上，这可能是他们用梦想中的产品打入市场的唯一机会。

**注意质量**，但也要注意**客户如何看待你的工作**。我们应该永远记住，即使是最先进的技术解决方案，尽管沟通良好，但在长时间内没有可见的进展时，也不会改变坏印象。
