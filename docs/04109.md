# 我们的工具塑造了我们的世界观，或者说是最不糟糕的融合技术

> 原文:[https://dev.to/wiredferret/our-tools-shape-our-worldview-or 最小差汇流技术-6g6](https://dev.to/wiredferret/our-tools-shape-our-worldview-or--least-bad-confluence-techniques-6g6)

如果你和我一起参加过会议，你就会知道每当你问我关于 Confluence 的工作时，我都会做一个恶心的鬼脸。我还没有遇到一个没有让我的灵魂受伤的合流的例子。但是如果我讨厌它，这并不是技术的错，不是吗？难道不是执行的问题吗？工具本质上不是中立的吗？

是也不是。我们谈论固执己见的编程语言，但是每个工具都有一个关于它应该如何使用的意见，一条快乐的道路。例如，WordPress 移动应用程序很棒，我喜欢它在旅途中撰写博客文章——这就是它的用途。但是如果我试图用它来代替 Trello，我会发现作为一个进度追踪者真的很令人恼火。Trello 在任务中思考，很难让它向你展示一个项目的全貌。我们可以像这样遍历我们的整个工具链——我们构建或购买的每一个软件都有它希望被使用的方式，尽管我们可以绕过它，但它总是会被使用。比使用工具时产生更多的摩擦。

* * *

## [](#problem-space)问题空间

在过去的几年里，我去过很多 DevOps 日，我一直看到同样的问题模式。

1.  文档存在于…某个地方，但是没有人能找到它。
2.  人们编写文档，但没有人能找到它。
3.  文档分散在许多地方，没有人能找到他们想要的部分。
4.  标准化或合并文档似乎是不可能的，因为有太多的文档，分布在太多的地方。

这几乎总是发生在合流商店——有些人使用其他维基，有些人有 Word 文件，但这个问题在使用合流的地方确实是反复出现的。

## [](#possible-solutions)可能的解决方案

我有一些关于如何在你的组织中做得更好的建议。

1.  安装一个更好的搜索引擎。合流带来的一个是非常无益的表面当前，相关的数据。它说它在搜索相关性，但返回的结果很难阅读，而且经常过时，以至于没有帮助。
2.  打破你的筒仓。Confluence 固执己见的一个原因是它被设计成在孤岛中工作。发展不会自动看到营销。有时，这是有意的访问控制，有时这只是组织结构图中思考的产物。不管是哪种情况，它都使人们无法在公司内部有效地共享信息。如果你正在建立一个 wiki 或 CMS，尽可能使它平坦和开放。按任务而不是职位来分类。
3.  删除一半左右的内容。如果你做分析，我打赌你会。发现有很多过时的信息。我不知道维基是怎么回事，但它们作为过期信息的墓地是很神奇的。我认为这可能是因为网页会定期刷新，但我工作过的每个维基百科都有一堆旧的、错误的数据。
4.  加入分析。查看人们实际使用的内容，以及他们正在寻找和没有找到的内容。这就是你如何知道你应该把你的第一次努力集中在哪里。
5.  每个文档都有一个明确的目的:是帮助技术支持回答问题吗？是否记录部署过程以增加寻呼机工作深度？为你的文档设定一个明确的目标可以让你根据这个标准检查所有的事情。

## [](#but)但是！

当然，你不可能实现所有这些解决方案。现在不会，也许永远也不会。但无论你是否得到公司其他人的支持，你都可以做其中的一些。定义你的公司、工作和文件的目的。一旦你知道了这一点，你就可以开始朝它努力了。接下来，让分析运行一段时间，并整理出一份报告。

当你意识到人们试图搜索一些已经存在的东西，但是他们找不到的时候，升级你的搜索引擎。一旦你的团队实际上使用了搜索，你可以使用同样的分析来剔除低价值或从未使用过的数据，为更多的数据腾出空间。
