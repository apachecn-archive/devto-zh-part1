# 编写清晰的代码，而不是聪明的代码

> 原文：<https://dev.to/mikesherov/writing-clear-code-not-clever-code-1lgc>

在《敏捷开发人员的实践》一书中，安迪·亨特和文卡特·苏布拉马年列出了一大串提高软件开发人员自我的方法。虽然我发现书中几乎所有的技巧都很有用，但我发现有一个技巧非常突出，因为它指出了我的一个特别坏的习惯:“编写清晰的代码，而不是聪明的代码”。

作为一名程序员，我的激情之一是找出新的更好的方法来做同样的事情。我喜欢重构代码，用 10 行代码就能发现正在做的事情，而我用 3 行就能做到。或者作为一种更好的方法，用一个 if else 块代替三重嵌套的 if else 语句。通常这很管用。代码更易于维护，对所有相关人员来说也不太复杂。

然而，在我不断追求更小、更清晰的代码的过程中，我经常会得到更小、更混乱的代码。2009 年，作为一个 7 人敏捷团队的成员，很容易衡量我在这一类别中的表现:另一个团队成员多久问我一段代码做了什么？当我开始查看这些有问题的代码块时，我注意到了一个模式，并开始着手解决这个问题。我的特殊问题是什么并不重要…重要的是我开始寻找它！

关注你的团队成员。调查他们为什么询问你的代码。也许你做的一些事情对你来说很有意义，但是可以写得更清楚。**考虑下一个将要看你代码的人。**很有可能，他们会更乐意看到更清晰的代码，而不是聪明的代码。你也会更快乐。
