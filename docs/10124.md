# 你怎么知道一个开发者做得很好？

> 原文:[https://dev . to/pbeekums/how-do-you-know-a-developer-is-doing-a-good-job](https://dev.to/pbeekums/how-do-you-know-a-developer-is-doing-a-good-job)

对于任何参与软件开发的人来说，对开发人员的评估都是一个重要的话题。许多开发人员非常关心职业发展或至少是加薪。经理需要能够证明有关晋升和加薪的决定是合理的。

[![](img/b61e97c0d1d41d059360baf5e57d5392.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--aHK-Ulrs--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/zzeumcubn43l65fc8xt3.jpg)

理想情况下，开发人员将根据他们为企业提供的价值获得报酬。这是一个简单的概念:你让公司赚更多的钱，公司付给你更多的钱。

复杂性在于软件开发的本质并不容易衡量单个开发人员带来的价值。如果你有一个拥有数百个功能的产品，并且一年能赚 1000 万美元，你怎么知道其中有多少价值是由一个功能贡献的呢？即使你能做到这一点，这个特性也很可能是由多个开发人员共同开发的。你如何评价每个人的贡献？

这个问题没有一个好的答案。试图进行这种测量是不现实的。留给我们的是经理们的主观判断电话。

这种方法的风险在于，从开发人员的角度来看，标准是模糊的。他们不可能真正知道他们的经理在想什么。现在有一种激励，让他们专注于学习向经理推销自己，而不是专注于成为一名更好的开发人员。

这个问题的一个有吸引力的解决方案是**量化指标**。如果我们不能测量一个开发人员给公司带来的确切价值，也许我们可以找到一些我们可以很容易测量的东西，这将是一个很好的创造价值的指标？

这里最天真的方法是计算开发人员编写的代码行数。对于那些不是开发人员的人来说，这可能是有意义的，至少一开始是这样。开发人员编写代码…所以也许更多的代码意味着更多的价值？

这里的问题是，高质量的代码通常意味着更少的代码行。一个非常简单的例子是:假设你想在你的电子商务网站上展示 100 种产品。你可以这样做:

> display product(1)
> 
> display product(2)
> 
> display product(3)
> 
> **…. 97 行以后…**
> 
> display product(100)100 个产品，100 行代码！

没有一个像样的开发者会这样做。

他们会改为两行代码:

> 对于 productNumber = 1，product number<= 100, productNumber++:
> 
> display product(product number)如果你向你的开发者支付每行代码的费用，你就会**奖励那些效率低下的**开发者。

与此类比的是写散文、小说、博客帖子等。你会仅仅根据一个作家写的字数来评价他吗？大概不会。理解一个复杂的观点需要最少数量的单词，但是当一个作家用无用的句子把他们的作品弄乱时，这些观点就会丢失。我知道我读过一些 3 页纸的好文章，但最终都是乏味的书。

所以代码行度量不起作用。不过，评估开发人员的量化指标的概念仍然很有吸引力。

有些人可能会认为创建许多代码分支是一个伟大开发人员的标志。然而，我曾经和一个开发人员一起工作过，他会创建代码分支来隐藏他效率不高的事实。

有些人可能会说，您可以度量开发人员提交的代码数量。但这只是让它更有可能获得许多非常小的提交，这些提交中几乎没有任何有意义的更改。或者，开发人员可以通过撤销和重做相同的几组更改来显示提交计数，从而轻松地解决这个问题。

也许修正的错误数量就是答案？尽管软件错误可能很复杂。通过将加薪和晋升与 bug 修复捆绑在一起，您为开发人员创造了修复表面问题的动机，而忽略了潜在的原因。这就像用胶带来固定断腿一样。最糟糕的情况是，你会奖励那些一开始就没有特别注意预防错误的开发人员。

像这样的量化指标的真正问题是它们改变了开发人员的动机。虽然目标是创造商业价值，但是对于开发人员来说，很难表现出他们正在创造商业价值。不管这些指标是否为业务带来价值，仅仅针对其他可量化的指标进行优化要容易得多。这对任何人来说都是不可取的。

我们仍然面临着一个挑战性的问题:你如何知道一个开发人员是否做得很好？

在我看来，最好的衡量标准都是主观的:

*   他们是好的团队成员吗？
*   他们能解决问题吗？
*   他们能写出好的代码吗？
*   他们渴望学习吗？
*   你能信任他们吗？

还有其他的，但重点是好的开发者的每一个衡量标准都是定性的。这需要经理们经常与向他们汇报的每个开发人员交流。管理人员必须创造一种文化，让开发人员相信他们会因为做正确的事情而得到奖励，而不是看起来像是在做正确的事情。

量化指标之所以吸引人，是因为它们简单易行。但是管理没有一件事是容易的。如果是的话，任何人都可以做到。
