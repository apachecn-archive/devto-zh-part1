# 如何写一份扎实的 dev 简历成为 2018 明星

> 原文:[https://dev . to/agazaboklicka/how-to-write-a-solid-dev-resume-to-be-2018-star-3h8a](https://dev.to/agazaboklicka/how-to-write-a-solid-dev-resume-to-be-2018-star-3h8a)

*这篇文章来自[我的博客](https://jumpstart.blog/2018/01/21/how-to-write-solid-dev-resume-to-be-2018-star/)，在那里你可以找到我的简历(或多或少)。感兴趣的话 pdf。*

自 11 月底以来，我一直在指导 IT 人员求职和撰写简历。我注意到，我们很多人都在努力(是的，我也是)写这张可怕的纸。所以这里是我到目前为止给出的知识和最常见的技巧的汇编。

### 接吻——保持简短

简历是你鼓励招聘人员或招聘经理面试你的营销材料(而不是一篇关于你的经历/项目的详细文章)。你的简历展示了你申请的职位最重要和最相关的技能。我不是让你写一页纸(这个信息早就过时了)，但是两页纸足够给别人你是谁和你知道什么。

### 使它坚固(如鲍勃大叔的坚固)

#### [](#single-responsibility-principle)单一责任原则

一份简历对应一个职位/一份招聘启事。IT 架构师的简历与高级软件工程师的简历会有所不同。对于 ABC 公司的高级软件工程师和 XYZ 公司的高级软件工程师来说是不同的

#### [](#openclose)开/关

通过打开扩展来关闭修改:你会想通过在求职信中添加额外的信息来扩展你的简历(是的，我知道在打开/关闭中不是 100%像这样，但我想你能理解；))

#### [](#liskov-substitution)里斯科夫换人

*“在不改变程序正确性的情况下，程序中的对象应该可以用它们的子类型的实例来替换”*
意思是:如果你在低年级时就能做某事，那你在更有经验的情况下也能做。
干在这里派上了用场。所以请**不要重复自己**。

不过，这里有一个例外。如果你在每份工作中所做的是你申请的职位的基本要求，那么在你的职业生涯中展示出来是很好的。例如:应聘 JavaScript Dev 描述所有 JavaScript 开发角色/任务。调整它，使它们看起来不一样，例如:

*软件开发商，ABC 公司*

*   开发、维护、调试和修复分布式 Java 和 JEE 应用程序中的错误

*软件开发商，XYZ 公司*

*   参与基于 Java、Struts、JavaScript、Hibernate 和 DB2 的全栈软件开发

#### [](#interface-segregation-principle)界面偏析原理

许多特定角色的简历比一份普通简历要好。(没什么可补充的了)

#### [](#dependency-inversion-principle)依存倒置原则

*“高层模块不应该依赖低层模块。两者都应该依赖于抽象。抽象不应该依赖于细节。细节应该依赖于抽象。”*

你的抽象概念就是你申请的工作，所以你写的所有东西都应该基于那份工作。只保留工作描述中要求的细节。再次吻在这里也很好:)

### [](#start-with-master-resume)从主简历开始

一份主简历就像一个数据库，包含了你迄今为止所做的一切。这是你未来简历的一个很好的参考点，列出了所有或最重要的技能和成就。

### [](#accomplishments-instead-of-tasks)成就代替任务

很少有比仅仅列出你的职责更好的成就了。例如:
**任务**:编写 JUnit 测试
**成果**:通过编写 JUnit 测试确保应用程序的预期行为，并确保 85%的代码覆盖率

你可以找到很多关于如何将任务转化为成就的普通文章，比如这篇关于缪斯的文章。

如果你仍然不确定如何去做，这里有一个小建议:

### [](#be-a-star-of-your-resume)成为简历中的明星

想一想当你执行那个**任务**时的**情景**。
你采取了什么**行动**结果**是什么**？

然后用 POR 格式写:
**P** 时态动作动词
**O** 陈述的宾语
**R** 作为你动作的结果

并尝试在中间添加一些关键词。

例如:*指导一个由 4 名开发人员组成的团队使用**看板**来设计、配置和测试 **RESTful** 忠诚卡管理系统运行在 **RedHat Linux*** 上
