# 不要用“MVP”作为借口去写不好的软件

> 原文：<https://dev.to/ben/dont-use-mvp-as-an-excuse-to-write-bad-software>

如果你正在读这篇文章，你的代码库可能是你的企业或你工作的组织的命脉。作为少数初创公司的技术联合创始人，我们为了编写糟糕的代码而构建“最小可行产品”的借口对我来说并不陌生。诞生了 MVP 概念的[精益创业方法](http://theleanstartup.com/principles)给人留下了这样的印象，即通过有限的功能集快速进入市场，以便了解更多信息并重复反馈。这个概念很受欢迎，对创业公司和更大的组织都有很大的影响。相同的基本概念显然已经成为软件开发的共识。敏捷软件管理提供了同样的智慧——唯一确定的是不确定性，我们不需要过早地规划出整个解决方案。

我看到的问题是，人们把“精益”和“敏捷”的概念作为借口，在承担大量技术债务的同时，简单地把事情赶出门外。我理解导致这种做法的情况:时间是一种稀缺资源，而你的资金不是无限的。然而，我认为问题在于，人们实际上并没有构建最小可行的产品，用有限的功能集来完成工作。他们正在极其不稳定的基础上快速构建功能完整的解决方案。

“功能蔓延”总是潜伏在角落里，它需要极大的纪律来避免。当一个组织，无论大小，要求现有竞争对手的高级特性集，同时仍然坚持这必须在两周内推出的想法时，程序员经常不得不编写糟糕的代码来完成它。

如果你是一个团队中的非技术部分，你真的很难理解维护和调试这种系统会有多麻烦。如果这种负担过于频繁地强加在程序员身上，他们会精疲力尽并放弃，而组织的负担是蹩脚的代码将不可能改变。当程序员创造奇迹，在几周内完成整个应用程序时，她可能是通过偷工减料和构建一个非常不稳定的系统来实现的。当初始版本的反馈回来时，必须进行更改，这些更改将难以置信地难以融入应用程序运行的意大利面条式代码中——更不用说以关键错误的形式灭火所需的时间和精力了。尽管软件最初很快就发布了，但是早期缺乏对质量的关注注定了这个项目的失败。如果你抵押未来去做，你可以在很短的时间内完成惊人的数量。

由于不确定性，一些技术债务——即任何软件系统设计的最终结果——是不可避免的。没有一个软件是没有缺陷的，但是作为一个指导原则，产品开发人员应该尝试限制功能需求，以便减轻技术基础设施的负担，而不是为了使软件功能完整而在技术基础设施中偷工减料。在实践中，限制项目中特性的范围有几种基本形式:立即消除对特性的需求，或者使用外部服务向您或您的客户提供特性。不考虑 focus 的技术优势，制造手术刀而不是瑞士军刀是一个很好的商业隐喻。

重点是培养一个欣赏好代码的环境。这并不意味着永远不发货或者害怕把你的软件放到野外。程序员应该[接受 bug](https://m.signalvnoise.com/software-has-bugs-this-is-normal-f64761a262ca#.x8c4zkov1)的存在，并被鼓励尽早、经常地发布他们的代码。如果产品决策与良好的代码理念一致，将会取得长期的成功。好的和坏的软件之间的界限是任意的，合理地说是不可能定义的，但是我们为了培养好的或坏的代码而设置的约束更容易交流和判定。关注这些限制，MVP 将会及时推出，项目将会足够灵活，你将会实际上能够根据策略要点的反馈采取行动。
