# 最有效的软件开发团队

> 原文:[https://dev . to/pbeekums/the-most-effective-software-development-team](https://dev.to/pbeekums/the-most-effective-software-development-team)

在构建软件时，我想了很多关于团队应该如何构建的问题。许多公司选择通过纪律来区分他们的团队。有开发团队。UX 团队。产品团队。质量保证团队。支持团队。营销团队。

[![](img/a97615ea67378b54ec8f0b939f75def6.png)T2】](/img/2017/1/team.png)

一开始是有道理的。特定领域的人员经理应具备该领域的工作经验。你不会希望一个开发商管理 UX 的设计师，反之亦然。

然而，这种方法存在许多挑战。首先是确保所有团队的目标一致。开发人员通常有动力提前做好评估并快速发布产品。他们的**经理经常有这种期望**不管是好是坏。

QA 测试人员的动机是产品不会发布 bug。他们的工作毕竟是保证质量。如果这意味着推迟发布以确保万无一失，那就这样吧。让一个 bug 通过就好像他们没有做好工作一样。

UX 设计师有动力做出最精美的产品。他们通常只根据产品的外观来判断。这意味着他们的偏好总是花更多的时间让事情看起来更好。

这些目标对于孤立的学科都是有意义的。请注意利益冲突。在软件开发中，我们可以关注发布日期、质量或特性。对于一个复杂的产品，只能完成其中的两个重点。没有什么是完美的，你不可能拥有一切。

这意味着**一个学科不会得到他们想要的**。你如何决定哪一个？很少有人公开谈论这一挑战，因此决议来自办公室政治。

我所在的最有效率的团队结构非常不同。每个需要交付软件的人都在一个单一的跨职能团队中。没有独立的开发团队、艺术团队或质量保证团队。我们开发游戏，所以我们有游戏团队。该团队由开发人员、美工、测试人员、游戏设计人员和项目经理组成。我们没有一个艺术家团队完成他们的工作，然后交给开发人员。大家一起做软件**。**

 **这种结构做的最重要的事情是它使每个人的目标一致。如果有一个截止日期，这个截止日期并不属于某一个学科。它属于整个团队。如果速度是公司最重要的事情，那么我们专注于更快地发布产品。如果让一个功能正确是公司最重要的事情，那么我们花时间让它正确。

没有“为什么游戏设计师总是想要不可能的功能？”没有人问“为什么开发人员这么难打交道？”

没有“为什么艺术家们不能齐心协力？”

只有一个团队。我们一起成功，也一起失败。

这种团队结构也有助于更好的沟通。艺术家和开发者没有坐在大厅对面或不同的楼层。团队坐在一起，这意味着开发人员就坐在艺术家旁边。考虑到这项工作是如何紧密联系在一起的，这是一个巨大的恩惠。没有时间浪费在等待别人回答关于工作应该如何整合的问题上。你只需要转过头。通常一个人的 5-10 分钟可以为另一个人节省一个小时甚至更多。这就是赋予企业的价值。

有趣的是，当时我们都不是各自领域的佼佼者。如今，我们在各自领域的效率都比过去高得多。然而，我从来没有在一个团队中能够如此快速地交付高质量的软件。

软件开发是一个复杂的过程。个人所能做的有严重的限制。这意味着个人能力能让你走多远也是有限度的。良好的团队合作没有限制。

[![](img/42eaf02039dcba362068ff5045fc78ad.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--WIKjAGdK--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://blog.professorbeekums.com/img/2017/1/sky.jpg)

也就是说，这种结构并不完美。最后你还是希望开发者向开发者汇报，艺术家向艺术家汇报，产品经理向产品经理汇报。开发人员很可能无法帮助艺术家提高他们的艺术技能。产品经理很可能无法帮助开发人员提高他们的开发技能。

我所在的跨职能团队工作正常，因为该公司高峰期有 36 人。这是一个足够低的数字，每个人都向公司的三个合伙人之一报告。每个团队都有自主权去做他们认为合适的事情，但是总会有一个经理来帮助特定学科的人在那个学科中成长。我还没有看到这种模式扩展到更大的公司，但我可以很容易地看到报告结构变得非常复杂。

尽管面临挑战，我还是无法忘记我们的工作效率。从那以后，我一直试图复制这种团队文化，但都失败了。我认为，让报告结构正确的困难是我遇到如此多阻力的原因。毕竟坚持现状要容易得多。

你认为怎样才能成为一个高效的团队？我很想收到你的来信！
