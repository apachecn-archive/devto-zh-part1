# 移情程序员

> 原文:[https://dev.to/dwestheide/the-empathic-programmer](https://dev.to/dwestheide/the-empathic-programmer)

*这篇文章最初发表在[丹尼尔·韦斯特海德的博客](http://danielwestheide.com/blog/2017/01/16/the-empathic-programmer.html)上。*

1999 年，安德鲁·亨特和迪夫·托马斯在他们的开创性著作中要求程序员要务实。十年后，查德·福勒在他关于职业发展的优秀著作中，要求程序员成为[激情](https://www.goodreads.com/book/show/6399113-the-passionate-programmer)。即使在今天，我仍然认为这两本书里的许多建议非常有价值，尤其是福勒的书，对我个人帮助很大。

然而，近年来，我一次又一次地看到程序员的另一个品质至少同样重要，但它甚至没有得到应有的重视。我们都应该努力成为的程序员是*移情程序员*。当然，我不是唯一意识到这一点的人，更不是第一个。首先，在我的泡泡中，Benjamin Reitzimmer 不久前写了一篇关于他认为成熟开发人员的[重要品质](http://squeakyvessel.com/2015/05/12/mature-developers/)的精彩文章，移情是其中之一。我认为缺乏同理心是我们行业和技术社区中一些最大问题的根本原因。在这篇文章中，我想分享一些关于缺乏同理心如何导致问题的观察。把它当作一个呼吁，去争取更多的同理心。

那么什么是共情呢？这里有一个来自韦氏词典的定义:

> 理解、意识到、敏感地感受和体验他人过去或现在的感受、思想和经验的行为，而没有以客观明确的方式充分交流这些感受、思想和经验；还有:这方面的能力

## 职场中的移情

毫不奇怪，在任何涉及与其他人合作的工作中，包括程序员的工作中，表现出同理心的能力都会派上用场。即使你远程工作也是如此——你在空闲频道看到的其他消息并不都是来自机器人。他们背后有真实的人类。

我们经常忘记考虑的情况之一是代码审查。仅仅写下拉式请求的错误，而不考虑语气，很容易导致拉式请求的创建者感到被冒犯了。April Wensel 有一些关于代码评审的好建议。至关重要的是，对你的话将如何被接受者理解培养一些敏感性，这需要你站在他们的角度，通过他们的眼睛看问题，并反映他们的感受。你越了解这个人，这就越容易，否则你将不得不做出一些假设，但我认为这仍然比完全不考虑其他人的感受要好得多。

我经常看到的另一种缺乏同理心的工作场合是，当两个不同团队的成员需要合作来解决一个问题或完成一个功能时。在一些公司，我看到了团队之间奇怪的、竞争性的“我们对他们”的态度。这种现象已经被社会和进化心理学家探索过，虽然这种行为可能仍然存在于我们的天性中，但这并不意味着我们不能试图克服它。“我们对他们”的一个变体是“开发人员对管理人员”。我们开发人员很难理解为什么管理人员要做他们所做的事情，但是坦率地说，通常我们并不努力。我经常看到开发人员对经理采取非常防御性的姿态，当然，在这些情况下，经理和开发人员之间的关系相当冷淡。了解“另一面”肯定会有助于与管理者产生共鸣。为了和他们建立健康的关系，理解他们为什么以特定的方式行动是绝对必要的。

## 科技界的同理心

同理心不仅在你的工作场所很重要，而且在你与我们社区中的其他人互动时也非常重要，无论是在邮件列表、会议上，还是在与你的开源库的用户或你正在使用的开源库的开发者交流时。在某些情况下，缺乏同理心会加剧排斥，最终导致一个被视为精英和傲慢的封闭社区。

作为一名使用开源库的开发人员，在你开始抱怨一个 bug，或者更好的是一个缺失的特性之前，请与库的开发人员产生共鸣。山姆·哈利迪写了一篇有趣的文章，名为[开源权利综合体](https://medium.com/@fommil/the-open-source-entitlement-complex-bcb718e2326d#.qtqmmnul7)。很难相信，但是很明显，许多开源库的用户都有这样的态度，认为这些库的开发者是某种服务提供者，乐意免费做你想做的事情。事情不是这样的。就像措辞和语气在代码评审中很重要一样，试着理解那些把空闲时间花在你使用的这个库上的开发人员。因为你没有阅读文档而为你服务和帮助你，这可能不是他们生活中的最高优先事项，所以不要把它们当作是最高优先事项。

另一方面，当向潜在用户展示您的开源库时，要考虑这些人对该展示的感受。这让他们感到被尊重了吗？这让他们感到受欢迎吗？很抱歉让你失望了，但我认为一个“为通情达理的人”开的 foo 酒吧不会有那种效果。就我个人而言，我觉得这是非常居高临下的，并认为这将恐吓很多人，把他们拒之门外。这意味着除了你的方法之外的任何其他方法都是不合理的，因此，那些还没有使用你的库，而是使用其他方法的人是不合理的。作为图书馆的作者，让我们也对我们的潜在用户表现出一些同理心。和往常一样，在科技领域，没有什么灵丹妙药，需要权衡取舍。可能有很好的理由来解释为什么有人到目前为止一直在使用不同的库，甚至在看了你的库之后，仍然有很好的理由不使用你的库。即使你确信你的图书馆好得多，你也不能通过告诉访问你的项目页面的人他们使用其他任何东西都是不合理的来创造一个开放和友好的氛围。

如果你在一次科技会议上，在对话一开始就问女性她们是不是开发人员，但没有问男性同样的问题，你这么做可能不是出于恶意，而是因为你在科技会议上没有看到很多女性实际上是开发人员。然而，对于接受者来说，这个看似无害和中性的问题却完全不是那么回事。她可能已经听过这个问题很多次了，不断听到对你是否真的是程序员的怀疑并不会让你感到受欢迎，或者自信。在科技会议上和其他人交谈时，表现出一些同理心。例如，想象一下不断被怀疑会是什么样子。如果你看不到包容的必要，那可能是因为你在融入社区方面没有问题。这可能意味着你是一个男人，可能是白人。因为你周围的大多数人都像你一样，所以你甚至可能不认识任何女性或其他无特权的开发人员。享有特权的问题在于你没有注意到它。在会议上与女性交谈，让她们告诉你她们的经历。通过表现出同理心，你可以创造一个更友好的包容环境，培养多样性。

## 总结

这是我对技术社区中同理心和缺乏同理心的两点看法，以及它与包容性和多样性的关系。同理心不仅在工作场所与同事互动时很重要，在我们作为会议访问者、开源开发者和开源图书馆用户参与技术社区时也很重要。只有表现出同理心，我们才能创造一个包容和开放的社区。让我们试着更加意识到我们对彼此的影响，并采取相应的行动。谢谢！
