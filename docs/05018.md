# 错误和教训。

> 原文:[https://dev.to/svemaraju/mistakes-and-learnings](https://dev.to/svemaraju/mistakes-and-learnings)

我已经在现在的公司工作一年多了。经历了许多起起落落。我知道这是老生常谈，但这是事实。在这篇文章中，我决定认真反思我在这一年中犯下的一些错误，并试图找到一些对未来和读者有帮助的见解。

出于一些背景，我一直在我目前的公司做后端开发，主要是用 Python 和 PHP 开发 web 服务，为我们的[酒店客房预订产品](http://app.selectrooms.co)处理各种操作。虽然我以前做过数据库程序员，但这对我来说确实是一个新的阶段。这是我第一次被委以将一个系统(尽管很小)作为一个整体来工作的责任，而不是在一个巨大系统的许多部分中的一个上工作。对数据库建模，找出合适的身份验证方案，尝试各种工具来解决问题，学习不同的框架和语言，了解现有的代码库，以及决定部署策略。回顾过去，就我的职业发展而言，过去的一年是非常充实的一年。但是大部分的学习来自于犯很多错误，有些是多次的。我将谈谈其中的一些。

我不知道这是否适用于每个人，但我也曾犯过一些直截了当的“愚蠢”错误。我可以把这些归因于大脑放屁，继续前进，不再为此烦恼。但是我认为那对我来说会是一个错过的学习机会。

在开始写代码之前配置不同的环境，是我已经学到并且还在学习的最重要的一课。

截断生产模式中的表，而不是 dev (aws 备份来拯救！)，将错误的数据库配置到错误的环境，绞尽脑汁找出请求失败的原因(这种情况发生了多次)，在服务器上设置 cron 作业，该作业在工作时间后关闭(这非常尴尬)。所有这些问题的一个共同因素是在执行前缺乏充分的规划。

当我开始从错误中吸取教训时，我开始写下事情，在执行哪怕是一个命令之前明确行动要点。这是很多人默认知道和认可的东西，但我是通过艰难的方式学会的。[清单宣言](https://www.amazon.com/Checklist-Manifesto-How-Things-Right/dp/0312430000/ref=sr_1_1?ie=UTF8&qid=1504723159&sr=8-1&keywords=The+Checklist+Manifesto)谈到为紧急情况准备清单的习惯，我开始在每次部署之前准备一份清单，除了尽可能自动化之外，还会有很多事情可能出错。

还有一些其他的例子，当我在我的代码中有一个重复出现的问题。这可能是环境问题(操作系统或数据库)，与语言相关的问题(我的例子是 Python)，与设计相关的问题(不知道反模式)等等。

在从初级水平过渡到中级水平的过程中，我感觉到了很多惰性。有些事情我已经习惯了，比如以一种特殊的方式处理逻辑，这在编程标准上是非常原始的。打破这种惯性需要一些努力。在一个小团队中工作可能会很棘手，因为没有机会向其他开发人员学习。承担处理整个后端系统的任务，我经常牺牲代码质量来让它工作。

我最终意识到坚持既定模式的必要性。但是真正的斗争是克服对这种模式的无知。当你不知道如何定义你的问题时，你就找不到正确的解决方案。这本关于[设计模式的书](https://www.amazon.com/Design-Patterns-Elements-Reusable-Object-Oriented/dp/0201633612/ref=sr_1_1?ie=UTF8&qid=1504723873&sr=8-1&keywords=design+patterns+gang+of+four)已经成为我的问题的正确答案。尽管书中的东西并不都适用于我的日常工作，但意识到它们的存在已经给了我很大的信心。

我很想听听你的自我改进技术，特别是如果你是一个像我一样在小团队中工作的自学型开发人员。
