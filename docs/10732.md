# 现场报告:榆树

> 原文:[https://dev.to/jskulski/field-report-elm-1l26](https://dev.to/jskulski/field-report-elm-1l26)

几年前，我带着 Elm 去兜风。

### [](#elm)榆树？

简而言之，Elm 是 ML 系列中的一种函数式语言，可以编译成 javascript 在浏览器中运行。它擅长创建可伸缩的 web 应用程序。榆树启发 Redux 的部分。心理模型是相似的。

这个社区在 http://elm-lang.org/做了很好的解释工作

### 我的经历很短暂，但我喜欢我所看到的。

我花了几天时间和 Elm 一起制作游戏《蛇》。我发现这个游戏对于学习新语言或工具的用户输入和事件是一个很好的练习。所以我试了一下:

算是成功吧！我还在努力，但是代码可以在这里找到。

Elm 是一个令人兴奋的项目。它有许多有利条件。Rishi Goomar 先生在他的文章“[为什么我认为 Elm 是前端开发的未来](https://medium.com/@rgoomar/why-i-think-elm-is-the-future-of-front-end-development-21e9b091fa05#.ns0pjprhf)”中阐述了许多技术方面。

我也看到了榆树的许多前景。

### [](#developer-experience-is-a-first-class%C2%A0citizen)开发者体验是一等公民

我的经验是快速反馈和最小的上下文切换。我不是在看编译器错误，就是在玩我的游戏。尽管我刚刚开始学习，但我觉得我可以在每一步都取得有意义的进步。

令我震惊的是“*没有运行时异常*”的说法竟然是真的。当我引入我没有预料到的行为时，那是因为我的心理模型是错误的(行、列与列、行等等)。没有副作用的虫子。

因为这个大胆的声明，我跳过了编写测试。我想看看它能坚持多久。没有退步。每次我添加一个特性时，它都会停留在那里。

编译器消息详细而准确。我唯一感到失落的时候，是我没有正确地阅读编译器消息。事实就是如此。

vscode 和它的 elm 插件。如果没有配置，这些编译器消息会在您编写代码时出现，而无需运行命令或切换到另一个窗口。

Elm 的既定目标是将函数式编程的好处带给大众。这需要开发者的用户体验，是一等公民。榆树提供了这一点。

### [](#in-the-real%C2%A0world)现实世界中？

Elm 是学术性的(是)还是可以在现实世界中使用？很棒的问题！以下是我的一些想法。

#### 一些公司正在报告重大胜利

Futurice 在这里报道了他们的转变。自从采用以来，还没有红墨水被大肆使用。这个话题是在接下来的开始讨论问&的一个专题小组:

#### [](#limits-talent%C2%A0pool)限制人才库？

我的一个担忧是限制了要雇佣的候选人的数量。工程团队需要成长，如果工人数量很少，这可能是一个业务问题。你必须通过竞争来培训或雇佣员工。有趣的是，没有红墨水(在上面发布的视频中)报告说，在“大声使用榆树”之后，候选人的质量飙升。

是不是很不可思议？限制你的候选人才库和你想雇用的人数增加？

我觉得这很有趣。保罗·格拉厄姆在他的文章《Python 悖论》中谈到了这一点。对我来说，这有点滑稽。伟大的工程师希望用伟大的工具工作，所以他们去他们能去的地方。

但是，这只在有限的时间内成立。 /参考文章，在足够长的时间线上，python 是新的 java。

#### [](#similar-to-reactredux)类似 React/Redux

由于 Elm 本质上是在语言层面上实施的 React/Redux，我可以看到一个团队获得了很大的生产力。它是同样的工具，具有更好更快的反馈，防止反模式，而不限制您可以构建什么。

转换的成本可能相当低。我在 react 和 redux 中开发了几个应用程序，所以我有几个小时的‘哦，这就是那个’时间，然后我就去比赛了。我想，对于一个习惯于反应/还原的团队来说，这也是类似的。

当然，从 react/redux 到 elm 重新编写一个应用程序没有什么商业意义。有许多工具可以对 react 应用程序进行零碎的修改，包括运行 Elm 应用程序的 React 组件。(反之亦然)这样你就可以分块工作。

#### [](#young-language-and-has-limited%C2%A0tooling)年轻的语言和有限的明面

Elm 仍在摸索中。例如，在 v0.17 中，他们将一个中心概念的命名从“动作”改为“消息”。一个小的改变，但却是 Html 心智模型的主要部分。语言的 App 部分)。当然，把这些事情做好是件好事，但会显示出年轻。

工具是很棒的，但是由于这个项目只有 4 年的历史，工具可能很少。例如，我使用了 VSCode，这是一种很好的体验，但缺乏正确的语法文件，所以我盯着大量带有蓝色变量名的白色文本。随着时间的推移，情况会有所好转。

#### [](#getting-stuck-with%C2%A0elm)被榆树卡住

我最担心的是收养 Elm，它已经‘完了’。现在你有了一个没人用的*语言*的代码库。

现在它是开源的，规范已经存在，所以你可以随时编写自己的工具链。但是由于非常合理的原因，大多数 web 应用商店不想支持一种语言。所以，让我们说，采用将取决于持续的社区或大型 facebook 风格的公司支持。

所以让我们把这一个看作是全面风险。虽然这个社区充满活力和活力(在闲暇时聊天，活跃的聚会，会议)，但它年轻，规模小，所以这一切都有可能消失。

也就是说，我认为一个企业的工程团队能够可靠地部署 web 应用程序，而不会出现错误或倒退，这是一个/显著的//竞争优势。埃尔姆承诺。

#### [](#active-friendly-community)活跃的友好社区

我在上面提到了社区，但是我想强调这是多么大的一笔资产。社区是采用+成功的重要指标。

我发现这个社区热情、乐于助人、活跃。文档和教程是全面的，选择的模块(大部分)是显而易见的。对 slack 频道的帮助是卓有成效的和支持性的。技术和设计问题的答案很快就出来了。谢谢大家！

#### [](#conclusion)结论

我想在一个小应用程序中尝试 Elm，并与一个团队一起测试这些说法。

目前，没有太多的动力去远离 React/Redux。它工作得很好，我们有点，嗯，刚到这里，伙计。选择榆树有不小的风险。所以搬到榆树批发没有意义。推/拉的东西不相等。这让我加入了早期采用者的阵营，我对此没有意见。更高层次的优势。

在 Elm 的大部分时间里，我感觉自己被引导向解决方案，而不是被问题困住。反馈是即时的，我从来没有感到卡住。当我想改变结构以适应新的问题时，我只是这样做，并使程序返回到工作状态。

你能想象没有运行时异常、最小回归和无缺陷重构的整个团队的生产力吗？
