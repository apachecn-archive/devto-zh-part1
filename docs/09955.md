# 它是跨平台的，大大降低了应用程序开发成本，但 WTF 是 React Native 吗？

> 原文:[https://dev . to/kylessg/its-cross-platform-and-massively-reduce-app-dev-costs-but-wtf-is-react-native](https://dev.to/kylessg/its-cross-platform-and-massively-reduces-app-dev-costs-but-wtf-is-react-native)

我们帮助一家初创公司将 3 个开发机构减少到 1 个，同时在 Web、iOS 和 Android 上镜像功能，同时发布所有 3 个开发机构。都是因为一个叫 React Native 的小东西。

在 [Solid State](http://www.solidstategroup.com) 我们是 [React Native](https://facebook.github.io/react-native/) 的早期采用者，自 2015 年首次公开发行以来，我们目睹了它的大幅增长。在这篇文章中，我强调了*我们如何*使用它以及*我们学到了什么-*这样你就可以在构建自己的应用程序时节省时间和金钱(或者我们可以代表你)。

React Native 是一个框架，允许你使用共享代码库用 Javascript 构建 iOS 和 Android 应用。React Native 中的应用程序不是“混合”或“HTML5”，它们完全是原生的。开发者可以利用 Android 和 iOS 上存在的任何原生功能，而不必成为他们回溯语言的专家。例如，我们编写了一个允许我们与福特汽车 hud 通信的程序！

在这一点上，有必要指出，使用 React Native 并不意味着您会自动拥有一个令人惊叹的架构，并且一切都会突然开始工作。它应该被看作是项目中有价值的活动部分之一，帮助您计划和实现跨所有平台的重用。

## [](#how-we-came-to-use-it)我们是如何使用它的

2015 年初，Solid State Group 开始开发 [Dugout FC](https://dugoutfc.com/) 的 fantasy football web 应用程序，该项目为其首次发布提供了一些相当苛刻的技术功能(现场球员得分模型和实时绘图系统，仅举几个例子)。此外，应用程序必须是跨平台的，而不牺牲一些不可避免的复杂性。

在开发 web 应用程序的过程中，还雇佣了另外两家代理公司来开发本地电话应用程序。让登录页面在移动平台上工作的最初进展相当快，但是要跟上速度却变得很困难。随着开发速度的放缓，我们意识到，要让这两款应用与网络应用旗鼓相当，将是一个巨大的挑战。

几个月过去了，React Native 开始流行起来。我们决定试着把炒作和真相分开，并尝试一下。我们创建了该应用程序最复杂的页面之一，以让我们了解它是否可以实际用于生产。我们在两天内就完成了这一页，包括环境设置。客户们震惊了，他们肯定要做出一个重大的决定！大约 6 个月后，我们在所有平台上都有了一个全功能的应用程序，并继续在所有三个平台上发布新功能。

## [](#jumping-from-web-to-mobile)从 web 跳到移动

第一步是确定您希望 web 和 mobile 能够使用的所有共享功能。对于 Dugout FC 来说，这是应用程序操作的重要部分(过滤、排序、从 API 获取数据等)，占了我们 web 应用程序代码库的 80%以上。这一步非常重要，因为每次开发人员开发应用程序时，您都可以利用这些节省的时间。

一旦我们将所有的通用组件放在一个容易接近的地方，这个挑战似乎就不那么令人生畏了。我们开发了 UI 组件，插入了交互部分，令我们惊喜的是，我们的第一个页面已经运行了。

这个过程对我们来说似乎出奇的自然，因为我们都在开发一种自己熟悉的语言，并且可以在 web 上继续使用 React。拥有一个可以在移动和 web 上开发到相同水平的团队对项目来说是一个巨大的性能增益。

## [](#what-weve-learned)我们学到了什么

在密切参与该项目(以及此后的许多类似项目)后，以下是我们目睹的一些常见挑战，以及如何通过将 React Native 集成到我们的前端架构中来解决这些挑战:

**1:用 JavaScript 在单个团队中开发**

在几个开发团队之间传递需求是很困难的**，* *如果团队处于项目的不同阶段，或者恰好是位于不同地点的完全不同的公司，情况尤其如此。在《防空洞 FC》的早期开发过程中，我经常不得不描述需要在移动设备上完成的复杂功能，在多个团队之间做这件事并达到相同的结果是非常具有挑战性的，并且经常涉及相当多的来回。

这种选择的架构的最大好处之一是能够作为一个团队处理需求，确信它们在 web 和移动设备上以相同的方式执行，并且能够在问题出现时快速改变。

**2:同时开发网络和移动功能**

功能通常需要一致发布，并依赖于所有平台的协调发布。这些应用可以由一个团队在一个代码库上发布，而不是每一个功能都重复三次。这种方法使我们能够更加灵活，并以更快的速度部署新功能。

**3:错误，如“当我点击这个保存按钮时，应用程序崩溃，但只在 iPhone 上”**

即使是一个简单的问题，开发人员也可以用多种方法解决，传统的跨平台项目最终会继承大量特定于平台的代码和临时解决方案。随着项目的发展，特定于设备的错误会突然出现，并且可能变得更加频繁。

通过将大约 80-90%的业务逻辑转移到一个公共“层”，我们能够消除许多跨平台的意外差异。我们几乎从来不用担心交易行为会在网络或手机上显示不同的结果。然而，在存在合法的平台特定问题的情况下，我们可以更容易地缩小问题的原因。

## [](#when-to-use-react-native)何时使用 React Native

虽然使用最新的华而不实的框架很有诱惑力，但客观地看待项目的需求并选择适合目的的架构是很重要的。尽管如此，我们还是强烈建议 React 和 React Native if:

*   您希望在 web 和移动之间共享功能，并打算在未来在所有这些平台上发布新功能。
*   你想在 iOS 和 Android 上都开发一个 app。即使你只使用移动设备，通过共享代码节省的时间也是很可观的。
*   你正在开始一个相当复杂的项目，并且怀疑你将来可能需要一个移动应用程序。React 是 web 的可靠选择，不管你是否进入 react native，当涉及到制作移动应用程序时，你会特别高兴你做了。
*   您希望能够有效地测试移动和网络的一致性。可以在一个地方编写单元测试来测试共享代码。
*   您已经使用了 React for web。从 React 到 React Native 的转换非常简单。
*   你希望保持在科技的前沿。React Native 是开源社区中开发最活跃的存储库之一，并随着沃尔玛和 Airbnb 等公司的积极开发而不断改进。

我们喜欢尝试新的框架，并保持在技术的前沿，因为这些框架已经改变了我们所能提供的东西。React Native 已被证明是我们开发过程中不可或缺的工具，我们能够发布高质量的跨平台应用程序，同时在此过程中为我们的客户节省资金。赢了。
