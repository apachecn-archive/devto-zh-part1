# 微服务会把你的团队搞得一团糟

> 原文：<https://dev.to/plainprogrammer/microservices-will-mess-your-team-up-55ie>

[![](img/47254fd1b797cc6a20e796cc95c82f50.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--frIyEXhw--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://james.thomps.onl/wp-content/uploads/2019/07/ed641-1ufpfetekvft74xsmxnka9w.jpeg)

[微服务](https://martinfowler.com/articles/microservices.html)受很多团队欢迎。但是，软件开发模式仍然围绕这种架构模式不断变化。在过去的几年里，这个领域已经成熟了很多。然而，太多的团队产生了糟糕的实现。

微服务通常与整体应用形成对比。这种比较很有启发性，但也很幼稚。微服务是模块化或基于组件的软件设计的扩展。它应用了边界定义的思想，并将其扩展到更清晰的垂直分割。

我观察到的问题是，很少有团队投入必要的时间来定义适当的界限。当涉及到整体应用程序时，这种失败会导致应用程序难以维护。缺少这种定义的边界的微服务系统要差一个数量级。在微服务生态系统中没有明确的边界，你最终得到的只是一个分布式的整体。这是一个只有经验丰富的团队才懂得的事实。

#### 清点成本

独立和细粒度的可部署性是许多团队不需要的最大好处。这种好处是以分布式系统的操作复杂性为代价的。从消息协议到事件和消息通道，再到协调和发现。即使是小的生态系统也需要更复杂的工具，需要专门的面向操作的知识和技能。

微服务的另一大好处是更强的模块边界。但是，太多的团队很难在单一的应用程序中定义边界。像微服务这样复杂的架构方法并不会降低这一难度。采用微服务的团队需要识别和定义边界的架构技能。这种技能首先在单一的应用程序中更容易磨练，但是很少有团队投资这种方法。

> ***微服务生态系统中没有明确的边界，你最终得到的只是一个分布式的整体。*T3】**

技术多样性可以让团队对他们的工作保持激情。微服务使得采用多样化的技术变得更加容易。像独立部署一样，这种优势是以操作复杂性为代价的。而且，如果缺乏指导，团队可以采用比他们能够管理的更多的技术。

#### 解决你的问题

像任何架构方法一样，微服务解决了某些问题，并为其他问题打开了大门。在采用任何方法之前，确定项目和团队的需求是很重要的。在大多数情况下，微服务是错误的起点。直到一个团队需要独立的可部署性，并且有能力识别和维护清晰的服务边界，Monolith 才是最好的。太多的团队出于错误的原因追求微服务，并且未能确定他们想要解决的问题。

#### 制作地图

任何团队都可以从领域驱动设计中借鉴的一种方法是在[上下文图](https://www.google.com/url?sa=t&rct=j&q=&esrc=s&source=web&cd=7&cad=rja&uact=8&ved=0ahUKEwjErLj35cvYAhVG12MKHW7AAm4QtwIIQDAG&url=https%3A%2F%2Fwww.youtube.com%2Fwatch%3Fv%3Dvwr2wAclIwg&usg=AOvVaw2rLoYY-puJq1MXdv8yHLYX)上工作。上下文图定义了系统中存在的各种[有界上下文](https://martinfowler.com/bliki/BoundedContext.html)。它独立于技术和实现。这种实践可以帮助团队深入了解系统中应该存在的界限。它还可以帮助他们探索这些界限如何重叠或需要交互。这种团队和业务发现将有助于确定微服务可能在哪些方面呈现真正的价值。

* * *

康威定律经常在软件架构中被引用，这是一个需要牢记的有价值的概念。但是，一个组织对其生产的系统的影响不是单向的。团队工作的系统也会影响组织。在分布式系统上工作的团队也可能倾向于分离和较低的凝聚力。如果没有具体的实践来管理这些倾向，团队可能会变得孤立和狭隘。因此，采用微服务不仅关乎技术，也关乎文化。如果不考虑这一点，选择微服务可能会损害而不是帮助您的团队。
