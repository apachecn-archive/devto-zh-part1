# 糟糕的软件是如何制造出来的

> 原文：<https://dev.to/yelluw/how-bad-software-gets-made>

我在许多不同的软件项目中工作过。有些成功，有些不成功。事实证明，糟糕的软件项目也有类似的特征。这些是我这些年来能够识别的一些:

## 技术领导能力差

混乱和糟糕的决策不会转化为质量。好的软件不是偶然出现的。它是基于事实的组织和决策的结果。强大的技术领导力也为团队的其他成员树立了习惯。人们通过例子来学习。技术领导者必须为其他人树立榜样。

## 职责未明确

人们需要知道他们必须做什么。他们的基本职责是什么。否则，事情失败了，他们就会指指点点，互相指责。我们给人们做大事的机会，这很重要。实现这一目标的方法是给他们一些有一定期望值的机会。

## 无检测

令人惊讶的是，今天有大量的软件是在没有任何测试的情况下编写的。没有单元测试。没有集成测试。甚至没有“它在你的机器上能用吗？测试。代码被编写(或从网上复制)，编译，然后推出。我见过一些公司因为没有测试而导致糟糕的产品质量而失败。

## 不愿训练

我们不能期望每个人都知道所有的事情，并对每一项技术都了如指掌，所以培训是必不可少的。一个没有得到训练的开发团队只会以最少训练的人的速度成长。如果你每月为一名开发人员支付数千美元，那么每月花一两个小时对他们进行一些技术方面的培训是值得的。无论是测试、编码、一门新语言、一个新范例，任何事情。

我认为不鼓励培训的环境来自于公司不投资于员工的态度。他们似乎担心，一旦人们掌握了新技能，他们就会离开去寻找更好的工作。这种情况确实发生了，但以我的经验来看，并不像他们说的那样是个大问题。不培养一个人是因为害怕他们离开，等于是因为害怕被伤害而不谈恋爱。开源项目也应该培训贡献者，无论是以文档、视频还是其他形式。如果你训练你的贡献者，他们会成长并产生更好的代码。这种情况对双方都是双赢的。我只见过一小部分训练人的开源项目。这些项目的成功并非巧合。

## 有毒成员

有些人擅长写代码，却不擅长与人打交道。软件是人为了人而做的。你必须确保有毒成员得到妥善处理。有时人们变得有毒是因为领导能力差(第一条)。很多有毒的人只是被烧坏了。与有毒的人打交道是一个复杂的课题，但能够识别出有毒的成员就足以改善受影响者的状况。

## 着眼于短期目标

在没有测试的情况下推出代码来解决“死角”目前听起来是个好主意，但是开发人员必须着眼于更大的图景。你可能已经同意其他人在更大范围内做同样的事情。你曾经做过一个有测试但现在没有的项目吗？猜猜是怎么发生的。

如果你的项目有一个或多个这样的特征呢？不要惊慌。可以修好的。重要的是要明白，坏软件是我们控制范围内的事情的结果。了解并接受我们可以改善事情是扭转项目的第一步。
