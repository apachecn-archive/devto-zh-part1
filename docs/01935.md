# 我如何测量软件测试质量

> 原文:[https://dev . to/kashifkazi/how-I-measured-the-software-testing-quality-b60](https://dev.to/kashifkazi/how-i-measured-the-software-testing-quality-b60)

我如何知道我的团队是否正在产生高质量的结果，以及他们是否在前进中有所改进？

这可以通过一个聪明又可爱的 KPI(关键绩效指标)来实现。像下面这样的 KPI 让它发生。它向我解释了如何快速及时地了解产品质量，并确定进一步改进的领域。KPI 只不过是从指标中提取出来的，它们的关注点很窄。

让我解释一下什么是指标和 KPI，以及它们如何相互帮助和联系。

软件质量度量是用来跟踪、监控和评估各种软件质量过程的成功或失败的可量化的度量。

在软件质量度量的世界中，有多种统计数据可以被测量，以理解开发过程以及开发中的产品的整体健康状况。

例如，考虑一个汽车质量指标，它有悬挂，变速箱，点火，插头，线路等。一大堆功能和流程。典型的汽车修理厂会测量车辆的所有指标，以确保它们在健康证书的正常范围内。让我们称这整个校准为汽车质量指标。

另一方面，KPI 是需要考虑的最重要的因素，例如，汽车里程 KPI(单一指标)对于驾驶员(最终用户或我们案例中负责产品成功的产品经理)来说很重要。它可以告诉司机一切是否正常——也许需要专家进行更深入的调查。姑且称之为汽车 KPI 吧。

KPI 是我们一直关注的东西，即使是在软件开发和测试领域。例如，我可以收集多个指标，但是每天以相同的准确度监控所有指标并不容易。用户必须给收集的指标添加一些上下文，使它们有意义。我看的是正确的指标吗？

bug 发现记分卡很方便，如下所示。它是可以测量和量化的。它在说明是否有问题，问题出在哪里。

要为任何产品构建这种高效且易于理解的 KPI，我们需要了解两个关键因素

起源:它定义了一个 bug 何时被注入系统。

检测到:它定义了在哪个阶段捕获或识别了一个 bug。

通常，我获得这一分数的方法是在我最喜欢的 [bug 跟踪系统](https://en.wikipedia.org/wiki/Bug_tracking_system)中添加 2 个自定义字段来记录 bug 起源和 bug 检测阶段。更不用说，这对于发布后的会议和像发布回顾这样的讨论是很好的。

**从该数据信息中获取含义的一些例子**

有多种方法可以从这个记分卡中获益

例如，在“需求”阶段产生一个 bug 是很好的，但是在生产的后期阶段发现它肯定是不对的。在游戏后期发现需求或设计层面的问题可能会导致大量的返工，或者导致产品或功能的失败。

bug 发现记分卡矩阵不仅仅适用于开发或 QA 团队。它帮助从分析师、设计师、开发人员、QA 测试人员和管理人员开始的所有级别的所有团队测量质量，以及识别差距和持续改进。如上图所示，在 PROD 中发现了一个编码错误(由客户发现),在 SQA 发现了一个 REQ 错误(但不是在编码阶段)。

从我管理团队的经验来看，我也在寻找表扬我的团队的方法。例如，如果在开发阶段发现了设计缺陷，但是在设计阶段没有发现，那么它需要大量的返工，但是当在 SQA 或产品阶段发现缺陷时，返工仍然会更少

我可以很容易地掌握我的团队在需求、设计和开发阶段的成熟度。并且，我可以为测试团队衡量质量标准。
