# REST vs. RPC 并提及 HATEOS

所以 RPC 很棒，对吧？我需要对我的数据执行 X 操作，所以我将创建一个名为/doX 的端点，点击它，然后砰的一声，API 执行了 X 操作！就是这么简单！除了这就等同于说你只想对你的数据做 X。创建全功能 RPC api 的唯一方法是无所不知。RPC APIs 就像是在说:“你只能对这些数据做这些事情，除此之外别无他法。只有这些东西。”

REST APIs 描述了数据本身。它们不是关于你可以用数据做 X。如果你想对数据做 X 操作，那么你就做 X 操作，然后把它发送给 API 进行持久化。REST 是数据的持久层，而不是处理数据的 API。如果您看过乐高电影，那么您可以将 REST APIs 视为主构建器 API。而 RPC 是“令人敬畏的”API(说明手册或者甚至可能是[kraggle](http://biffbampop.files.wordpress.com/2014/02/lego-kraggle.jpg)……)。如果你没有看过乐高电影…去看看吧！

休息说“嘿！这是积木，用它们来建造东西吧！”而不是“这是你能用这些积木搭建的东西”。这种灵活性是未来发展的关键。它允许您构建一个系统，为使用您没有预料到的数据做事情提供灵活性。

HATEOS 是类固醇休息(有些人会告诉你，没有 HATEOS 就没有休息...我不同意。他们有不同的缩写。HATEOS 是 REST 的超集，可能并不总是有意义)。但是它和 REST 一样认为它是一个持久层，而不是通过 http(或您使用的任何协议)的一系列函数调用。

一个好的 RESTful API 的美妙之处在于，它不会告诉你可以构建什么。需要发邮件吗？发布到/email 终点。想要查看队列中的邮件或您之前发送的邮件吗？从/email/this-email-id-thing 获取。不需要/sendEmail RPC 样式的端点。因为像所有队列一样，我们只是在谈论一个对象集合，或者一个数组。

希望这有助于让事情变得更清楚...你的好处是因为系统的简单性而降低了维护成本。您的用户将能够灵活地构建出色的东西，创造性地使用您的数据。是的，它给个人消费应用程序带来了更大的负担，但这是一个很小的成本，也是非常值得的。
