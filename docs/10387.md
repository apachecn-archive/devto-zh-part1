# 如果我今天必须开发一个应用程序，我的技术堆栈

> 原文:[https://dev . to/n1 try/my-Teck-stack-if-I-had-to-build-a-app-today](https://dev.to/n1try/my-teck-stack-if-i-had-to-build-an-app-today)

如果我必须完全从零开始开发一个 web 应用程序，我会选择什么技术堆栈？这就是本文将要讨论的问题。

首先:我所说的 web 应用指的是介于普通的静态 HTML 页面和整个脸书之间的东西。基本上，一个为用户完成某个领域任务的应用程序，它需要用户管理、后端数据库、多个 UI 视图和控件等常见功能。我在考虑的应用程序的大小可能是一个基于浏览器的聊天应用程序，密码管理器或类似的东西。既不太简单，也不太复杂。

回到正题。为网络应用选择合适的技术就像定制一台新电脑甚至一辆新车。有几乎无穷无尽的选择需要权衡，以最终为新的 web 应用程序选择一批。这篇超级著名的文章[2016 年学习 JavaScript 的感受](https://hackernoon.com/how-it-feels-to-learn-javascript-in-2016-d3a717dd577f#.m1nodqu6f)以讽刺的方式抱怨了新 JavaScript 框架令人困惑且不断增长的混乱丛林。事实上，这些天我从许多开发商那里听到了类似的观点。他们中的许多人声称 web 上的代码质量越来越差，每个新的 JavaScript 程序员都在另一个框架上抛弃了自己的新框架。io 域。虽然在某种程度上这可能是真的，但我个人仍然喜欢巨大的技术多样性和创新。我喜欢浏览 GitHub、Reddit、Hackernews 等网站，发现新的酷库，并在某天的项目中试用。如果我今天就必须实现这样一个项目，如果没有任何限制，我会选择以下内容。

当然，技术的选择在一定程度上取决于具体的项目需求，但并不完全如此。因此，一个新项目总是一个尝试新事物的机会。ThoughtWorks 刚刚公布了他们 2016 年的新[技术雷达，其中他们分为几类*采用*、*试用*、*评估*和*持有*。当然，*搁置*-科技不是新项目的一个选项，实际上我非常同意他们关于什么应该被归入*搁置*类别的观点。*采用*基本上是现代的东西，但也是成熟的，足以避免太多的风险。*试用*-可以说，技术人员更具实验性，而*评估*是最新的花式 s***。由于我非常渴望尝试新事物，我的堆栈可能主要由最后一类技术组成。那么，我的筹码现在会是什么样的呢？实际上，我无法决定一个堆栈，但设置了两个:*花式的*和*超级花式的*。此外，我将它们的交集定义为*基础栈*，由基础工具等组成。两者的共同点。](https://www.thoughtworks.com/de/radar)

### 基栈

首先，我将使用 **Git** 进行版本控制，使用 **Visual Studio Code** 作为代码编辑器，使用 **GitLab** 作为存储库托管和构建服务器。如果我不需要自己实现用户管理，我会选择 **Auth0** 来实现。对于部署，我会在 **DigitalOcean** 机器上使用带有 **Docker** 的容器，如果我需要多个实例， **Rancher** 会帮我管理它们。作为后端前的反向代理，我会选择 **nginx** ，因为它非常高效、高性能并且支持 **HTTP/2.0** 。对于捆绑， **Webpack** 将是我的选择，任务自动化将使用普通的 **npm 脚本**来完成。为了设计用户界面，我简单地使用了**引导程序 4** 和 **SCSS** 。

### 超级花哨的堆栈

这里的关键点是，我想在我的项目中放弃传统的 REST API，而使用 **GraphQL** 来代替。后端将用 **NodeJS** 和 [**Graffiti**](https://github.com/RisingStack/graffiti) 写成 GraphQL 实现。我对后一个还不太了解，只知道它实际上是针对 Node 的 GraphQL 解决方案。为什么是 Node？因为它是网络的最佳选择(我的观点...).它是高性能的，开发起来很舒服，特别是通过在前端和后端使用 JS 带来了一致性。通过始终保持最新的 ES*特性，Node 不会变得乏味。由于 GraphQL 被告知最好与其他脸书技术一起工作，我不会在这里做实验，而是在 **React** 加上 **Relay** 上构建前端(这对我来说还是全新的)。数据库可能是一个 **MongoDB** (JSON 无处不在！)与 [**水线**](https://github.com/balderdashy/waterline) ORM。为了在蛋糕上放一颗樱桃，我还要另外介绍一下 **Redux** 。我还没怎么用过它，我听说它在开始的时候有点令人兴奋。然而，我认为它的概念涵盖了在我的应用程序中管理一致性的巨大潜力。这里的最后一件事是，我非常想要的是 ES6 语法。React 编译器 afaik 不支持它(如果我错了，请纠正我)，所以我会使用**巴别塔**来获得最新的 JavaScript 特性。如果不得不移动，React Native 将是合理的选择。

### 花式叠

这个堆栈与*超级花式堆栈*有几点不同。关键的一点是，它不会使用 GraphQL，而是一个很好的旧的 REST API。这个 API 将用 **Go** 编写，因为我喜欢这种语言——尤其是它的效率和对 web 开发的良好适应性。更准确地说，我会使用 [**Iris**](http://iris-go.com/) 框架。我已经阅读了文档，对我来说它看起来非常强大(就功能和性能而言) *(* *编辑* *):不是看起来那样！请看我下面的评论！)*。对于前端，我在**角度 2** 和 [**Aurelia**](http://aurelia.io/) 之间进行平衡。Angular 2 保证适用于任何潜在的情况，非常强大，并且拥有强大的社区和图书馆支持。然而，Aurelia 看起来也很有前途，而且可能更加清晰，样板代码更少。因此，我想试一试。但如果必须使用移动设备，我还是会倾向于 Angular 2，因为它与 Ionic 2 完全一致。

另外两个选项，在我看来很有趣的是**流星**和**地平线**。然而，我不确定在整个堆栈中只致力于一个全面的框架是否是一个好主意。

这是我在 webdev 丛林中的两种选择方式，顺便说一下，这篇好文章描述了另一种方式，尤其是对新来的 web 开发者。抱歉，我没有证明所有的选择。实际上，正如您可能知道的，如果您是一名开发人员，像这样的主观观点通常甚至不是基于纯粹的理性考虑，而是相当情绪化和自发的。
