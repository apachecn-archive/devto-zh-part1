# 技术领导者的 11 项主要职责和 10 个常见错误

# [](#on-leadership)论领导

领导是一门艺术；这不仅是一个我真心相信是真的陈述，也是 Max Depree 写的一本优秀的书的标题。在他的书中，德普瑞给我们描述了一个领导者的责任:

> 领导者的首要职责是定义现实。最后是说谢谢。介于两者之间，领导者是仆人。

这些话白天经常在我脑海中回响。当我不确定如何增加价值时，他们一次又一次地指导我。它们还包含一个很容易被忽略的深刻信息:现实不只是“存在”，它是需要被定义的东西。

现实需要被定义，因为它很少只有一个版本。你经常不得不选择一个观点，你不得不在不完全信息的情况下做出这个选择。定义现实就是赌博，下注，引导别人和你做同样的事情。你永远不会得到你想要的所有信息，但是你仍然必须做出决定。作为领导者，你也必须对他们负责。

# [](#lack-of-leadership-is-painful)缺乏领导是痛苦的

你可能有过这样的不幸经历:参与到需要做出决策的讨论中，但却没有人扮演领导角色。那些对话是痛苦的。总是有很多讨论——太多了——但是这个小组很少得出可靠的结论。最终，不管进行了多少讨论，都没有具体的决定或坚实的基础来制定清晰的计划。

在这种情况下，每个人都倾向于回避或忽略问题最棘手的方面。这个小组经常在结束谈话时感到不确定该做什么。这些情况下的“现实”仍未界定。结果，时间流逝，毫无进展；人们回到他们的活动中，却没有与现实保持一致；混乱占主导地位。

没有技术领导，一群软件工程师也只能猜测什么是技术卓越的好例子。高级工程师可能已经形成了自己的观点，但初级工程师需要看到它的实际行动。如果没有人带头，就有人猜测，而猜测往往是错误的。

# [](#the-role-of-a-leader)领导者的角色

领导者是推动对话的人，被信任在选项之间做出选择，看穿不确定性的迷雾。要想获得信任，一个领导者需要被认为在大多数时候是正确的，或者至少没有错。此外，他或她必须做出选择，而不让人觉得被迫走向他们不相信的方向。事实上，一个领导者很少把决定强加给任何人。

领导者引导对话，并确保在讨论结束前明确定义现实。或者，至少，他或她必须确定需要发生什么才能达到那个点

技术领导者也有责任通过向团队展示如何在项目环境中超越他人来定义现实。树立一个值得效仿的黄金标准对于一个团队的成功极其重要。如果没有这个标准，一群工程师经常会走向不同的方向并发生冲突。

# [](#the-role-of-the-group)群体的作用

当 Max Depree 说一个领导者的最后责任是说“谢谢”，他是在解释追随者必须感觉在控制中。领导说“谢谢”，因为他们不会把选择强加给他们的人民；相反，他们引导他们走上一条路，然后感谢他们被允许成为他们的向导的特权。

领导一个团队意味着一个领导者需要在头脑中有一个计划，然后引导团队朝着总的方向前进。随着讨论的展开，领导者需要准备好根据需要改变行动的过程，随时调整和改进。调整和提炼的过程是领导者调用集体智慧，并在此基础上构建现实版本的过程。

# [](#building-lego-towers)建造乐高积木塔

[![](../Images/3755ef1276089224f135417f23344a25.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--8Q5EuLMW--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/v4oqjlv095gtu1qs79mf.jpg)

我把引导对话的过程想象成和一群朋友一起建造乐高积木塔的过程。作为一名领导者，你开始建造这座塔的时候，脑子里已经有了一个计划。你和你的朋友分享你的愿景，他们和你一起实现它。在对概念进行修改之前，或者在需要做出必要的决定时，他们会征求你的意见。当你在团队建议、你拥有的部分和事情进展的指导下工作时，你最初的愿景会发生变化。

在整个过程中，你要让每个人都知道愿景是什么，这样团队就可以针对转移计划提出建议。你帮助你的员工以一种让每个人都最有效率的方式来组织工作。构建是一个团队的努力，但是作为一个领导者，你要根据你的专业知识和经验采取看起来最有益的行动。

你吸收好的想法，并温和地改变任何看起来不好的想法。最终，当你看到每个人都朝着正确的方向前进，愿景和目标都很明确时，你就离开了这个群体。在这一点上，你可以选择继续工作，或者只是盯着事情。最终的结果看起来会与你最初的计划不同；很可能是一个更好的版本，由团队的集体智慧打造。

在整个过程中，你不会告诉人们该做什么。你给出建议，并提醒他们整体愿景。你也为他们定义现实，并尽可能不去干涉他们，帮助解决问题，回答他们的问题。当团队完成塔的建造时，你赞扬并感谢他们的工作。他们做到了。你，主要是，为他们服务。

# [](#technical-leadership)技术领导

[![](../Images/a8d0f74e31a5be37e0d4419c2ca4257f.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--AkGFefw2--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/eqfti2k2v1gs21niwn9k.jpg)

类似于搭建乐高塔，搭建软件需要从头到尾做决策。技术领导者的首要职责是定义工程现实:需要构建什么、总体技术方向、生产力和技术卓越的黄金标准示例、业务和技术背景以及时间和资源限制。

对现实的最初定义是一个起点，它会随着进化而改变。开发团队是 it 形成过程中的积极参与者。在这个过程中，领导者必须回答问题，解决问题，协调和同步活动。领导者是一个仆人，他们的目标是确保每个人都在一个变化的愿景的同一个版本上工作，并且项目以一种自洽的方式发展。

一个强有力的领导者必须了解整个项目的技术细节。他们不必做所有的决定，但是他们必须足够详细地了解这些决定，以捕捉不一致的地方。一个技术领导者必须记住很多细节，并且必须能够看到工作的当前状态以及它如何与愿景保持一致。

一个技术领导者还必须具备理解团队正在做的工作细节的技能，不能落后。

# [](#the-11-top-responsibilities-of-a-technical-leader)技术领导者的 11 项最高职责

1.  **定义现实。**这包括为团队的其他成员提供一个清晰的技术优势和生产力的例子。团队应该将技术领导者视为可以效仿和追随的人。以身作则在软件工程中至关重要。
2.  从始至终保持项目技术愿景对每个人都清晰。当团队对特定项目或技术方向有疑问时，领导者需要有答案或获得答案的方法。一个没有答案的领导者是一个瓶颈。
3.  牢记一个项目或系统的大多数工程细节。构建部分项目的开发人员必须专注于他们正在做的事情。这取决于技术领导者是否牢记大局和大多数高级工程细节。没有这种观点，团队就有做出不一致甚至不相容的选择的风险。
4.  **回答团队或利益相关者的技术问题。**技术负责人需要帮助团队保持专注。一种方法是回答来自团队成员或外部利益相关者的大多数技术问题。回答问题不仅能提供适当的信息流，还能让领导者了解在他人眼中什么是重要的。
5.  帮助团队解决困难的工程问题。当工程师正在解决一个难题时，他们通常需要一个共鸣板来进行技术头脑风暴。技术领导者自然应该被视为这种讨论的理想候选人，并且应该能够就如何解决工程挑战提供建议。
6.  * *审查团队做出的决策，以确保一致性并与愿景保持一致。* *作为项目或系统的总体高级技术知识的保管者，技术领导者最有资格确保方向的一致性。审查决策并使用建议和指导调整决策是技术领导者的主要职责。
7.  * *为非技术利益相关者将工程现实转化为简单的术语。* *工程师对代码的细节了如指掌，如果他们不习惯将细节转化为非技术语言，他们不应该担心。一个技术领导者需要能够提供这样的翻译，根据受众调整细节的级别。
8.  * *不要妨碍团队，保护他们不受干扰，让团队取得进步。技术领导者需要履行自己的职责，同时不要妨碍团队。他/她必须通过限制不必要的干扰，让其他工程师尽可能地高效工作。这是一种微妙的平衡，需要时间来掌握，而且肯定会向任何一个方向摇摆得太远。
9.  了解团队成员的优势和劣势。技术领导者有责任了解他或她的队员，了解他们的技术优势和劣势。有了这些知识，领导者就能够将工作适当地分配给合适人。
10.  **指导团队成员。**技术领导者需要被视为团队成员的导师；能够帮助其他工程师在他们的技术旅程中进步和前进的人
11.  * *对团队说“谢谢”。* *领导者引导人们走上一条路，然后让他或她的下属去做工作。当工作完成时，领导必须感谢他们的努力和取得的成果。从长远来看，一个不对团队表示感激的领导者不会受到团队的尊重。

# [](#10-common-technical-leadership-mistakes)10 个常见的技术领导错误

我见过技术领导者陷入一些常见的陷阱。下面是一个简短的列表:

1.  将决策强加给团队。工程领导者不应该表现得像暴君。我在初级领导身上见过很多次，很痛苦。想要做所有决定的强势领导者往往会疏远开发团队。人们开始厌倦它，放弃项目，甚至放弃公司。相反，团队必须被授权在一个知识渊博且值得信赖的技术领导者的指导下做出独立的决策。
2.  **不听团队指挥。**成为一个无效领导者的最快方法就是不去关注团队的声音。记住，作为一名领导者，你是团队的仆人，你必须倾听让你领导的人的意见。领导力不是一个赋予你凌驾于他人之上的权力的头衔。这是一个你必须配得上的角色，让你在计划的设计和执行中处于中心位置。倾听做大部分工作的人对成功至关重要。如果你不听团队的话，你很快就成了问题所在，团队也不会再听你的了。
3.  **落在后面。至少在大多数时候，技术领导者必须有答案。如果他们不知道事情是如何运作的，他们的角色就大打折扣。对于一个技术领导者来说，长时间从一个项目中分心无异于死刑判决。这导致离细节太远，无法维持指导决策的能力和技术远景。**
4.  **失去团队的信任。**工程领导者需要获得并保持团队的信任。当团队失去信心的时候，当领袖就变得非常困难，失去了意义。信任必须通过在大多数时候都是正确的，并且总是对团队可用来获得。这意味着领导者必须有机会在团队面前证明自己。这是我不喜欢雇佣技术领导者的原因之一，也是为什么我相信应该从内部培养他们。
5.  在做任何决定之前等待所有的数据。这被称为“分析麻痹”，这是另一个死刑判决。一个领导者必须能够在大多数情况下利用现有的信息做出决策。总是需要更多的信息来做决定是软弱和领导能力差的表现。总是等待“更多数据”是阻碍技术领导者履行其职能的滑坡。能够面对不确定性进行操作是经验对于技术领导者如此重要的原因。“以前做过”使领导者能够基于不完美、不完整甚至矛盾的信息做出决策。经验和模式匹配很容易胜过数据。
6.  **对团队不可用。**技术领导者必须与团队紧密合作。一个超然的领导者冒着落后和失去项目细节的风险。落后的领导者很快就会失去团队的信任。如果团队成员不能从技术领导者那里得到答案，他们会冒着走错方向的风险自己做决定。
7.  **与执行管理层不一致。**走上与执行管理层不一致的道路的技术领导者是无效的。技术领导必须在公司规定的限制范围内，按照执行小组制定的路线运作。技术方向与业务方向的错位将领导者转化为问题，而不是解决方案。再说一次，我已经多次看到这种情况，但结局总是很糟糕。
8.  让骄傲、自我和自恋引导决策。当骄傲、自我或自恋倾向开始驱动决策时，领导者正走在一条危险的道路上。人们在一英里外就能看到它，最终会厌倦与它打交道。为自己的工作感到骄傲没有错，但你不能让这种骄傲左右你的决定。作为一名领导者，你是团队和公司的仆人；因此，你需要把你的自我放在一边。决策时必须考虑到企业，也就是你的客户，而不是你的自我。当你错了，承认它。当你的团队与你意见相左时，珍惜它，认真倾听。
9.  不了解你的人民。作为一名技术领导者，你需要了解你的队员。如果你看不到你的人的优点和缺点，你就不能正确地领导他们。相当注意每个人的技术技能、他们做出的决策类型、他们的协作风格、他们与谁相处和与谁冲突、他们的习惯和行为。了解你的球员。
10.  **举个坏例子。**技术领导者必须为团队提供遵循的黄金标准。他或她不应该陷入坏习惯或破坏性的模式。一些需要避免的事情的例子:
    *   马虎和写糟糕的代码。
    *   不遵守约定的标准。
    *   迟到早退。
    *   没有完成承诺的工作。
    *   反抗或者无视这个过程。
    *   战斗组织标准。
    *   作为一个糟糕的合作者。
    *   卷入非善意的摩擦。
    *   等等...
