# 从来没有一个人在你的代码上工作

> 原文:[https://dev . to/Brian nipper/从来没有一个人在编写你的代码 ia9](https://dev.to/briannipper/there-is-never-one-person-working-on-your-code-ia9)

*备选标题:避免“黑盒”和“在我的机器上工作”*

即使是独自处理一个项目，尝试创建一个只需点击一下就能“完全”构建您的项目的构建脚本也很重要。但是，您可能会问，如果只有一个人在项目中工作，我为什么要为构建脚本费心呢？事实是，从来没有一个人在一个项目上工作。

在一个给定的时间点上，可能只有一个人在一个项目上工作，但是如果项目成功了(也就是说，将来会有更多的版本)，就会有其他开发人员参与进来。要么未来的开发人员将独立完成这个项目，要么其他开发人员将加入进来。我认为，如果构建脚本已经存在，并且其他开发人员会感谢你(我也一样)，他们对如何为部署编排代码有一定的透明度，或者换句话说，避免了“黑盒”问题，那么后者更有可能。

除了编译并打包发布的构建脚本之外，我认为开发人员应该努力实现的一个有益目标是尝试创建一个可以在任何机器上检出并立即构建的包(也称为源代码和构建脚本)。注意构建工具(也称为 nAnt、Bash 脚本等)。)是建筑的先决条件。唉，对于大多数构建过程来说，没有先决条件是不可能的，也就是说，操作系统已经安装了构建工具，除了像构建脚本的批处理文件方法之外，我不喜欢这种方法。通过尝试实现这一目标，开发人员有助于避免“在我的机器上工作”综合症，当只有一两个开发人员在一个项目上工作并且没有构建脚本时，通常会出现这种综合症。对“新人”加入项目也很有帮助。我应该注意到，我特别说明了开发人员应该“努力”实现这种单次签出/单击构建，因为通常有一些部分需要在构建之前预先安装或安装本地依赖项，例如 DB 客户端或第三方插件。通过尝试实现“更加自动化的构建”的目标，它可以极大地减少其他人的挫折感。

对于那些钻研构建脚本的人来说，最后一个技巧是，建立一个构建脚本与代码库一起被版本化的过程是很有帮助的。在持续集成环境中，理想情况下，开发人员可以更改构建脚本和签入，这将触发构建，以确保对脚本的更改本身不会破坏构建。
