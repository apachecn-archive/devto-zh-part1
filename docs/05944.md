当你参加黑客马拉松时，这里有一些可能会有帮助的提示。

## 它们是关于想法，而不是实施。

如果你认为你可以用纯粹的编码技巧取胜，那你就大错特错了。

要赢得黑客马拉松，你需要一个**坚实的**想法。这是所有的大写和粗体强调。如果你觉得自己执行不完也没关系。你可以让它成为静态的，只要你能向他们展示如果你真的实现了它会是什么样子。

那里的法官知道他们的狗屎。因此，你的价值主张最好是伟大的，你必须确保你真的有竞争优势。是什么让你的应用比其他应用更胜一筹？这就是你想弄明白的。

## [](#start-with-a-problem)从一个问题开始。

要发展一个想法，你需要从一个问题开始。解决一个还没有好的解决方案的问题就好了。如果你决定用现有的解决方案来解决一个问题，确保你所构建的实际上比他们的解决方案更好。

## [](#build-around-the-idea-not-an-api)围绕理念而建，而不是 API。

当我们被告知使用 UnionBank API 时，我们的思维立即被框定为围绕 API 构建，而我们本应该通过使用 API 来构建我们的想法。

## [](#thoroughly-research-the-market)深入研究市场。

确保你了解竞争对手的商业模式。知道是什么让他们滴答作响。知道它们的不同之处。这是为了让你知道你的商业模式实际上比他们的更好，如果不是，你可以改进它，或者放弃这个想法，另找一个。

## [](#bring-all-the-people-you-need)带上所有需要的人。

我们严重缺乏人才和各种技能。我们都是拥有大致相同技能的开发人员，我们没有足够的人来处理其他重要的事情，如演示和推销。我们一心多用，这对我们的案子没什么帮助。

### [](#bring-a-designer)带上设计师。

这是一个严重的问题。我们的演示看起来很糟糕。我们没有人专门负责商标、模型等工作。我们非常需要一个设计师，但是我们没有。

### [](#bring-a-businesssavvy-person)带一个有商业头脑的人。

我们没有领域专家——在这种情况下，我们有一位银行业专家。但不一定是银行业。在你的团队中有一个精通业务的人会让你从一开始就免于多次修改你的想法。

## [](#prepare-a-dynamic-pitch)准备一个动态音高。

推销并不代表一切，但一个好的推销会让人们倾听你要说的话，以及你的产品能提供什么价值。

## [](#prepare-a-pretty-presentation)准备一份漂亮的演示文稿。

让你的演示看起来很棒。如果它看起来很糟糕，它会告诉人们你的工作很马虎，你没有做好足够的准备。此外，如果你的演示看起来不令人满意，人们也不会关注你的演示。

可以的话拍个视频。人们更容易看到它的实际操作，而不是仅仅听到你产品的功能。
