# 团队的团队

> 原文:[https://dev.to/pbeekums/team-of-teams](https://dev.to/pbeekums/team-of-teams)

我刚刚读完《团队的团队》，这是我现在最喜欢的书之一。我以前写过我所在的[最有效的开发团队](https://dev.to/pbeekums/the-most-effective-software-development-team)，但是它附带了一个警告，我没有看到过这种结构的扩大。“团队中的团队”描述了一个类似的结构……扩大到阿富汗所有美军的规模。

[![](img/f0f1ac7631981a0e2a40c9ac7ac94a47.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--UiyB-uIh--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://blog.professorbeekums.com/img/2017/8/teams.jpg)

虽然科技公司和军队之间存在显著差异，但一个重要的相似之处是它们都是由人组成的。激励人们的因素是什么，人们的行为方式是什么，人们的感受是什么:不管什么类型的组织，这些都是相似的。

严格的等级制度往往效率低下。美国军方拥有更多的资金、人力和技术。然而，它正在输掉阿富汗战争。问题是严格的等级制度会产生孤岛。人们不互相交谈。他们专注于自己被赋予的狭隘职责，而不是着眼于整个组织的利益。

理想情况下，一个组织有一套每个人都努力实现的目标。但是当人们在竖井中工作时，每个竖井都有自己的目标子集。软件开发中的一个例子是，一个公司的目标是为用户提供价值。用户毕竟需要一个给公司钱的理由。然而，根据所涉及的团队，这个目标有可能被缩小。开发团队被激励发布代码。QA 团队被激励去确保没有 bug 进入生产。这两个目标需要平衡，以最大化用户的价值。孤立的团队不会试图达到这种平衡。他们只关注他们狭隘的目标。

筒仓的另一个问题是信息共享。麦克克里斯托谈到不同的部门拥有不同类型的英特尔。不共享这一情报意味着各个单位不知道他们的努力在哪里最有用。这影响了他们完成整个组织所需的能力。通常这是无意的。如果你在一个筒仓里，你不会故意向其他团队隐瞒信息。你只是不想和他们说话。

有时候是故意的。我们都看到了这些行为。躲在“技术细节太难向你解释”背后的开发人员。挥舞着手的 UX 设计师说“这对用户来说更好，但没有解释怎么做。推动宠物项目而不解释它如何实现公司目标的产品经理。不管这些行为的动机是什么，一些人从执行筒仓中获得了个人利益。

还有其他隐瞒信息的例子。在《团队中的团队》中，麦克克里斯托考察了特拉法尔加战役。在这场战争中，拿破仑的海军数量超过了纳尔逊。但是拿破仑的海军等级森严。不是每个船长都知道整体战略。每艘船的行动都由一艘中心船发出的信号控制。

拿破仑的海军输掉了那场战役。纳尔逊确保每个船长都知道他的计划的所有细节。他确保每个船长都有能力扮演纳尔逊。他们不需要中央飞船的命令来行动。他们获得了独立行动和快速行动所需的知识。这是至关重要的，因为意外事件经常发生，能够快速应对这些事件使纳尔逊的海军更加有效。

我曾经认为我可以通过设计一个软件系统并让别人来写代码来节省时间。这肯定比训练他们自己设计系统要快得多！

不幸的是，在软件开发过程中会出现许多意想不到的事情。因为我的懒惰，这个开发者没有被给予必要的基础来解释意外。他最后不得不来找我做每一个决定。如果我不在，他就瘫痪了。我花在我们两个人身上的时间比我花在训练他更好地设计软件系统上的时间还要多。

只有在别无选择时，领导者才应该做出决定。对每一个决定都这样做是非常低效的。好的领导者确保即使他们不在身边，事情也能继续运转。这意味着给每个人自己做决定的能力。
