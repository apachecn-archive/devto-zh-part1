# 无头/API CMS 与传统 CMS

> 原文：<https://dev.to/rogerjin12/headlessapi-cms-vs-traditional-cms>

*想了解更多类似的内容，请在推特上关注 [ButterCMS，并订阅我们的](https://twitter.com/ButterCMS)[博客](https://buttercms.com/blog/)。*

如果您过去曾经使用过内容管理系统，那么您可能使用过传统的 CMS，比如 WordPress、Drupal 或 RefineryCMS。

基于 API 的 CMS，也称为“无头的”或“分离的”CMS，为编辑内容提供了一个 UI 和存储，以及一个将动态内容集成到您的网站的 API。

那么，为什么要使用基于 API 的 CMS 而不是传统的 CMS 呢？

总的来说，基于 API 的 CMS 提供了与传统 CMS 相同的编辑能力，但是对于开发者来说，设置和维护要简单得多。这意味着你的开发者花更少的时间和金钱在你的 CMS 上，而把更多的时间放在对你的业务重要的事情上。

下面是表格对比:

| Â | 传统 CMS | 基于 API 的内容管理系统 |
| --- | --- | --- |
| 设置 | 网站必须建立在 CMS 之上，要求你学习和(重新)建立基于 CMS 规则和流程的网站。 | 只需几行代码就可以在需要的地方添加 CMS 功能。 |
| 主办；主持 | 需要一个数据库和特定的服务器托管。 | 需要无宿主(尽管有自宿主选项) |
| 代码复杂性 | 内容、CMS 代码和网站代码共存并交织在一起，形成了复杂性和相互依赖性。 | 内容独立存在，并通过简单的 API 调用呈现在网站代码库中。 |
| 灵活性 | 网站必须建立在一个特定的编程语言或框架。 | 与任何代码库集成。 |
| 维护 | 安全升级、数据库备份等。 | 没有人 |

问问大多数开发人员，他们会告诉你，内容管理工作在他们的有趣工作列表中排名靠后。基于 API 的 CMS 消除了 CMS 的许多痛苦，因此开发团队可以完成工作并转移到更重要的项目上。
