# 不要用设计模式！

> 原文:[https://dev.to/geraldcroes/dont-use-design-patterns-8pn](https://dev.to/geraldcroes/dont-use-design-patterns-8pn)

不要使用设计模式，而是要了解它们——掌握它们！

[![](../Images/651b376a089532b46bc939949af227c9.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--KdVm0KzJ--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://cdn-images-1.medium.com/max/1024/1%2AMETwQ-SE2TaENAP6Ix6G7A.jpeg)

在这个吸引人的标题(抱歉)背后，隐藏着我最宝贵的建议之一，它来自于与热情的开发人员的不幸经历，这些开发人员在一次培训会议后刚刚学会了模式。有多少次我回顾了代码行，它们布满了无用的类和毫无意义的抽象？我不想知道。(而且我也不想知道我是多少次的罪魁祸首。)

永远不要尝试将设计模式付诸实践。永远不要试图将它们包含在新应用程序的设计中。不要为了“让它看起来更好”，或者为了灵活性，“这样代码在将来更容易更新”，而在任何现有的代码中加入模式，进行激进的重构

然而，研究它们，理解它们，重新阅读它们，思考它们，给它们一些关注。

### [](#the-metaphor-game)比喻游戏

#### [](#if-i-were-a-pianist)如果我是钢琴家

设计模式将是我的尺度。天平培养手指的灵活性，训练精神。钢琴家必须了解它们，但音阶不是音乐。

想象一下，如果在一个时髦的礼堂里演出时，观众在节目中读到“大调音阶以 3 拍的节奏演奏，然后是有趣的切分音小调音阶目录”，他们会有多惊讶。

类似地，如果一个作曲家在《月光奏鸣曲》中插入一个温和的 C#音阶来缓解演奏者手指的痉挛...对于一个不存在的问题，这将是最糟糕的答案。

#### [](#if-i-were-a-chess-player)如果我是棋手

设计模式将是我浏览报纸、杂志或专门书籍时解决的问题:这些谜题突出了游戏的战术方面，如偏转、牵制、诱饵、传球棋子、牺牲、过载……但它们不是游戏，它们只是帮助我学习象棋子集的工具。如果我只学习战术，我不可能成为一个大冠军。

当然，成为一名伟大的战术家会有所帮助。了解策略有助于你找到解决方案，否则你会忘记。但是，如果我在游戏中试图做的只是毫无理由地牺牲我的棋子，除了这是游戏的一个战术方面(然而也是最聪明的一个方面)之外，我最终会成为一个糟糕的棋手——没有人会说我的风格是浪漫的，纯粹是愚蠢的。

### [](#no-more-metaphors)不再比喻

当设计模式被正确地应用到实践中时，它是非常棒的。当它们是额外的灵感来源，而不是目的地，而不是终点时，它们会帮助我们。

设计模式让开发人员讨论常见问题的可能解决方案。他们展示了简单的研究案例，这些案例很容易理解，也很容易应用到更实际的项目中，应用到现实生活的设计中。放在一起，它们形成了欢迎初学者和专家的架构目录。他们在面向对象编程的特定和不同方面训练我们。

当且仅当每个模式适合手头的问题时，才应该使用它。不记住这一点将使我们的代码生病，我们很快就会看到已知的疾病，如“单身疼痛”，以及“onlytheformerdeveloperknowsaboutthismoducedcode”。

模式不是一个定义好的、不变的解决方案。模式绝不是实现。它解释了如何克服我们在前进道路上会遇到的常见障碍。

不要让模式引导你。不要试图改变你的问题，使之符合你的偏好模式，否则你最终会写出平庸的解决方案来解决你根本没有的问题。

### [](#kiss%E2%80%8A%E2%80%8Akeep-it-simple-stupid)接吻——保持简单，笨蛋。

最优雅的解决方案是最简单的，最容易理解的。

没有一句“你好，世界！”示例证明了模式的使用是正确的，没有一个“Hello World”示例符合用户的需求。Hello World 本身甚至可能已经成为一种模式——它的唯一目的是“向开发者展示一种新技术”但是没有人专门用“你好，世界！”来教开发人员一项新技术

如果一个问题有两个解决方案，一个适合十行代码，另一个有数百行代码和一个模式，请考虑不要使用这个模式。他们的存在并不是质量的衡量标准。

当然，如果您的程序中散布着相同的十行代码，那么是时候重新考虑一下了。

### [](#but-if-)但是如果……

永远不要为了“如果有事发生……”而发展，因为你当前的问题而发展。为一个模糊的假设的未来发展是把两天的项目变成两个月的项目的最好方法。如果你碰巧只通过一句“你知道，如果…”来证明一个模式在你的代码中的存在，那么删除这个模式。
