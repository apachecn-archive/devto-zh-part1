# 语义版本和编辑器

> 原文：<https://dev.to/sarfraznawaz2005/semantic-versioning-and-composer-15fn>

今天，任何认真编写的应用程序/软件都遵循(也称为*永远*)。归结起来就是这个(强调我的):

1.  主要版本时，你作出**不兼容的 API 变化**
2.  以向后兼容的方式添加功能时的次要版本
3.  补丁版本当你做**向后兼容的 bug 修正**

让我们以 SemVer 之后的 [Symfony](http://symfony.com/) 框架为例。在撰写本文时，它的稳定版本是`2.6.6`

**Symfony 版本控制**

```
+-------+---------+ | Name | Version | +-------+---------+ | MAJOR | 2 | +-------+---------+ | MINOR | 6 | +-------+---------+ | PATCH | 6 | +-------+---------+ 
```

在这里，我们可以看到次要版本和补丁版本是向后兼容的。这意味着如果你安装 Symfony 版本`2.0.0`或`2.x.x`，其中`x`是任意数字，它将是向后兼容的。这意味着如果你最初安装了 Symfony 版本`2.0.0`，现在你想安装版本`2.6.6`，你可以这样做**，而不用担心**它破坏功能。然而，当 Symfony 宣布一个以`3`开始的版本作为主要版本时，它很可能会破坏所有低于`3.x.x`的先前版本的一些功能。多亏了 SemVer，我们现在可以很容易地发现这一点。更不用说，现在几乎所有写得很好的框架或库或软件都遵循 SemVer。[请阅读更多相关内容](http://semver.org/)。

作为一名 PHP 开发人员， [composer](https://getcomposer.org/) 是如今我们离不开的东西。如果你认为你是一个 PHP 开发人员，还没有开始使用 composer，那么你不是 PHP 开发人员。在这种情况下，在谷歌上搜索它是什么及其教程，或者查看它的[官方文档](https://getcomposer.org/doc/)。简而言之，PHP 包管理器已经取代了 PEAR 等老方法。这就像 NPM 对于 NodeJS 或者 RubyGem 对于 Ruby 开发者一样。

比方说，我想通过 composer 安装 Symfony，我可以指定具体的版本来使用，比如:

```
+--------------------------------+----------------------------------------------------------------+ | Version | Description | +--------------------------------+----------------------------------------------------------------+ | symfony/symfony: "2.6.6" | Exact 2.6.6 version | +--------------------------------+----------------------------------------------------------------+ | symfony/symfony: "\>= 2.6" | Any version greater than or equal to 2.6 like 2.6.7, 6.6.9 | +--------------------------------+----------------------------------------------------------------+ | symfony/symfony: "\>= 2.6,\<3.0" | Any version greater than or equal to 2.6 but less than 3 Major | +--------------------------------+----------------------------------------------------------------+ | symfony/symfony: "2.\*" | Any version as long as Major version is 2 | +--------------------------------+----------------------------------------------------------------+ | symfony/symfony: "\*" | Latest version | +--------------------------------+----------------------------------------------------------------+ | symfony/symfony: "dev-master" | Latest version from "master" branch | +--------------------------------+----------------------------------------------------------------+ | symfony/symfony: "dev-testing" | Latest version from "testing" branch | +--------------------------------+----------------------------------------------------------------+ 
```

我们也可以使用特殊的波浪号字符`~` :

```
+---------------------------+-----------------------------------------------------------------------+ | Version | Description | +---------------------------+-----------------------------------------------------------------------+ | symfony/symfony: "~2.6.6" | Any version starting from 2.6.0 to 2.6.x with x being any number | +---------------------------+-----------------------------------------------------------------------+ | symfony/symfony: "~2.6" | Any version starting from 2.0 to 2.x or 2.x.x with x being any number | +---------------------------+-----------------------------------------------------------------------+ | symfony/symfony: "~2" | Any Major version greater than or equal to 2 like 2.x or 5.x.x | +---------------------------+-----------------------------------------------------------------------+ 
```

一旦您知道如何在 composer 中指定版本，您就可以很容易地确保您只安装那些**向后兼容的**包，除非您想要全新安装某个包。类似地，如果你已经写了一些包，你应该明智地选择版本控制，这样如果你没有对你的包进行大的修改，你的包的用户就不会意外地变得**不兼容**。
