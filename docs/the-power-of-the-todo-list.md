# 待办事项列表的力量

> 原文：<https://dev.to/jlhcoder/the-power-of-the-todo-list>

*这篇文章最初发表在[我的博客](http://jlhood.com/the-power-of-the-todo-list/)T3 上*

我想分享一个简单而强大的技术，这是我在做软件开发员的第二年发现的，从那以后，这个技术对我非常有用。我把它推荐给我指导的每一个人，并且从其他使用过这种技术的开发人员那里得到了积极的反馈。请击鼓...

我为自己管理一份每日待办事项清单。

好吧，我知道，这可能有点虎头蛇尾。在你给我起绰号 MOTO:显而易见的大师之前，让我解释一下。😋

待办事项列表的格式非常简单。它只是一个纯文本文件，包含两部分项目符号列表。看起来是这样的:

```
TODO
 -most important thing I need to do
 -next most important thing I need to do
 -...

Accomplished
 -stuff I did
 -more stuff I did 
```

我把我的待办事项按优先顺序排列，并用完成部分来记录我那天做了什么。当我将项目添加到已完成部分时，我将它们从 TODO 部分中移除。当新任务到来时，我会按照优先级顺序将它们插入 TODO 部分。每天，我将前一天的文件复制到一个新文件中，并删除已完成的部分。我以日期(MM:DD.txt)命名文本文件，并按年份将文件组织到文件夹中。同样，我知道这可能看起来简单或显而易见，但随着时间的推移，有纪律地使用这一技术会促进一些非常健康的行为。

首先，它迫使我列出我必须做的工作，并对其进行优先排序。这让我可以继续工作，因为如果我不确定下一步该做什么，我可以参考列表。它也给了我一个工作队列，所以如果我在一个任务上受阻，我可以在队列中的下一个任务上取得进展。当我对某些任务不清楚时，它也能帮助我发现。如果待办事项清单上的某个项目拖了一周或更长时间，那一定是出了问题。也许任务的可交付性没有很好的定义，或者任务应该被分解成更小的任务。

另一个重要的好处是，当我有太多事情要做的时候，清单会迫使我承认。我的经验是，如果待办事项列表超过 4 件，是时候做出改变了。这可能意味着与我的经理或 scrum master 交谈，以(1)确保他们同意我的优先事项，并(2)看看是否有替代方案，如将任务交给团队中的其他人。如果无法做出改变，它至少有助于重新设定(我和我的经理)对我能完成多少任务以及何时完成的期望。

完成部分在每天结束时提供一点士气激励，因为你可以直观地看到你的影响。然而，真正有帮助的是，当你在一家要求你写年度自我评估的公司工作时。在我的职业生涯中，我很幸运，有几个好的经理，但是我从不依赖我的经理来记住我一年中做的所有事情。我总是希望能够为自己担保。我的待办事项技巧帮助我做到了这一点。

在写自我评估之前，我会去上一年的待办事项文件夹，做一个简单的“少*”。txt”。然后我可以按顺序浏览那一年的所有文件。我关注已完成的部分，当我看到值得注意的东西时，我会记下来。我通常用不到 20 分钟的时间浏览一年的所有文件，最后，我会有一个粗略的自我评估大纲。然后，我围绕这些关键成就建立一个叙述。

一年是一段很长的时间。有好几次，我翻阅一月或二月的待办事项列表，心想，“哇，那是去年的事了？我以为那是前年的事了。”如果没有待办事项日志，我就不会把它放在我的自我评估中。此外，虽然软件开发人员的一些贡献很容易追踪，例如，提交历史，但其他的不容易，例如参与设计评审或发表演讲。待办事项列表机制帮助我捕捉所有的成就，无论它们是否导致代码被编写。

结果是我的自我评价非常详细和完整。我曾经有一个经理，他对我的绩效反馈评估基本上与我的自我评估相呼应，并说“我承认是詹姆斯做的。我承认詹姆斯也这样做过……”这有点滑稽，但也说明了这种机制的威力。我对自己的职业生涯有了更多的掌控，因为我相信自己有能力提供关于自己贡献的完整数据。它还可以让你的经理的生活更轻松，这从来都不是一件坏事。

如果你持怀疑态度，我仍然建议试一试。这么简单，成本又低，有什么坏处呢？至少给它一个月，看看你怎么想。你不必使用我的确切格式，但要保留的关键要素是(1)使工作可见，(2)强制优先决策和早期沟通的期望，以及(3)维护成就日志。
