# 大多数受访者没有发现的事情

这是成为 Rails 开发人员的大好时机。左右两边的公司都在转向 Rails，或者已经在使用它进行高效的 web 开发。如果你很了解 Rails 和 web 栈，那么你有很多选择:让我们确保你做出了一个好的选择！

# 你未来团队的代码质量

你潜在的同事应该编写你欣赏的代码。就像他们评估你的编码技能一样，你也需要评估他们的。

如果团队没有任何公开可用的代码:要求一些！同时，询问他们对 Rails 最新发展的看法。他们对“我做错了吗”这一要点有什么看法吗？Rails 中的服务对象怎么样？TDD？他们使用了什么宝石/插件(例如，用于后台处理、分析)？有你喜欢使用的测试框架吗？

如果你在看到他们的代码并与他们谈论编程趋势后感到高兴，那么你就更有可能对你在公司回购中发现的东西感到高兴。

# 团队文化

和你喜欢的同事在一起。你可能认为你不需要喜欢你的同事:但是这对每个人都是一种伤害。有足够多的 Rails 工作，你应该可以找到一个你喜欢的团队。工作是对你时间的巨大投资，所以要好好做。

这个团队是一个充满无聊笑话和网络迷因的有趣、轻松的氛围，还是一个合适、专业的环境？哪个都没关系，只要适合你就行。

# 项目管理

团队如何管理任务？工作是如何分配的？代码是如何部署的？你应该对所有这些答案感到满意。

# 在管道中工作

如果你加入这个团队，你会从事什么工作？路线图上有哪些新项目或新特性？如果公司没有明确的答案:担心。没有明确的方向对开发团队来说意味着颠簸。如果他们不能进入具体的细节，这很好，但你应该能够得到一个大致的画面。

# 典型日子

公司开发人员的典型一天是怎样的？是因为应用程序抛出 500 个错误(又一次！)?工作时间长是因为公司已经认为开发团队进展太慢了吗？是进来(或登录)，完成工作，和同事出去玩，然后在合理的时间回家？

# 津贴

不用说，你应该对公司的福利和补偿感到满意。但是不要忽视额外津贴！

公司会派员工(不仅仅是开发人员)去参加会议吗？

对于拥有大量远程员工的公司:it 部门是否定期将他们召集在一起？它能让他们一起去很棒的度假胜地吗？

公司给你提供电脑还是需要你自己带？如果公司提供机器，它属于你还是公司？
