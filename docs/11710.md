# Rails 不适合初学者

> 原文:[https://dev.to/sdball/rails-isn-t-for-beginners-3030](https://dev.to/sdball/rails-isn-t-for-beginners-3030)

网上有一些关于 Rails 如何失去对初学者的关注或者它变得太复杂对它自己没有好处的言论。我有不同的看法:Rails 从来不是为初学者编写的。让我们回顾一下 Rails 的历史和哲学。

Rails 并不是为了让编写 web 应用程序变得更容易而编写的。它被写成一组观点，减轻了在 web 应用程序上工作的痛苦，并允许我们用 Ruby 编写它们。看看 DHH 在 2004 年发布的 Rails 公告。

> 经过近一年的匿名开发，我发布了完整的 Rails。它旨在提升 PHP 程序员的水平，减轻 Java/C#人群的痛苦。哦，这都是用红宝石做的。
> 
> — DHH [Ruby on Rails:完全堆叠的网络应用框架](http://www.artima.com/weblogs/viewpost.jsp?thread=62974)

回想一下 2001 年的 web 应用程序。无论你使用的是 Java、mod_perl 还是 PHP，你都必须在启动应用程序之前做出大量的决定。样式表在哪里？后端部分在哪里？你甚至把业务逻辑从表示层中分离出来了吗？如果有的话，你应该使用哪个 JavaScript 框架？你应该使用 HTML 模板语言吗？哪一个？您应该如何与数据库对话？您应该如何开发您的 web 应用程序？您应该在哪里以及如何附加测试，您应该测试什么？

有了 Rails，您可以对所有这些问题做出决定。这就是它的用途。Rails 不是为了让事情变得如此简单而创建的，即使是一个完全的初学者也可以编写它。它的创建是为了做出许多开箱即用的决定，这样你就可以马上享受到构建 web 应用程序的乐趣。这就是“约定胜于配置”的含义。更好的是，没有一个惯例是一成不变的。

不喜欢 ERB？好吧，用 HAML。(虽然 ERB 很棒。)

不喜欢 jQuery？当然用…别的东西。

不喜欢测试单元？当然使用 RSpec。

jQuery 成为了 Rails 3.1 的默认选项(取代了 prototype ),这凸显了 Rails 的一个优点:约定将永远是良好实践的指南。就我而言，直到我看到 CoffeeScript 成为 Rails 中的默认选项，我才真正注意到它。这让我真正坐起来，注意语言，我很高兴我做到了。

说得诗意一点:Rails 是 web 开发的可怕森林中一条走过的路。“但是等一下，斯蒂芬。”你可能会说，“穿过树林的小路听起来很像 Rails 在帮助初学者。”那么你在技术上是正确的(最好的一种正确)。通过帮助每一个想编写 web 应用程序的人，Rails 帮助了初学者。但是 Rails 不适合初学者。Rails 是路径而不是目的地。

学习编程就是学习精神抽象。初学者对 web 应用程序能够或应该做什么没有概念。任何语言的有经验的 web 程序员都会听到需求，并立即(几乎是无意识地)开始建立问题领域的心智模型，而初学者仍然会纠结于系统本身是如何工作的。这在任何 web 开发框架中都是正确的，因为任何框架都没有办法替你考虑这个问题。

Rails 不是一个干草乘坐。作为 web 程序员，您仍然需要对您的 web 应用程序做出许多决策。Rails 最棒的一点是，您可以直接跳到有趣的决策和实际的编程。
