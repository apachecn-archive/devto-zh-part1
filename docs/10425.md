# 在承担新的软件依赖之前，进行更深入的评估

> 原文:[https://dev . to/Ben/conduct-deeper-evaluations-before-taking-on-new-software-dependencies](https://dev.to/ben/conduct-deeper-evaluations-before-taking-on-new-software-dependencies)

2017 年有效软件开发的一个巨大部分是有效地选择工具和服务并将其引入项目。任何足够复杂的项目都应该使用其他人的代码。有这么多伟大的开源项目和越来越多优秀的商业软件产品可用，熟练有效地依赖他人的工作是至关重要的。这是对我观察到的趋势的认可，也是对采取更系统的方法的一些建议。

我们只是人类，我们经常依靠错误的试探法来选择正确的依赖。有时候，选择是显而易见的，但我们经常根据错误的线索过快地做出决定。有很多信号在起作用，但最有可能引导我们偏离轨道的是对每个项目的伟大设计和有效营销的期望。创造伟大软件所需的技能和奉献精神与设计和营销技能并不一致，因为许多伟大的软件是由个人免费开发的，依赖这些信号通常是愚蠢的。

我有一种感觉，更有经验的开发人员，尤其是那些在 web 变得如此美丽之前的开发人员，不太可能落入这个陷阱，但是每个人都可以使用这个提醒。许多优秀的软件都是由对设计和营销有着糟糕感觉的人开发的。有很多数据库有比 SQLite 更好的标识和网站，但你不会找到很多软件像它一样面临更多的挑战和更多的依赖。

如果你确实为其他开发者发布软件，尽你所能拥抱设计的重要性是值得的。不管是好是坏，人们确实会考虑这些东西。但是如果你在搜索软件，承认软件的美学和质量之间缺乏相关性是值得的。当项目是关于设计或市场营销的时候使用它，也许可以作为一个微妙的暗示，表明开发者有多关心这个项目。归根结底，这是关于对其他开发人员的技能集的同理心。由于无意识的偏见，设计和营销将在你的决策中扮演太大的角色。

像 GitHub 和 package registry 站点[npmjs.org](https://npmjs.org)这样的站点有助于创建某种一致性，但是搜索者仍然需要挖掘表示层并为正确的原因选择软件。那么什么是正确的理由呢？当然，这要视情况而定，但如果我们在评估项目时有一个可靠的问题清单，我们就不太可能过于依赖错误的线索。

### 这里有一些我们可能会问的问题

*   这些特性符合我们现在和未来的需求吗？
*   它是否会让我们做出其他将来可能会后悔的选择？
*   我们对项目的依赖性有信心吗？
*   我们信任项目的方向吗
*   项目稳定吗？
*   该项目是否过于雄心勃勃？
*   现在的维护人员都致力于项目吗？
*   维护人员有组织吗？
*   维护人员是好的沟通者吗？
*   项目是否有(或将会有)足够的牵引力让维护人员在未来保持积极性？

这些问题的精神适用于自由开源软件和商业软件。在任何情况下，我们都不可能真正知道答案，但是要避免快速做出判断，而不是花额外的时间来评估项目更深层次的价值。我们承担的每一种依赖都是一种隐性的伙伴关系。如果我们做更深入的研究并思考我们的选择，我们将获得更好的依赖性并构建更好的软件。
