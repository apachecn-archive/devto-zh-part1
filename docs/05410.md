# 避免个人代码所有权的 12 个理由

> 原文:[https://dev . to/lpas qualis/12-避免个人代码所有权的原因](https://dev.to/lpasqualis/12-reasons-to-avoid-individual-code-ownership)

这篇文章最初发表在 [CoderHood](https://coderhood.com) 上，被称为[避免个人代码所有权的 12 个理由](https://coderhood.com/12-reasons-avoid-individual-code-ownership/)。CoderHood 是一个致力于软件工程人类层面的博客。

编码是一种艺术形式，艺术家对他们的作品非常依恋。这种发自内心的依恋让程序员充满激情和激情。这可能是一件好事，但也可能导致问题。

除非整个开发组织只有一两个开发人员，否则个人代码所有权是一种危险的疾病。我不推荐它，也不鼓励它，原因有很多。

## # 1——对代码的情感依恋伤害了集体智慧。

认同他们代码的程序员会对它产生依恋。这种情绪导致任何人都无法接触到代码，并产生教条，进而发展成地盘之争。当涉及到情绪时，代码审查就很困难。除非代码的所有者批准这些更改，否则不可能做出改进。

我把 3 个或更多的软件工程组织看作是团队运动。这种团队意识应该扩展到所产生的工件的所有权、目标、过程、衡量和庆祝成功的方式，以及工作的所有其他方面。运动队中的每个运动员并不拥有胜利。团队集体拥有它。每个成员都是能够完成工作的单一有机体的重要组成部分。赢得比赛的是团队，而不是球员。类似地，开发人员不应该觉得他们拥有代码。相反，他们应该感觉自己是构建代码的必要部分，为组织而不是自己玩游戏。

作为工程文化的每一个方面，这种对代码的感觉需要从高层鼓励，理想的是在工程组织的最开始。如果鼓励是成功的，这种文化将在战壕中形成并成为现实。查看我关于公司文化的帖子，了解更多关于如何影响文化形成的想法。

## # 2–个人代码所有权是一个单点故障。

开发人员生病了，他们辞职了，升职了，转到了其他部门，离开了州，死了..你说吧。将公司知识产权的重要部分捆绑在一个人身上是通向灾难的单行道。单个代码所有权将您公司的健康与代码所有者的健康联系在一起，或者与代码所有者和公司之间的健康关系联系在一起。

如果一个开发人员拥有代码，那么这个开发人员就是单点故障。如果他或她出了什么事，本组织将负责处理后果。代码越复杂，公司要承受的损失就越大，回到正轨的时间就越长。

## # 3——团队比他们各个部分的总和还要强大。

一个有天赋的开发人员可以创造难以置信的价值，但是一个由 N 个开发人员组成的团队可以创造超过 N 倍的价值。集思广益、协作和反馈可以提高任何开发人员的质量和产出，不管他或她有多有才华。当一个人拥有代码时，头脑风暴和协作就变得次要了，其结果会影响产品的质量。

认为几个单独工作的开发人员比一个小团队一起工作更好是一种谬误。一个单独工作的开发人员可能会更快地生成一些实际的结果，这对原型来说是很好的，但是结果的质量和寿命将会受到限制，这对业务来说不是很理想。

## # 4——代码是产品。该组织拥有它。

没有人会在确定房子是属于建造它的木匠，还是属于购买它的房主方面有问题。这是因为房子是一个物理对象，具有我们所认为的买方所拥有的物理属性。当你在商店买食物时，那就是你的食物。如果你买了一辆车，那就是你的车。相当明显！

当一个产品是概念性的，事情就不那么明显了。例如，一部小说的作者拥有故事，而一本书的购买者只拥有印刷小说的纸张。软件更接近于一个概念，而不是一个物理对象，开发者，尤其是小公司的开发者，感觉就像是他们个人拥有它。他们有那种感觉，因为他们是唯一了解细节的人。这是一种自然的趋势，但也是一种组织和开发人员都应该避免的谬误。

软件归公司所有，而不是由编写它的开发者所有。从法律上讲，这是软件开发人员和他们的雇主之间的大多数雇佣合同中陈述的事实，但是人们每天的感觉可能非常不同。虽然工程团队应该对这项工作有责任感，但是开发人员不应该有个人所有权的感觉。

## # 5——创新带一村。

如果一个开发人员单独拥有代码，那么这个开发人员就应该完成所有的创新。听起来很可笑？确实是！作为一名单飞的工程师，你确实可以快速创新，但很快你就会遇到收益递减点。另一方面，一个团队通过集体智慧、头脑风暴、从许多不同的方向看待一个问题，并带来一个人无法拥有的丰富经验来继续创新。

如果团队随着时间的推移而改变，并且有不同经验的新人被注入组合中，创新的质量会提高，新概念的产生会重新点燃。辩论和良性摩擦刺激创新；秘密和精心保护的知识领域不会。

## # 6——个人代码所有权导致停滞。

缺乏创新导致停滞。长时间独立工作的编码人员容易对他们已经习惯的想法产生确认偏差。他们会拒绝接受对他们工作的任何改变，因为概念对他们来说已经获得了一种纠结的意义。任何来自外部来源的想法或影响最终都会被过滤掉，如果它与之前的决定不一致的话。

以这种方式工作一段时间，尤其是在创业初期，可能是一件好事。然而，最终这种开发软件的方式会停止扩展，创新会很快消亡。停滞是独立开发者的死敌，必须全力以赴。

## # 7–个人代码所有权阻止个人学习。

整天长时间独自在同一个代码库上工作的开发人员会被束之高阁。这意味着他们对任何不属于自己专业领域的东西都变得生疏，并停止成长。

我见过来自大企业的开发人员，他们在一个应用程序的同一个对话框上工作了多年(微软的一些部门曾经因为这个而臭名昭著)。那种专业化是极端的个人代码所有权的结果，导致个人成长的损失。

## # 8——个人代码所有权阻碍职业发展。

如果一个开发人员越来越依赖他们的代码，他们就不仅仅是上面提到的单点故障；他们也很难晋升到更高的职位，因为没有简单的方法来替代他们所负责的工作。

我之前在一篇名为[“让自己丢掉工作”的文章中谈到过这个问题要获得晋升，你必须放弃你最珍贵的财产，努力摆脱目前的工作，这样你就可以开始一份更高级别的新工作。个人代码所有权限制了这种可能性。开发人员要获得晋升，首先要做的是确保他们不是代码库某个部分的个人所有者。](https://www.coderhood.com/career-advice-work-yourself-out-of-your-job/)

个人代码所有权通常是自己造成的，很容易被逆转。要做到这一点，开发人员需要拥抱和鼓励其他工程师接触他们编写的代码，并帮助他们熟悉这些代码。起初，放弃所有权会让人觉得有风险。这感觉就像你在把自己从工作中挤出来，但这正是去做不同的、更好的事情的正确做法。

## # 9——个人代码所有权是工程师的监狱。

由于“拥有部分代码”而感到不可或缺，这可能会给人一种错误的印象，即有些人所说的工作保障。然而，工作保障看起来更像是一个牢房，而不是一个舒适的地方。没有学习新技能，没有职业发展，一个软件工程师被关在一个越来越小的自制监狱里。

几年前，我读了一篇名为[放弃你的乐高玩具，以及创业的其他戒律](http://firstround.com/review/give-away-your-legos-and-other-commandments-for-scaling-startups/)的优秀文章。它谈到了创业公司的可扩展性；同样的概念也适用于个人可扩展性和职业发展。一个开发者把他们的乐高积木放在旁边，小心翼翼地保护着它们，他正在用乐高积木搭建一个监狱牢房。放弃你的乐高玩具，重塑自我，继续前进。

## # 10——坚持自己代码的开发人员与领导发生冲突。

在技术领域，市场现实一直在变化，产品也是如此。有时候产品的变化必须是激进的，开发人员需要能够适应和支持领导层要求的变化。当个人代码所有权意识太强时，开发人员往往会与领导斗争，抵制变革。这种行为可能会导致冲突，最终要么减缓进度，要么让开发人员被解雇。

在一些大型组织中，由于项目被取消，整个团队都被解雇了。想一想。为什么一个公司会因为一个项目被取消而解雇一整队有才华的工程师呢？如果工程师能够毫无问题地从事新项目，你认为公司会解雇他们吗？现实是，有时候，团队是围绕项目而形成的。个人变得全神贯注于项目，拥有它并小心翼翼地守护它，失去了适应新事物的能力。拒绝改变和放手的，不仅仅是监狱；有时候就变成专业棺材了。

## # 11——个人代码所有权导致不信任。

觉得有必要拥有和捍卫代码库的开发人员正在传播一种不健康的信息:

请勿入内！我是唯一一个能理解或接触这个代码的人。

这一信息可能不明确，但在整个工程组织中都能感觉到，并有意识或无意识地被认为是不信任的标志。当同一个团队的成员彼此不信任，并且不清楚这一点时，团队就会受到影响，失去快速行动的能力。不信任导致不健康的摩擦，燃烧了宝贵的能量，却没有产生任何积极的回报。

当开发人员开始对在同一个代码库上工作感到舒适，并放下他们的个人所有权意识时，事情就变了。信任成为工作方式的一部分，团队合作成为驱动力。在植根于信任的环境中运作的团队会产生更好的结果，变得更有弹性。

## # 12——拒绝加入有组织的开发过程不会扩大规模。

感觉自己拥有代码的开发人员，经常感觉他们不需要任何过程来指导他们。敏捷方法，如 Scrum 或任何其他形式的组织结构，对那些开发人员来说似乎是陌生的，不必要的。这与拒绝在任何压力、结构或约束下工作的艺术家没有什么不同。

这种软件开发的观点根植于早期的黑客时代，这在年轻开发人员的生活中非常典型。在单飞的那些日子里，学习是有机地发生的。长时间的工作主宰了程序员的生活，而流程在一个人的团队中毫无意义。这种工作方式是不可扩展的。当不止一两个人参与一个项目时，在辉煌的黑客时代使用的方法不再有效。

在开发人员作为团队的一部分运作的组织中，过程成为一个重要的工具。这有助于协调努力，确保高效、可预测和有组织地交付成果。黑客和开发软件有着巨大的区别。虽然前者可能很有趣，但后者是工程组织扩展的方式。
