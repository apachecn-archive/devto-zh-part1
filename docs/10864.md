# 有摇滚明星，也有乐队

> 原文:[https://dev . to/developerscode/there-are-rock stars-and-there-are-bands](https://dev.to/developerscode/there-are-rockstars-and-there-are-bands)

在某个地方，某个时候，有人提出了术语“摇滚明星程序员”。也许就像我们这个行业中大多数朗朗上口的术语一样，它开始是一个天真的想法，只是被正确的人滥用，最终成为今天这样的营销噱头。

编程不是关于一个人的救援队，除了那些你只是在一旁涉猎的宠物项目。然而，一些企业仍然在寻找神秘的摇滚明星程序员来纠正他们所有现存的错误。他们有一大堆工作要做，那堆工作可能马上就要着火了，他们需要有人来帮他们摆脱困境！你们好吗？！

我从来没见过用这种心态创作出来的好的、扎实的软件。编写好代码的工作是艰巨的。这需要时间。它需要沉思和合作。更有经验的程序员可能知道哪里存在更多的陷阱，但他们仍然需要时间来思考问题。任何人都可以建造一些看起来在短期内有效的东西，但真正的工作在于让它长期有效(或可靠地调整它以使其有效)。否则，这个行业就会陷入一场无休止的游戏，“谁想暂时成为下一个李尚义·罗斯？”

更重要的是，好的软件*团队*不能围绕一个人的表演来建立。当我想到开发优秀软件的人时，我会想到摇滚明星的音乐对立面:乐队本身。最好的团队是小型的(比如 2-5 人)。而且，他们粘在一起。他们知道每个成员的倾向，因为他们一起工作了很长时间。他们知道什么时候让彼此独处来培养他们的产品，什么时候(是的)，在正确的时间“让乐队重新在一起”。对我来说，这是保证产品畅销的最简单的方法:建立一个愿意长期坚持的紧密团结的团队，一起*。*

 *与一个忠诚的小团队一起构建软件的最大好处是，它免除了你必须应用无数过程的麻烦——这些过程对于一群彼此不熟悉的技术人员来说是必不可少的。当团队非常小并且已经一起工作多年时，像正式的冲刺、scrums 或复杂的版本控制策略这样的“人员流程”是多余的。所有这些策略都是为了给即将到来的混乱强加一种持续的仪式秩序。而且，即将到来的混乱会发生在不断更换团队成员的团队中——不管他们中是否有摇滚明星。

当然，小的、持久的团队也不能免于走上正常的编程黑暗之路。代码库仍然很糟糕。变革的力量仍然将代码推向笨拙的形状。但是，在这些团队中，摆脱困境要容易得多。意图是什么，做出了什么样的权衡，以及现实如何变得可能，这些想法仍然存在于团队成员的头脑中。通往优雅代码的系统化路线带来的摩擦更少，因为你们都曾一起战斗过。

在我们的行业中，长寿的编程团队很少被看好。同样，编码员之间的凝聚力和友情也被严重低估了。相反，我们所拥有的是一种文化，这种文化支持个人，并关注强制过程，以最小化团队开发人员不可避免的变动的影响。

或者，如果我们支持老年乐队而不是摇滚明星，也许我们也间接消除了对神话摇滚明星的需求。
