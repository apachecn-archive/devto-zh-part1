# 导师的十字路口

> 原文:[https://dev.to/rubynista/a-mentors-crossroad](https://dev.to/rubynista/a-mentors-crossroad)

作为一名有经验的开发人员，我处于一个独特的位置:我在寻找指导的机会，同时还能吸收周围人的经验。在某人的编码之旅中成为一个支持者的最佳方式是我脑海中的新鲜事物，因为我仍然在精心打造自己的旅程。我可以理解学员的想法，比如，“我希望某某人会采用这种方法……”，或者“我觉得我可以解决这个问题，如果我得到的支持是……”并立即将其作为导师应用。

这个视角也是一个来回。我遇到过一些学员，他们诱人的举止和对学习机会的开放态度让我觉得帮助他们是一种休闲，而不是工作。这些人也给了我一些指导，告诉我应该做些什么样的调整，以便向那些向我寻求编码智慧的人展示同样有益的体验。

这些是我从能够看到硬币的两面中学到的一些经验:

## 师徒小贴士

1.  **提问**
    仅仅因为我有更多的编程经验，并不意味着我知道我的学员应该走的路。提问让我对情况有了清晰的认识，这样我就可以提出最好的建议。你所帮助的人需要了解他们问题的根源吗，或者他们需要知道如何找到他们问题的根源吗？也许他们还没有问题，但他们只是不确定手头的任务。当你只看到一条错误消息或一段代码时，这些场景都不太清楚。你的学员可能羞于开口，或者在说出“你能帮我一下吗”这句话后不知道从何说起。此外，提问会促使学员练习说出他们的技术决策/意图。就我自己而言，在所有我熟悉的领域(不仅仅是网络开发)，我都是一个擅长写作而不是演讲的人。当我请我的技术合伙人帮我解决我正在开发的一个功能的问题时，尽管我们合作得很好，他可能会读懂我的想法，但他不会让我在描述某个东西时说出我最喜欢的通用词:“thingy”。这个小小的折磨真的帮助我培养了说话的自信，我希望也能把这种自信灌输给我的学生。

2.  **表现出对其他选项的偏好而不轻视**“一路香草 JS——谁还会再用 Coffeescript？”“你可以忘记 Heroku- Docker 是未来”当你有这样强有力的陈述时，你可能会让你的学员处于分析瘫痪的状态，因为他们不确定最初应该致力于哪种语言/工具/框架。要真正掌握你正在做的事情，你需要花真正的时间去做，但是让一个超级坚定的人告诉你你做了错误的决定是没有帮助的。你的意见来自一个好的地方，应该表达出来，但避免对它做出全面的陈述，因为当你告诉他们不要为 erb 费心，因为 haml 是高效、干净的(gorgous 和我都喜欢它)-令人惊讶！他们会回来告诉你，他们的新工作只有雇员再培训局的应用程序。包括我自己在内，开发人员可能对他们的技术着迷，但现实世界的工作场所并不总是尖端的，学习一些东西是有价值的，即使它不是最新的热门。关于这一点，最重要的是，如果你减少了对你不喜欢的选项的厌恶，你就创造了一个更支持的社区，我们都希望在未来的几年里成为其中的一部分。

3.  投入真正的时间和注意力
    当我花时间和我可能指导的人一起编写代码时，我会带着一个我可以做的项目，但我并不期望我会取得很大进展。将上下文从一个代码库切换到另一个代码库需要时间，即使只是评估一下情况。我从来不想催促我所帮助的人，因为有人看着你打字已经够让人焦虑的了，增加额外的压力是没有意义的。最终，我花在帮助别人上的时间会三倍地回报给我。我开始回顾我可能已经知道的信息，我开始练习在不同的环境中应用解决方案，我已经建立了一个盟友，如果出现类似但更复杂的问题，我可以打电话来帮助我。双赢的局面...情况。=)

4.  不要做个混蛋 T2 你累了。指导不是你唯一要做的事情。尽管如此，不要做一个混蛋。很明显，你知道足够多的代码来给别人提供建议，所以你可能喜欢编码。不要为了别人而毁了它。你有好的导师，也有糟糕的导师。做某人的良师益友。(注意:我本来打算省略这一点，但这是真的，它很好地提醒了我自己要保持积极/耐心/乐于助人)。

5.  当指导一个你认为比你级别高的人时，不要犹豫
    一个编程时间是你两倍的人，他对每一门学科的了解都没有你多！记住这一点。编程趋势是不断发展的，即使是你在 meetup 上遇到的最天才的结对伙伴也不会什么都知道。成为这个人需要的开放的耳朵/大脑。在有些情况下，你比他们更有经验，即使在你明显没有经验的情况下，你对他们面临的功能和结构障碍的了解问题也是有帮助的，并会指导他们。社区需要你。出去吧。

## 学员提示

1.  让意识流完成
    当导师向你传递信息时，他们是在试图为你构建问题。有时他们不得不从边缘开始，因为他们不确定你目前的知识水平，他们不想跳过你在这个问题上的任何空白。其他时候，这就是他们说话的方式——我们中的一些人用简短的句子说话，而我们中的一些人用长篇大论的篝火旁讲故事的句子说话。在任何一种情况下，等待你的开发伙伴完成他们的想法允许他们维持他们的流程并交付解决方案。如果碰巧与您正在尝试诊断的特定问题无关，您将有另一个机会在事后重新表述您的问题。这给了你一个机会，在你的导师找到他们的解决方案时，你可以学到一些额外的东西(或者回顾一些你已经知道的东西)。

2.  **重申手头的任务，而不仅仅是我面临的具体问题**
    也许你一直在试图理解如何获得你正在制作的广告中所有动态元素的具体高度，以便你可以从窗口顶部暴露出正确的距离。也许你已经花了一个小时试图找出正确的公式。开始回答问题“你在做什么？”用“我试图在这个广告上实现一个‘阅读更多’的横幅”，而不是描述你用来完成这项任务的确切方法。这让指导者对你的任务有了更全面的了解——这很有帮助，尤其是因为他们不像你那样熟悉这些代码行和特性需求。如果你想知道，我放弃了所有元素的高度曝光策略，而是选择了点击显示区块策略。Javascript 没那么糟糕，但是用来实现它的 jQuery 让我转到了其他方面。

3.  **展示并讲述你的代码，即使你不需要帮助**
    即使你没有遇到任何问题，保持关于你的技术堆栈的对话是一个很好的方式，可以获得你自己可能不会获得的编程方面的小知识。我经常听说有人用不同的语言/框架构建了我所构建的东西。这些叙述通常会有一个比较/对比，说明该实现与我所做的相比是如何工作的。从他人的经验中学习是普遍的好建议，但尤其适用于开源社区。

总而言之，我会说——走出去，指导几个人，找几个人来指导你。在这位导师的十字路口，将你所寻求的伙伴混在一起，将会拓宽你作为开发者和教师的有效范围。软件最终是关于人的，但是这些人不仅仅是你的最终用户。他们也是你在人生道路上遇到的人。

本周，当我重新整理我一直想开始/完成的“未完成”帖子的列表时(这一个是在 2015 年 8 月开始的)，我丈夫注册了自己的 GitHub 帐户，所以我真的把这些技巧放在心上。我不知道他的编码之旅会把他带到哪里，但我很高兴以我所知道的最好的方式成为其中的一员。
