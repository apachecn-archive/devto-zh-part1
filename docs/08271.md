# 管理安装

你需要安装和设置什么来让这个项目工作。

*   在本地机器上安装 Ruby，Node.js，npm。
*   初始化你喜欢的任何版本的版本控制(jk，安装 Git)
*   安装 Postman 和 AWS CLI 工具(谷歌这些，非常简单)。以后你需要工具来管理我们的 API。
*   如果你真的想在某个时候发送电子邮件，你需要设置一个 Sendgrid 帐户或者 Mailgun 帐户。如果你只是需要测试，我推荐 Mailgun，因为如果你不是一家成熟的企业，Sendgrid 很有可能拒绝你。我是基于 IP 被拒的，我相信(不爱日本 T_T)。在本教程中，我们将使用 Mailgun。您可以根据需要换成 Sendgrid 它非常类似于 Mailgun。

如果你使用 Mailgun，你也很可能需要一个域名链接到你的 Mailgun 账户。一开始我没有链接域名——它发送了大约 3 条消息，然后就离开了我。链接取决于您的主机提供商；到时候用你的 google-fu。

## 版本控制

我们需要就这个项目中不同寻常的版本控制流程进行一次简短但严肃的谈话。

我假设我们做的每件事，你都会备份到版本控制。这意味着:

我们制作的每个 Lambda 都应该在你的 VCS 中有自己的文件夹。这样做有一个非常具体的原因:当你想上传库到 AWS Lambda 时(如果你不明白，我稍后会解释)，你会将库和代码存储在同一个文件夹中。

如果您在 aws 控制台 Lambda 浏览器内代码编辑器中更改 Lambda，而不是在您的本地系统上进行更改并上传更改，我假设您将这些更改复制粘贴到您的 VCS 中适当的 index.js 文件中并保存它，因为我们**不信任 AWS 为您管理您的代码**。

每当我们将 API 网关部署到一个阶段时，我假设您转到阶段->导出->中间选项(swagger / yaml)，然后将 yaml 文件放在您的 VCS 中的某个地方，因为我们**不信任 aws 为您管理您的代码**。

我遇到了一个严重的事件，API gateway 已经(四次！)删除了一些我部署到一个舞台后非常需要的设置。当我联系支持人员时，他们说只有在使用 AWS CLI 出错时才会出现错误。我仍然不知道我做错了什么，有了出口，重建就快多了。无缘无故失业是很可怕的。

我可能会提到，我们应该不时地在版本控制中保存一些东西，但总体而言，我认为您可以处理好自己。
