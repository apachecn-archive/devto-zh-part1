# 无可指责的验尸模板

**议程**:

*   剖析我们所理解的事件(时间线)
*   讨论可采取的可操作步骤，以保证此类错误(据我们所知)不再发生

**结果**:

*   可行想法列表(故事/史诗)
*   *不是*:“注意！或者“小心点！"
*   后续会议，观察以下方面的进展

# 无可指责的札记:

如果你不得不多次手动完成某件事，那就把它自动化，这样你就再也不用做了。

## 主指令

> 不管我们发现了什么，我们理解并真诚地相信每个人都尽了最大努力，考虑到他们当时所知道的、他们的技能和能力、可用的资源以及手头的情况。

## [“无可指责的剖验](https://codeascraft.com/2012/05/22/blameless-postmortems/)

> 对事故是如何发生的缺乏了解，几乎可以肯定它会再次发生。如果不是和原来的工程师，将来还有一个。

#### 拥有一个“无可指责的事后分析过程，意味着其行为导致事故的工程师能够给出详细的说明:

*   他们在什么时间采取了什么行动，
*   他们观察到了什么影响，
*   他们的期望，
*   他们所做的假设，
*   以及他们对事件发生时间线的理解。
*   他们可以给出详细的解释——不用担心惩罚或报复。

#### 避免名/咎/耻的循环:

1.  工程师采取行动，导致失败或事故。
2.  工程师受到惩罚、羞辱、指责或重新培训。
3.  现场工程师(“尖端”)和管理层(“钝端”)之间的信任减少，寻找替罪羊
4.  工程师对行动/情况/观察的细节保持沉默，导致“掩盖你的屁股工程(由于害怕惩罚)”
5.  由于上述第 4 条中提到的沉默，管理层对日常工作的了解和信息变得越来越少，工程师对潜在的失败条件的了解也越来越少
6.  更有可能的错误是，由于上面的第 5 条，潜在条件无法识别
7.  从步骤 1 开始重复
