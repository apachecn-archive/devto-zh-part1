# 评估软件缺陷

> 原文:[https://dev.to/pbeekums/estimating-software-bugs](https://dev.to/pbeekums/estimating-software-bugs)

软件错误是任何复杂应用程序不可避免的一部分。人类的大脑也许能够理解一个应用程序是如何在一个高层次上工作的，但是能够让每一个细节在你的头脑中浮动是不可能的。这意味着程序员很容易错过处理代码中的特定情况，也就是边缘情况，因为他们没有考虑这个问题。

结果是软件中非预期行为或软件错误。由于 bug 是无意的，所以很难说要花多长时间才能修复。

为了提供一个估计，你需要知道需要做什么来修复它。

为了知道需要做什么来修复这个 bug，您需要知道是什么导致了这个 bug。

为了知道是什么导致了错误，你必须调查错误的原因。

因为错误是无意的，所以您通常不会马上知道错误的原因。你将不得不花时间去调查它。通常情况下，调查会花费最多的时间。当您知道足够多的信息来评估一个 bug 的修复时，您已经完成了修复它的大部分工作。

从项目管理的角度来看，这是具有挑战性的，因为目标是最大化公司的投资回报。开发者极其宝贵，他们的时间需要好好利用。要计算修复 bug 的 ROI，您需要知道修复 bug 会给业务带来多少价值，以及修复需要多少时间。

确定修复一个 bug 的价值通常并不太具有挑战性(尽管有时需要一点猜测)。由于上述原因，等式的另一半是一个不同的故事。

有趣的是:**软件开发人员无时无刻不在为软件 bug 做估算。**

修复一个错别字需要多长时间？可能不多。

bug 是一个糟糕的数学计算，可以 100%重现，并且在单元测试覆盖的范围内吗？这可能不会花太长时间来修复。

bug 是在开发者的机器上无法重现的东西吗？那可能需要一段时间。

这些都不是“2 小时”或“2 天”形式的估计。他们会更加模糊，例如“应该很快”或“可能很棘手”。但是修复一个 bug 的价值也不会是一个确切的金额。粗略的猜测总比没有好。如果你怀疑一个很少用户会看到的 bug 可能需要几天或几周的时间，那么它可能不值得修复。

问题是开发人员不喜欢交流这些评估，这是有原因的。这些估计是直觉，很有可能是错误的。如果系统构建得很差(例如，代码在开发人员没有写权限的数据库中)，即使是像打字错误这样简单的事情也会花费一段时间。无论这是不是一种合理的感觉，开发人员都不想给出一个很有可能是错误的估计，因为他们可能觉得他们的经理会把它当作一个承诺。

不管管理者是否真的这样做，他们会产生反效果的感觉。如前所述，粗略的估计比没有估计要好。有沟通总比没有沟通好。对软件缺陷进行评估的方法是让开发人员在交流他们已经有的想法时感到舒服。
