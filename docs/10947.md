# jQuery 万岁

> 原文:[https://dev.to/ben/long-live-jquery](https://dev.to/ben/long-live-jquery)

关于 jQuery 死亡的报道被大大夸大了。

jQuery 最近宣布了它的第一个 [3.0 候选版本](https://blog.jquery.com/2016/05/20/jquery-3-0-release-candidate-released/)，从一开始它就是 web 开发的重要工具。该库提供了一个坚如磐石的 API，用于与 DOM 和其他关键浏览器功能进行交互，同时隐藏难看的语法和糟糕的浏览器怪癖。尽管无处不在的图书馆为网络提供了一切，但它仍然名声不佳。在我看来，人们已经呼吁它的消亡有一段时间了，取而代之的是“现代”范式，如 React 或 Ember 或 Vue 或 Angular 或你自己的蹩脚框架。这些框架很棒，但是我相信它们根本不会“取代”jQuery。它们补充了 jQuery，并且具有根本不同的用例。

流行的现代框架被构建来处理复杂的客户端数据和状态。坦率地说，它们为出现之前难以处理的问题提供了重要的解决方案。但是它们也带来了自身固有的复杂性，以及文件大小、框架锁定和堆积如山的复杂性等诸多问题。jQuery 仍然是处理基本客户端交互的非常直观的 API。我相信，当开发人员避免追逐闪亮的 UX·帕特森时，他们可以提供非常好的体验，甚至不需要首先处理客户端上所有复杂的状态。有许多不同的方式将复杂性设计到应用程序中，坦率地说，它不必像今天流行的风格那样呈现在客户端。

我个人采取的方法是在明显需要 React.js 模式的项目中使用 react . js，避免固执己见，如果需求不明确，就像瘟疫一样构建依赖工具的库和框架。这种方法对我很有效，帮助我避免了您经常听说的可怕的 JavaScript 疲劳。虽然回归自然编写普通的旧 JavaScript 很好，但从实际意义上来说，jQuery 是具有 web 前端的基本开发项目的主干技术。

在某种程度上，jQuery 为从未真正学习过 JavaScript 的开发人员赢得了拐杖的名声。对任何被认为是拐杖的东西的批评都是有一定道理的，但是不要恨玩家，要恨游戏。这种声誉确实证明了 API 的易用性，就像任何具有这种低门槛的开发工具一样，将会有许多用它编写的残酷代码。

jQuery 仍然是向不需要维护复杂全局状态的网站添加交互层的首选工具。据 BuiltWith 统计，排名前 100 万的网站中，有 77.9%使用 jQuery。实际上，你带进这个项目的任何人要么已经知道 jQuery，要么很容易掌握它。它是如此流行，以至于当我使用普通 JavaScript 时，我发现当我在栈溢出上查找 JavaScript web 解决方案时，我需要在搜索查询中添加“无 jQuery”。虽然它是一些人的拐杖，但在实际快速完成工作方面，它往往胜过编写普通的 JavaScript，并且重量为 28kb Gzipped，就带宽而言，它是一个相对实惠的选择。

与任何工具一样，jQuery 最好是有原则、有目的地使用。如果您能够以这样一种方式来设计应用程序，使得复杂性不会上升到太接近表面，那么 jQuery 是完全合理的选择。它愚蠢的名声是基于它被滥用的方式，而不是基于它的 API 或者任何在网络开发中已经发生根本改变的东西。随着 3.0 的发布，jQuery 将成为 web 开发人员需要的大多数功能的合理选择，3.0 构建于成熟的 API 之上，具有增强的 promise 支持和性能提升。
