# 最重要的编码指南

> 原文：<https://dev.to/_patrickgod/the-most-important-coding-guidelines>

拜托，没那么难。当你开始一项新的工作或项目时，当你进入一个新的团队时，就一些编码准则达成一致真的很有帮助。你不需要一整本书。一页其实就够了。但是有些事情，你绝对应该在这一页上提到。

让我给你举几个例子。事实上，我认为只有三个是必须的。其他的我不在乎。比如忘记前缀。如果你觉得有必要在你的变量前面加上“str”或“f”或“m ”,那么就这么做吧。如果你来自 C++世界，只是不能忘记这些特定的风格，我没关系。但是，请不要错过以下指南。

### 吻或者换句话说:单纯

是的，我知道你听过一千遍了，这是下一个称赞这个规则的人。“保持简短&简单”或者“保持简单，笨蛋”。绝对是三大准则之一。

我知道，正则表达式太酷了，哦，你用二进制标志代替了简单的布尔值，你真是个伟大的程序员！那是什么？你把这个 30 行的方法减少到 5 行，但是增加了 10 行注释，因为下周没有人理解这个算法，包括你？**不要这样**。

你知道什么是伟大的代码吗？**大家都懂的代码**。真的就这么简单。当然，也有例外。如果您希望验证输入的电子邮件地址，并且可能对您的权限管理系统使用二进制，请使用这个标准的 regex 代码片段。但请让这些事情成为例外。

不仅仅是这些技术。当你思考一个算法时，你可能会觉得它太简单，可能不够聪明，就此打住**使用你现在头脑中的这个算法**！如果可以改进，也许可以在重构代码的同时进行。但是请记住，其他人——甚至是你未来的自己——最终会阅读你的代码，并且不希望再次经历一段艰难的时间来阅读你巧妙的代码。

保持简单。真的。使用 if 语句，如果你认为这样更容易阅读，就用 foreach。但是永远不要达到这样的地步，你想成为一个代码魔术师，不想透露他的技巧，没有人知道她是如何做到的。

对你的同事表示关心。谢谢你。

### 命名

Robert C. Martin 很早就在他的书《C#中的敏捷原则、模式和实践》中提到了这一点，我也认为这非常重要:给你的变量、成员、方法和类**命名，这不需要任何进一步的注释**或文档。

同样，也有有效的例外。for 循环中的“int i”是可以的。但是如果你用 foreach 遍历一个用户列表，不要只使用“u”。将当前值称为“用户”。或者更好:currentUser。我是说，为什么不呢？真的有那么难吗？你写一个稍微长一点的名字会浪费很多时间吗？到现在为止，我真的无法忍受所有这些单字符变量名。物体“o”是什么？还是字符串“str”？

方法名也是如此，它让我对方法的目的有了正确的认识。GetUserName()应该给我用户名**并返回它**。PutUserProfile()应该调用 web 服务并更改用户的配置文件信息。它不应该做得更多。

如果您发现自己正在考虑一个可以被命名为 RegisterNewUserAndSendConfirmationMail()的方法，您可能会考虑重构您的代码，这将我们带到最后一个重要的指导原则。

### 行代码

不要误解我。代码行从来都不是高质量代码的有效基准。我在这一节讲的不是这个。不在于你写的代码有多少，**而在于你的方法和类有多大**。

他们说不要让方法比你的屏幕大。嗯，这不是很精确。比方说，你的方法应该不超过 30 行，你的类最多 500 到 1000 行。这并不意味着你把 300 行的方法简单地分成十个连续编号的方法。不要！

这条规则应该让你思考。这更像是一个设计规则。当你有 31 行方法时，没有人会解雇你。但是你应该经常问自己，你是否能以一种适合这个指导方针的方式来设计你的类或方法。

考虑上面提到的方法，它注册了一个新用户，并向该用户发送一封确认电子邮件。你可能已经猜到了，你可以用这个做两个方法。更好的是，您可以在某种程度上实现 SendEmailConfirmation()方法，以便其他方法甚至您的同事能够使用它。

正如我所说的，这更多的是关于设计规则或模式的指导方针。这足够再写一篇博文了。但是现在，只要记住这三种简单的编码方式:

*   保持简单愚蠢(吻)
*   合理命名
*   简短的方法和类

如果你还是不相信我，那就等到你不得不修改一个有 60.000 行代码的类的时候吧。六万。我是认真的。不要做那个程序员。谢了。
