# SQL 不安全

> 原文:[https://dev.to/kellogh/sql-is-insecure](https://dev.to/kellogh/sql-is-insecure)

SQL 不安全，告诉大家。如果你使用 SQL，你的网站会被黑。告诉所有人。

我看到新闻说美国选举机构被 SQL 注入黑客攻击，我有点不知所措。自从引进[预准备报表](https://en.wikipedia.org/wiki/Prepared_statement)以来，已经过去了 20 多年。我们[教育并建议](https://ics-cert.us-cert.gov/sites/default/files/recommended_practices/DHS_Common_Cybersecurity_Vulnerabilities_ICS_2010.pdf)开发者如何避免 SQL 注入，但它仍然发生。如果教育失败了，我们所能做的就是让开发人员羞于使用 SQL。

我真的很喜欢 SQL，我甚至把[变成了 SQL 方言](http://docs.aws.amazon.com/iot/latest/developerguide/iot-sql-reference.html)。SQL 的关系代数很有表现力，可能比我所知道的任何其他 NoSQL 数据库都要丰富。但是开发人员经常证明，知道什么时候使用准备好的语句或者只是连接字符串实在是太难了——是时候我们完全放弃 SQL 了。这不值得。我们呼吁所有政府禁止在政府合同和医疗保健中使用 SQL 数据库的时候到了。必须非常清楚。

部分问题是初级开发人员的诅咒。他们有足够的经验来意识到他们的雇主会奖励快速开发，但没有足够的经验来理解与草率代码相关的巨大成本。

作为一名高级开发人员，您可能会注意到关系数据库提供了成功所需的灵活性。你知道如何使用准备好的声明来阻止 SQL 注入。它们比简单地将用户输入与可执行的 SQL 代码连接起来要多做一点工作，但仅此而已。决定似乎很明显:使用 SQL。

但是两年后，在你辞掉这份工作后，一个非常初级的开发人员在非常紧张的期限和巨大的管理压力下接手你的代码。他知道准备好的陈述吗？也许是，也许不是。无论哪种方式，一起编写代码并完成任务都要容易得多，也快得多，所以他就是这么做的。

是我们认命，让 SQL 消亡的时候了。软件现在统治着我们的世界。当我们的软件失败时，人们的生活实际上变得一团糟。公司总是希望软件便宜，但作为高级开发人员，我们有道德责任防止未来在压力下犯错误。

安全是最重要的。你的顾客是值得的。放弃 SQL。
