# 辅助项目-避免启动失败

> 原文:[https://dev.to/liquidise/side-projects 避免发射失败](https://dev.to/liquidise/side-projects---avoiding-failures-to-launch)

*本帖原载于[登高](https://blog.benroux.me/avoiding-failures-to-launch/)T3】*

这个故事很熟悉。

一个很好的想法，简单但有价值。一个全新的代码库，就像一个充满可能性的世界。一个副业项目诞生了。一个周末的外卖披萨和旋风般的编码奠定了美好事物的基础。但是新鲜感会消失。生产率下降。最终，在工作和家庭之间，进步几乎停滞。失去了动力。但是你有了新的想法。就是这个。循环重复。

势头是项目的一切。保持它可能很难。诀窍有两个:避免降低动力的事情，保持你的范围足够小，以便在动力耗尽之前完成。

我已经建立了许多辅助项目。大多数从未发射过。但是对于每一次发射失败，这篇文章中至少有一个错误是原因。这些年来，我已经知道应该避免什么。

我最新的努力就是建立在这些经验之上的，并且已经成为我最成功的项目。它的成功在很大程度上是因为我没有重复过去的错误。我希望他们能帮助其他人避免我花了几年时间才发现的陷阱。

### 挑选一个熟悉的堆栈

很容易，启动一个附带项目的最大障碍是从一个你不熟悉的堆栈开始。虽然副业项目是玩新技术的正确地方，但你打算实际启动的项目却不是。当你开始一个副业项目时，问问自己这个项目是用于实验还是启动。如果你选择发射，选择一个你知道的堆栈。句号。

这一硬性规定的原因完全在于动量。没有什么比那些时刻更能扼杀动力了，当你对自己说“我该怎么做？我需要 2 条线……”。不要落入那个陷阱。我坚信，每次你不得不谷歌一个代码问题，你就不太可能完成那个附带项目。

### 写代码前发布

一旦你选择并初始化了你的栈，就释放它。是的。说真的。

当你开始一个项目时，动力很少会像现在这样强劲。这意味着这是一个伟大的时刻，让这个必要的邪恶的方式。你最不希望的事情就是在项目开始 3 周后才开始发布过程，却发现你遇到了特定的生产问题。不要玩那个游戏。

选择你想释放的地方，并尽早把它解决掉。等待太久，它就不会发生。正如我们选择了一个熟悉的堆栈一样，这也是选择一个您熟悉的托管环境的好时机。如果 heroku 过去对你很好，那就这样吧。如果你托管自己的服务器，欺负到你，托管在那里。无论您选择什么地方，只要您初始化了项目和数据库，就应该发布它。

### 建立一个购买按钮

没有在我的项目中添加购买按钮是我职业生涯中最大的失败。虽然这些要点中有许多是关于避免动量损失的，但添加一个购买按钮是一个巨大的动量提升。

整合条纹。它将需要一个 [EIN 号码](https://www.irs.gov/businesses/small-businesses-self-employed/apply-for-an-employer-identification-number-ein-online)。通过在线表格获取只需 5 分钟。政府效率的罕见例子。沼泽中的希望灯塔...我跑题了。

你能够运行测试结帐并看到收入图表上升的那一天是一个伟大的日子。仅仅是从你建立的网站中赚钱的可能性，就是一个经常被忽视的强大驱动力。不要否认这一点。

此外，副业不一定是慈善劳动。你的时间比你正常工作的报酬更有价值。也就是说，你的副业至少应该有集资的能力。

所以建立一个购买按钮。我认为你应该在添加网站的核心功能之前就建立它。

### 拥抱天才的时刻

我发现程序员经历了生产力的波动。我们倾向于将这些波动与灵感时刻联系在一起。更准确的说法是，我们的思维过程会随着时间的推移而改变，那些鼓舞人心的时刻就是当它转换到我们试图工作的模式时。

在构建项目时，有 4 种不同的思维类型:

1.  特色建设(解决问题)
2.  造型和文案写作(创意展示)
3.  验证和错误修复(普通任务)
4.  手工测试(批判性思维)

当你在做副业的时候，你会在短时间内完成。当你开始工作时，要学会认清自己的精神状态。当你这样做的时候，允许自己在你目前的天才领域工作。

有规律的上下文交换不仅会让你不觉得事情拖拖拉拉，还能更有效地利用你的时间。

### 清楚地描述你的产品

当你有一个早期的创意时，从写下你产品的一些基本描述开始。它解决什么问题？是给谁的？为什么用起来是一种享受？

原因是，副业，实际上根据定义，比传统的销售工作更依赖于第一印象和自我服务。为了促进这一点，你将需要一个营销页面，清晰而简洁地介绍你的产品。

我强调“简洁”，因为这是一个挑战。这就是为什么我建议尽早编写这些文本块，即使它们只有一行 CSS。

如果你写了一些草稿，随着你对项目理解的加深，你会发现自己又回到了草稿上。这些修改通常只需要很少的努力，但在最终产品中会有很大的回报。但是如果没有一份营销文案的初稿，这是不可能的，所以尽早写下一些想法。

就像上面一样，忽略 CSS 直到你有灵感。当你不觉得特别艺术的时候，尝试将字体页面设计成副产品是一个让你沮丧到极点的好方法。

### 获得反馈

好了，真正谈一会儿。我们这些 1 和 0 的鼓吹者往往不擅长这个。我们要么过于内向，要么过于疲惫，要么过于自信，以至于无法就我们的项目寻求真正的反馈。但如果说我学到了什么的话，那就是在打造产品的时候不应该有小我。

事实是每个产品都是错的，直到它为很多人解决了问题。在那之前，它要么没有被发现，要么还没有完成，要么没有被展示给正确的人。

一旦你有了一个项目的工作版本，是时候找出问题所在了。从朋友和同事开始。转向潜在客户。让人们注册并阅读你的营销。让他们在回答你的时候无情一点。虽然赞美是令人鼓舞的，但我们的产品并没有变得更好，因为我们正高高在上。我们现在需要的是批评。

就我个人而言，我用要点列出了每个人的反馈。趋势会出现。人们会对某个特征或标签感到困惑。人们会想知道你营销页面上的某个句子是什么意思。他们会被卡住。这些绊脚石应该是你最优先的改变。你最好拥有一个人们理解的功能不足的产品，而不是一个让他们困惑的功能齐全的产品。

### 发货吧

最后，出货。

事实上，这是最重要的一步。这里的其他一切都是达到这个目的的一种方式。所以你跳过了上面的一些吗？谁在乎。装运它。你是否还有一个你想添加但不是绝对需要的功能列表？装运它。在 IE9 上有没有一些你向其寻求建议的人告诉你的风格问题？装运它。

作为开发人员，我们对自己的个人项目有一种奇怪的错觉。我们觉得发布不完美的项目会降低我们的声誉，或者诸如此类的废话。放开那些禁忌。

向世界发布一个附带项目所带来的成就感是值得体验的。你在这个过程中学到的关于范围、细化、特征切割和产品展示的经验对你现在和将来工作的公司来说是无价的。

古老的格言是，发明是 10%的灵感和 90%的汗水。许多开发人员会有关于一个伟大产品的灵感。很少有人采取措施来实现这个想法，并从中获益。但它们是真实的，值得的。

所以，带着你的想法，开始行动吧。建立一些我们一直在谈论的势头。终点线可能只有几个星期了。
