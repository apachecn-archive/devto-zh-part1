# 为什么我从[语言 _1]切换到[语言 _2]

> 原文:[https://dev.to/tra/why-i-switch-from-language1-to-language2](https://dev.to/tra/why-i-switch-from-language1-to-language2)

我是[Language_1]的忠实粉丝，也是它的早期采用者之一，对[Language_0]的彻底失败感到失望。我是许多开源项目的热心贡献者，例如[Obscure_Project_1]、[Obscure_Project_2]和[Obscure_Project_3]。然而，在使用[Language_1]超过 5 年后，我一直在处理[Minor_Technical_Flaws]。起初，我忽略甚至容忍这些瑕疵，但我被迫面对现实。我不能忍受这些缺陷，因为[Language_1]是一种成熟的语言，要真正修复[Minor_Technical_Flaws]即使不是不可能，也是很困难的。

我也讨厌[Language_1]的观点。当我第一次使用这些语言时，我认为这些观点是一股清新的空气。我会经常在黑客新闻上为这些观点辩护，因为我认为这些观点会促进良好的编码实践，并最终导致更高的生产率。然而，经过 5 年的编码，我意识到这些观点可能有轻微的缺陷。

然后，我听说了[Language_2]。[Language_2]承诺修复[Language_1]中的所有技术缺陷。我还阅读了[Language_2]中固有的观点，并意识到这些观点与[Language_1]中错误和错误的观点相比是完全正确的。这就是我换语言的原因。

让我给你看一个例子来证明我的观点。

下面是一个“用[Language_1]编写的 Hello World 程序:
的例子

```
PROGRAM HELLOWORLD
 10 FORMAT (1X,11HHELLO WORLD)
 WRITE(6,10) 
 END 
```

你可以明白为什么我会被这种语言吸引。它的句法很美。但是它太神奇了。我不知道到底发生了什么。*格式*是什么意思？还是*写*？当我不得不扩展我的程序时，我会整夜咒骂，试图调试[Language_1]决定对我使用的最新“魔法”。一半的时间，我都在与[Language_1]的约束性限制作斗争。

这里有一个[Language_2]中“Hello World”的例子:

```
>+++++++++[<++++++++>-]<.>+++++++[<++++>-]<+.+++++++..+++. [-]>++++++++[<++++>-]

[<++++>-]<+. [-]++++++++++. 
```

你可以看到[Language_2]比[Language_1]更容易阅读和维护。它的语法和[Language_1]一样优雅，但是没有任何魔力。我需要知道的一切都在那里，以一种易于理解的方式准备好了。这里没有语言之争，只是我在用我可信赖的新工具工作。

在工作中总是使用最好的工具。这就是为什么我选择使用[Language_2]来完成我所有晦涩的副业项目。它可能不会有任何技术上的缺陷，它的观点也不会被证明是错误的。而且，在[Language_2]让我失望的极少数情况下，嗯，总有[Language_3]…
