# “当软件变得复杂时”

> 原文：<https://dev.to/ben/when-software-gets-complicated>

软件天生复杂。发生了很多事。开发人员往往对复杂的系统着迷。尽管过度复杂化会让事情变得更难，但我们有办法接受它。有些事情*需要*才能变得复杂，有时我们只是预料到事情会变得复杂，所以我们不会反抗。事情变得越来越复杂是自然规律，但是这种趋势的复合副作用必须不断被检查。

我们的软件可以无限扩展，但我们的大脑不能。这就是为什么我们需要构建可组合的应用程序，开发干净的 API，并用自动化测试覆盖我们的代码。我们创造心智模型是为了更好地理解和交流到底发生了什么。我们用隐喻来简化和交流，或者至少我们应该这样做。我们往往会对自己处理复杂问题和找到解决方案的能力感到兴奋。我可以与另一位编码人员进行长时间的交谈，我们可能会不停地谈论我们如何令人印象深刻地解决了这些问题，但如果我们没有以可沟通的方式解决这些问题，我们就会越来越重，如果我们需要的话，将很难摆脱。我们假设的解决方案变得越来越复杂，超出了我们与他人交流的能力。

随着项目的增长，无论是商业应用还是开源框架，或者其他什么，我们解决了更多的问题，但是我们创造了表面的复杂性，使得整个心智模型崩溃。我们曾经能够交流的简单性完全被打破了。为了提供一个我在自己的开发环境中试图描述的具体例子，脸书流行的 JavaScript 库 React 经历了一段减少复杂性的动荡时期。当它第一次开源时，这个想法受到了抵制:共同定位标记和逻辑的想法对许多人来说很讨厌，人们没有理解它。最终，在 React.js Conf 2015 前后，团队能够找到让它如此令人兴奋的点，即以解决问题、降低复杂性和提供性能优势的方式利用数据流的承诺。反应本地是樱桃在上面。“学一次，随处写”。这是一个美丽的承诺，人们蜂拥而至。很明显，自拐点出现以来，React 一直是一件大事。

但从那以后，事情变得棘手了。人们在试图用这个库做很多事情时遇到了复杂性问题，虽然我认为开发人员倾向于找到解决方案，但新的解决方案并不像当初让人们实现的承诺那样优雅和简单。建立在 React 之上的结构提供了很好的技术解决方案，但是没有提供最初使采用如此顺利的心智模型。尽管 Redux、Relay、GraphQL 等都是很棒的技术解决方案，但它们缺乏人类开发人员理想需要的可开发性。我并不想对那些比我聪明的人创造的解决方案嗤之以鼻，但是实际上我觉得他们的角色是开源解决方案。完美是好的敌人，但在 React 中表现如此出色的同样清晰的目标在更大的生态系统中还不太存在。可能这个问题本质上太复杂了，以至于不能再“简单”了，但是你可以很容易地说，关于核心的 React API，团队能够充分地分散复杂性，让人们[“用 React 思考”](https://facebook.github.io/react/docs/thinking-in-react.html)。这个社区仍然有很大的发展势头，但是复杂性的增长已经超过了社区提供模型来减轻开发人员精神负担的能力。

随着软件开发、React 开发或组织内开发等活动越来越受欢迎，越来越多的用户成为超级用户，因为人类的交流能力不会像代码或社区规模那样扩展，所以趋势是这些超级用户在整个对话中占据越来越大的份额。他们是对决策有最大影响力的 1%。他们也可能对复杂性有着超乎寻常的容忍度，并且最有可能存在于具有相同容忍度的人们的共鸣室中。React 正在通过 [create-react-app](https://facebook.github.io/react/blog/2016/07/22/create-apps-with-no-configuration.html) 朝着正确的方向迈出步伐，但这将是一场持续的斗争。

复杂性不能被战胜而屈服。每个解决方案都有一定程度的“适当”复杂性。这在很大程度上也是个人品味的问题，这也是为什么开源项目不需要“胜过”其他项目的一个重要原因。但是复杂性绝对必须被检查，而且要经常检查，因为创造复杂性要比压制它容易得多。
