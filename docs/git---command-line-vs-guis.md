# Git -命令行与图形用户界面

> 原文：<https://dev.to/gonedark/git---command-line-vs-guis>

在*获取 Git* 的[介绍视频中，我谈到了从命令行而不是通过 GUI 使用 Git。我将 GUI 标记为*邪恶巫师*并用自定义的`die`命令杀死他们。](https://vimeo.com/album/4319966/video/196220914)

但是*为什么是*？GUI 的*怎么了*？

我没有随意给 GUI 贴上*邪恶巫师*的标签。这个术语来自我读过的第一本也是最顶级的编程书籍之一——《实用程序员[》](https://pragprog.com/book/tpp/the-pragmatic-programmer)。

务实的程序员充满了故事来突出你作为一个程序员会遇到的日常场景。大多数故事都以一个*提示*结尾。这本书包含了 100 多条建议。

在这种情况下，我将参考*提示 50* :

> 不要使用你不理解的向导代码

这个提示来自第 35 节:邪恶的巫师 ，可以应用于命令行和 GUI 的争论。由于 GUI 代表我们生成 Git 命令，它被归类为*向导*。

所以问题变成了，使用一个*向导*有什么问题？

我改编了这一节中的以下段落:

> 除非你真正理解代表你的命令，否则你是在欺骗自己。你编程纯属巧合...如果它们产生的命令不太正确，或者情况发生了变化，您需要修改命令，那么您只能靠自己了。

我认为这击中了要点。你必须真正理解 Git。作为程序员，这是我们每天都在使用的工具。它有很少的命令集——大约 15 个，其中不到一半是经常使用的。

最后，问题不在于 GUI，而在于缺乏理解。我将使用图形用户界面作为视觉差异或合并工具。但是，我也喜欢从命令行查看差异和解决合并冲突。

如果你正在使用 Git 的 GUI，我鼓励你挑战自己。当您使用它时，尝试识别用于生成当前屏幕的底层命令。如果你不能，你正在使用一个*邪恶的巫师*。
