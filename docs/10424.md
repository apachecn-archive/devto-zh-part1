# 来个 refactorthon 怎么样？

> 原文：<https://dev.to/developerscode/how-about-a-refactorthon>

重构。

这是我们大多数人谈论的比我们实际做的要多得多的概念之一。软件预算很少给开发人员时间来重塑代码，而没有任何可见的结果。没有客户愿意为此买单。

相反，我们必须找出边写边重构的方法。尽早重构，经常重构——这就是答案！然而，软件欺诈仍然像死亡和税收一样不可避免。理论通常会遇到严酷的现实。在下一个截止日期之前让该死的东西工作通常会侵占编写优雅代码的权利。所有这些有利于现在的*的小折衷都表现为我们以后必须维护的混乱、模糊、没有方向的代码。*

除此之外，你可能不知道什么是真正的重构，直到代码库闻起来足够了。早期和经常的重构可以在前面的草坪上形成一套漂亮的灌木，但是有一天你可能会后退一步，意识到你真正需要的是在后院的一套草坪椅子。

在代码库生命中的某个时刻，当前进的齿轮开始停止转动时，就会积累足够多的积垢。我们可以埋头苦干，在技术债务的泥沼中奋力前行。或者，奇迹般地，试图让我们的利益相关者相信我们已经竭尽所能，是时候重新开始了。这两种方法都不像是最佳方法。

在我的职业生涯中，有几次我感受到了“大重构”的强烈冲动。我已经把谨慎抛到了九霄云外，在下班后和周末闯进了 IDE 只要有时间，我就可以完全专注于我对令人不快的代码库的个人恩怨。秘密行动有时是唯一可行的方法。

在这些会议中，总有一个时刻，投资最终会有回报。在某些时候，你知道最艰难的工作已经过去，你的劳动成果就在你面前。俄罗斯方块的最后一块到位。脓肿可以消除。味道没了。代码不再依靠黑魔法运行。

那个时刻，对我来说，是*编程*最满足的部分。我知道许多其他人也有同样的感觉。以我的经验来看，在一个紧张的周末重构会议中可以做大量的好事。令人遗憾的是，这种事情经常不得不在众所周知的夜幕掩护下发生。

这里有一个新的想法:**一个重构图**。

黑客马拉松通常面向新生想法的快速发展，一群人刚刚见面，喝着过量的咖啡因和披萨，在没有正式休息的情况下，为新技术编码一两天。哦，你知道...正是这种有利于急需重构的代码的环境。

我们的工业近来发展得太快了。摆在我们面前的胡萝卜是建设新的东西，而且要快。但是，今天编程中更重要的需求不是另一个基于最新的无结构数据库的单页面 mashup 应用程序，它使用的是 JavaScript framework。相反，它应该吸收所有被证明对社会有用的技术材料，并确保它长期保持有用。

那么，如果我们把这个想法反过来呢？如果我们召集一群程序员，用一个周末的时间来重构现有的产品代码，会怎么样？

在第一天，每个参与者都会收到相同的真实世界的工作代码库，以及对其功能的详细解释。也许，积极致力于代码库的开发团队可以解释最大的痛点在哪里，并随时回答问题*(当然，根据代码库，可能有 NDA 和其他文件要签署，但请与我呆一会儿)*。

个人将有时间审查代码库。在某个时候，团队会形成，工作会开始只关注重构。没有新增功能。没有让应用程序更有市场的想法。这个房间的精力将完全投入到从头到脚改进工作代码库上。几天后，每个团队都能够以某种方式显著地改进代码。

在重构结束时，团队将展示他们重构的内容。他们为什么选择某种方式？它是如何改进代码库的？他们能移除多少代码？现在是什么让代码可读性更好？积极的开发团队可以选择 refactorthon 获胜者。

想象一下，当几十个聪明人连续两天都在处理同一个目标——更好的代码——时，房间里会有多少伟大的想法和讨论。也许 refactorthon 是我们的行业需要的平衡物，将酷的东西与现在迫切需要的东西结合起来。
