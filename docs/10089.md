# 数据驱动开发/精益编程

> 原文：<https://dev.to/topriddy/data-driven-developmentlean-programming>

我已经做了几年的开发人员，所以我参与了不同种类的软件、团队、软件方法和“非方法”。我曾在一些团队中工作过，在这些团队中，软件开发过程紧密反映了传统的瀑布模型，在这种模型中，您只是收到一组需求/功能，这些需求/功能由“业务团队”传递，然后传递给开发团队，开发团队试图分析这些需求/功能，然后将其分解为模块和部分，主要是为了开发方便，而不一定是因为它通常是作为一个大爆炸版本交付的。

之后，新的工作以新功能、对现有功能的修改和错误修复的形式出现。大多数情况下，并不是因为这些应用程序通常没有内置的分析和监控功能，很少了解用户如何使用应用程序，因此这些功能和更新通常是由(错误的)前提驱动的，即越多越好，或者业务团队的随机想法甚至更糟。在这个世界中，您通常会以大型“企业”(有时这个词可能意味着大型复杂系统，其中应用程序已经超出了每个人的掌控范围，大型复杂的事情正在幕后发生——后端，或者只是多个模块/视图，它们只是提供了一种 CRUD 数据的方法，而没有过多考虑实际的用户体验/工作流)系统告终，该系统具有多个维护最差的功能，很少使用，并且没有提供所需的价值。大量的时间被浪费在无关紧要的事情上，导致不愉快的软件团队和困惑的业务团队(产品所有者等)。

上述场景与敏捷软件开发相去甚远，也绝对不是数据驱动开发或精益编程(因此被称为 DDD/LP)我的帖子的标题，但我认为过去如何做事情以及我们过去在哪里的背景可能会为理解我稍后给出的 DDL/LP 的非常简单的示例提供良好的基础。实践本身很简单，大部分的努力可能都花在推动心态的改变和让利益相关者接受上。

本文中的数据驱动开发指的是一种软件方法，其中新功能和现有功能的更新基于从用户如何与系统交互的度量和分析中收集的见解。嘿，你可能想争论没有应用程序时你如何获得数据？我们这里有鸡和蛋的问题吗？不，通常对于一个新项目，你可能会让 UX 团队模拟出一些可以用于用户测试的东西，从中获得的数据可以用来决定新功能是否可以交付价值。显然，我已经简化了，但这里的关键是，团队工作流中优先考虑的新功能/更新有一些科学依据。

精益编程就是你如何快速、少量地实现这一点，这样你就可以快速获得关于这一点的执行情况的反馈，然后使用获得的数据来告知所需工作的方向，这可能是增强功能，将其提供给更多类别的用户，提供相关功能，或者有时必须分析新功能表现不佳的原因，删除它，然后从中学习，以告知未来的工作。很多学习发生在 LP/DDD，他们被当作一等公民对待。

最近，我有机会在一家不错的公司的一个小型敏捷团队中工作，我已经能够看到、学习和实践一种不同的软件开发方法，其中我们使用数据驱动开发和精益编程。这让我再次对软件开发感到兴奋，几乎足以在写作中断很久之后，在早上 7 点之前起草这篇文章，来描述我参与的 DDD/LP 工作的一个非常简单的真实例子。

*   产品负责人 UX 分析公司从网站上收集指标，发现了网站的用户模式，通过在页面上添加他们认为有助于转化的小功能，指出了一个大改进的机会。
*   创建了一个模型，并进行了一些用户测试以收集一些反馈。结果将提交给团队，并形成一份工作计划书
*   在估计了所需的工作之后，这个特性可以交付给两种类型的系统用户，并且可以包含至少两种“详细”类型的信息。应用 LP，并决定将其分解以交付给第一类系统用户——用户 A，并且最初仅交付第一类细节 I
*   然后由团队开发和测试
*   A/B 测试方法和功能切换是作为一等公民建立的。分析已经过测试，以确认其有效，并且收集了正确类型的信息！
*   部署和性能监控
*   反馈是积极的，因此证明了需要另一次迭代，以便交付给更多的受众。
*   考虑为第二类用户提供服务
*   额外收获:该功能需要根据分析数据为您的顶级用户及其设备提供支持/服务

这种软件开发的方法，可能是本文的反高潮，特别是如果你实践这种方法，实际上已经允许你向用户交付价值，提高你的转换率，最重要的是通过帮助优先化需要完成的工作来管理你有限的资源。
