# 开发人员的工作什么时候完成？

> 原文:[https://dev.to/pbeekums/when-is-a-developers-job-done](https://dev.to/pbeekums/when-is-a-developers-job-done)

这听起来可能是一个愚蠢的问题。“开发人员的工作什么时候完成？”乍一看似乎很明显。开发人员编写代码。他们的工作在代码编写时就完成了。

[![](img/a00dd70f190e7676df539ef316386339.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--AACZsfSh--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://blog.professorbeekums.com/img/2017/7/done.jpg)

我曾经有过这样的看法。我大学毕业后的第一份工作是在每次发布前冻结 3 个月的代码。这 3 个月是 QA 团队全面测试的时间。我写的代码经常在我写完几周后被测试。如果有什么东西坏了，也许我会收到他们的回复。如果有用的话，可能什么也听不到。

这是一个巨大的反馈延迟。当我把一些东西发给 QA 时，确实让我觉得我的工作完成了。我经常听不到任何关于我的代码的消息。如果 QA 确实给我发回了任何东西，我会认为这是我要做的新工作，而不是完成一些我认为已经完成的早期工作。

当我加入一个 12 人的公司后，这种态度很快改变了。在那个规模下，没有 QA 团队。我测试了我的代码。也没有时间进行 3 小时的代码冻结，更不用说 3 个月的代码冻结了。我必须在很短的时间内测试好代码。

更重要的是，我还将自己编写的代码部署到了我们的实时服务器上。直到我的代码出现在用户面前，我的工作才算完成。

这是一个重要的区别，原因有很多。第一:当代码无所事事时，它提供的价值为零。也有一些例外，比如原型代码，但是大多数情况下，如果代码没有被使用，它就没有任何用处。也许开发者写它很有趣，但是它没有产生任何用户可以使用的东西，因此公司的收入为零。

没有收入，一个公司就不能付给开发者我们想要的薪水。

从公司的角度来看，当代码出现在用户面前时，工作就完成了。这意味着开发人员需要照看他们的代码，直到它出现在用户面前。

这是好事。这种态度让我们成为更好的开发者。我们不只是确保代码在我们的开发机器上工作，而是进行额外的思考，以确保代码在运行时工作。由于代码不会消失在另一个团队处理的黑盒中，我们最终会看到它的整个生命周期。这提供了更大的主人翁感，并提高了我们做更好工作的风险。

然而，实现这一目标并不仅仅取决于开发人员。公司的结构必须鼓励它。如果你让一个有这种理想态度的开发人员，把他们放在一个他们的工作被移交给另一个团队的组织中，那么他们将开始脱离在用户面前看到他们的代码的需要。他们将开始认为他们的工作已经完成，仅仅是写代码。
