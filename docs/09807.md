# 在岸/离岸程序员布莱克

> 原文:[https://dev . to/shakycode/Blake-the-on shore offshore-programmer](https://dev.to/shakycode/blake-the-onshoreoffshore-programmer)

如果你读过我之前的文章《欺诈编码者的故事》，你就会知道我对胡说八道几乎没有耐心。这是我想与你们分享的另一个开发者战争的故事。为了匿名起见，我将称这位程序员为 Blake。故事中的其他一切都是关于重大痛苦和冲突的真实故事，好吧，这听起来很戏剧化，但对我来说是赤裸裸的。

我当时在一家很棒的初创公司工作，我们有一个名叫“布莱克”的“资深”程序员。布雷克是一个非常有礼貌的人，他碰巧持有海外(印度)的 H1 B 签证。我们雇佣他是为了给我们的团队带来更强的技术洞察力，因为他有近 10 年的软件开发经验和 8 年的 Ruby 和 Rails 经验。我们认为他会给我们急需帮助的团队带来惊喜。

当时，我们公司是一个 100%分散的团队，这意味着只要工作完成，我们就可以在任何地方工作。我们通过 Squiggle 保持联系，进行视频会议和聊天，开始时一切都很好。

布莱克非常安静，真的很难联系，因为他似乎总是在他的个人生活中发生一些事情。我们最初认为这是正常的人性，因为我们都有糟糕的事情发生，但这都是关于你如何管理它。但在与布莱克共事一个月后，我开始注意到一些模式和危险信号。

#### 红旗#1(行动中失踪)

我们保持宽松的工作时间，但我们大多数人被要求从中部时间 10:00 到 17:00 保持在线。通常我们会站起来开始我们的冲刺，但大多数时候布莱克会有连接问题，不会加入我们的视频会议。他声称在谷歌光纤上，但他当时应该在东海岸，而谷歌在他的地区不提供光纤。我们会整天看着他在网上和网下蹦跳。在分布式团队中工作的一部分是拥有可靠的网络访问，以便我们可以继续工作和协作。我们可以通过 Squiggle 在网上看到 Blake，但每当我们联系他时，要么得不到任何回应，要么收到一条固定信息，比如“我正在分析这个问题，今晚晚些时候我会报告我的发现”。有趣的是，没有需要分析的问题，他做的是基本的编程任务和简单的调试分类，这最多需要 1-2 个小时，即使对于新手来说也是如此，更不用说高级开发人员了。

#### 红旗#2(变性宝宝)

布莱克是一个新生女儿的父亲，所以当他的妻子在工作时，他经常不得不离开去照顾他的孩子。这对我们来说不是问题，因为我们中的许多人在完成工作的同时会同时处理多项任务。在与布莱克的谈话中，我们问他是否有一个大家庭，他说，“没有，只有我的妻子和女儿”。

一个月后，布莱克没有来上班，我们收到一封疯狂的电子邮件，说他的小儿子因为发高烧不得不去医院，所以他今天不能上班了。我的老板没有发现，但我注意到他刚出生的女儿不知何故改变了性别。在我们和布莱克在一起的时候，他会在提到他的“儿子”和“女儿”之间转换，而事实上他告诉我们他只有一个女儿，是个女孩。真的很奇怪，但它变得更糟。

#### 红旗#3(不准白天工作)

我们有一个适当的政策，即我们将在正常工作时间内为我们的客户提供支持，但我们足够灵活，如果你需要回避或休息，你可以去哪里。我们从不鼓励员工在下班后工作，但我们中的许多人都这样做了，因为我们对工作和愿景充满热情。但我们开始看到布雷克身上出现了一种模式。

在白天(大约 7-8 小时)，我们听不到布莱克的任何消息。我们试着让他弯曲或松弛，他没有反应。几个小时后的下午，我们会得到典型的回答“我正在分析问题，今晚会有答案”。

在我们和布莱克一起工作的 3 个月里，我们从没见过有人在白天作案。所有的提交都发生在 21:00-22:00 之间，而我们其余的人则在白天和晚上提交代码。好像这家伙是夜间活动的。这最终成为了一个大问题，因为当团队的其他成员都在线或者正在冲刺的时候，他总是不在。这真的很可疑，正如你将在后面读到的，我们找到了原因。

#### 红旗#4(我老婆的车坏了)

我们正处于一个非常紧张的最后期限，我们指望布莱克来帮助我们发布这一功能。我和其他 3 名开发人员负责处理大部分工作，Blake 负责功能 sprint。他应该有一些代码要提交，而我们没有时间发货了。我们在最后期限那天给他施加了压力，他给我们发了电子邮件，“伙计们，我妻子的油箱盖在高速公路上从她的车上掉了下来，她被困住了，所以我今天不能工作，因为我必须把车拖走，然后送她去工作”。

你好。听说过 AAA 吗？见鬼，99%的汽车保险公司都有某种形式的道路救援。我的意思是，如果他的妻子遭遇了可怕的事故，而他需要在这里，那就不一样了。但是说真的，她的车坏了，她可以很容易地叫一辆拖车。相反，布莱克决定他需要照顾它。

我们都觉得这是一个扯淡的借口，我们只是接管了他的那部分工作，在没有他的情况下发货，同时几乎满足了我们的最后期限。但这真的不是一个可行的借口，听起来像是某种狗屁编造的故事。从那时起，我们都怀疑他。

#### 红旗#5(节点/跨站点脚本)

我们正在开发一个新的 API，它可以帮助我们更好地聚合和规范化数据。我们的应用程序是一个整体，安装在几个不同的服务器上(每个客户一个)，所以我们必须将这个 API 引入代码库，并向它传递参数，以便它可以从应用程序实例所属的正确组织(即 Acme Company)传输数据。在 API 开发期间，Blake 说我们的 API 会受到跨站点脚本的影响，我们绝对必须使用 NodeJS 来防止 XSS 攻击的发生。

我不是 NodeJS 开发人员，但我知道引入一个成熟的 MVC JavaScript 框架并不能防止 XSS 攻击，这是我们必须通过 Rails 内部处理的事情。发现没有会导致 XSS 攻击的攻击媒介出现。我们觉得这很奇怪，他一直在推动 NodeJS。不想扩大我们的技术堆栈，对 Node 没有任何需求，我们推后并对 Node 说不，在 Rails 中处理它。他的回答是，“我不能对我们的应用程序和 API 中的公开漏洞负责”。

有趣的是，我们找了一家非常有信誉的安全公司来攻击我们运行 API 和 Rails 框架的临时服务器，他们无法执行 XSS 攻击或除 DDoS 之外的任何成功攻击。

所以他是从哪里想到 NodeJS 会解决 XSS 攻击向量的，我不知道。我们向我们的首席技术官提出了这个问题，他不高兴，因为没有取得进展，并说，“去他妈的节点，我们坚持使用 Rails 和我们正在构建的 API”。还是怪怪的。

#### 红旗#6(比赛状态)

当时我正在处理一个伸缩问题，它基本上是一个 SQL 存储过程，实时计算一个基于得分的算法。这是在页面加载期间发生的，在有许多用户的客户端上，页面加载时间会超过 1-2 分钟，这是完全不可接受的。因此，我认为我们并不真正需要对数据库中的每个主题进行实时分数计算，并决定将其转移到 Sidekiq Worker(后台作业),该工作程序将在请求之外每 3 分钟触发一次，最终导致页面加载的响应时间超过 400 毫秒，并提供“足够好”的实时分数数据。

当我将 PR 分配给 Blake 时，他拒绝了它，并说我由于没有实时计算而引入了竞争条件。他接着说，如果我们将评分工作转移到后台工作人员，我们会经历数据库死锁。

根据我对我们的代码库和比赛条件的了解，这毫无意义。所以我和另一个开发者跳过他的头，去找我们的 CTO，告诉他这个情况。布雷克被整个车队问及他的比赛状况和死锁理论的解释，但他能想到的只是，“我仍在分析问题，但今晚会有解决方案”。如果闻起来像扯淡，那很可能就是。

#### 红旗# 7(1 月流感)

当我们试图满足最后期限时，布莱克有许多借口不去工作或运输代码，但有一个特别让我觉得奇怪。一天，布莱克发邮件给我们，说他得了流感，今天不能上班了。我们祝福他，一个星期都没有他的消息。我们最终通过电子邮件联系了他，因为他不接电话，他回复说他仍然生病，不能工作。有趣的是，他的妻子应该是一名急诊室医生，所以她可以很容易地给他开一些达菲或其他抗病毒药物来对抗流感。我甚至问他妻子能否给他开些药，他说不行，因为她在上班，联系不上。我觉得很奇怪的是，他不能给他的妻子发短信，让她给我们开药方，或者让她的医生同事开。或者更好的是，布莱克可以简单地把它吸起来，然后开车去紧急护理诊所接受治疗。

几个星期过去了，我们终于让布莱克在 Squiggle 上进行视频聊天，他看起来非常健康，但在身边放了一盒纸巾，尽管他没有流鼻涕。他咳嗽得很厉害，但你可以看出这是被迫的咳嗽，并不是疾病的真正症状。我不是医生，但我知道一种典型的在家治疗的流感病毒在消失前最多能持续 14 天。4 周后，布莱克神奇地“康复”了，并问需要做什么。抱歉，伙计，我们已经在你不在的情况下忙了一个月了。与此同时，你得到了报酬(非常好),却没有为团队做出任何贡献。

#### 红旗#7(我不能和你配对)

作为一个分布式团队，意味着我们非常重视结对编程。在我与 Blake 一起工作的 3 个月中，有一次我与他配对，他只是看着代码，然后打断我说，“我需要分析问题，我会再联系你”。现在，我们正在讨论一些小错误，比如 NILClass 问题或者数据没有被规范化，或者简单的 4 行函数只需要 5 分钟的重构就可以通过我们的测试套件。没什么大事。

他只会使用 Teamviewer 进行配对，该工具在所有平台上都可用，但我们是一家苹果商店，我们给包括 Blake 在内的每位员工都发了全新的 Macbooks，这样我们就可以使用 ScreenHero 等配对工具来保持工作效率。布雷克拒绝使用 Macbook，总是找借口说它快崩溃了，不能用了。我们甚至更换了他的 Macbook 两次，他仍然抱怨它不能工作。所以最终的结果是我们从来没有和布莱克配对，这真的是我们文化的对立面。应该指出的是，Blake 有一个他将要使用的 Ubuntu 和 Windows 桌面，由于安全限制和我们数据的性质，这两个桌面都没有得到我们 CTO 的批准。我们有 2FA VPN 连接，我们公司的政策是只在工作中使用我们的工作笔记本电脑，而不是个人电脑。布莱克从我们雇佣他的那天起就违反了规定，拒绝使用公司提供给他的 Macbook。

因此，我们尽最大努力与 Blake 配对，但每次我们要求配对时，他都会给出典型的回答:“我正在分析问题，稍后再联系你”。然后神奇地在 21:00–22:00 我们会看到某种 Github 活动，它与之前提到的白天工作无关。对于记录来说，提交是可怕的，特别是对于一个有这么多经验的高级开发人员来说。这些是从 Stack Overflow 中剪切/粘贴的解决方案，没有注明原始来源。

#### 我们发现了什么

因为所有这些借口和废话因素，我们决定做一些秘密调查，这是我们在解雇他之前发现的。

*   Blake 从 APNIC(亚太网络)的 IP 地址多次登录他的 Ubuntu 机器，特别是印度。他的机器上有多个用户帐户，所有这些帐户的主目录中都有一个 github repo 的分支。是的，Blake 在日常工作中使用离岸开发者来完成他的工作。

*   我们进一步发现，由于作案时间的原因，他正在将自己的工作外包。在我们与 Blake 合作的 3 个月中，我们发现他的提交时间总是在 21:00-22:00 之间，在此时间范围之外没有其他提交。你猜怎么着？在印度是 08:00–09:00。

*   我们还发现，他与前雇主签有合同，从事另一份全职工作。我们是如何发现这一点的？他犯了一个错误，在他的 Linkedin 个人资料上离开了他的旧雇主，在一次秘密任务中，我们的一名开发人员打电话给公司，要求与他通话，公司说他出去吃午饭了，但一小时后会回来。

*   我们还通过 github 发现，他的账户正被另一个(或者几个)开发者使用。我们怎么知道的？因为他们的 git 名称配置列出了另一个名称，当我们查找它时，它是来自班加罗尔的 oDesk 开发人员。终结了。

*   我们查看了 Blake 的 github，看看他在做什么样的项目，他的账户里有一堆基本的 Ruby 和 Rails 教程。他开始教育服务了吗？负面幽灵战士，那家伙真的没有经验。

*   我们统计了他与我们一起工作的 3 个月的提交量，最终结果是 8 次提交和 27 行代码提交。与此同时，我们其余的人走了很长一段路，犯下了更多的错误。布莱克只是没有完成工作，而当他完成工作时，这不是他的准则。我们怎么知道的？因为我们查看了样式和注释，并将其与我们之前发现的 oDesk 开发人员进行了比较，它与 t。

*   在几次因不归还公司财产而受到法律诉讼威胁后，当 Blake 最终归还他的 Macbook 时，它是在我们寄出的原始包装中到达的，无论如何都没有安装。他从来没有使用 Macbook 来找出答案，最终使用了他的 Ubuntu 桌面，这违反了政策。在 Blake 被解雇后，我们中的一个人开始研究他的 github，果然他有一些与他从未停止工作的旧公司有关的回复。布莱克肯定欺骗了我们。

#### 吸取教训

我们都遵循荣誉制度，相信信任和诚信。这是我们公司的基础。不幸的是，布莱克没有诚信，不值得信任。遗憾的是，我们花了 3 个月才搞清楚这一切，同时我们拒绝了几个重量级的高级开发人员，因为这个职位已经由 Blake 担任了。

我们学到了一个艰难的教训，在布莱克被解雇后，我们在候选人审查方面提升了我们的水平，并最终雇佣了真正把我们带到下一个层次的合法开发者。

我对那些和世界上的黑人一起工作的人的建议是。早点叫他们出来，然后快速开火。你的公司和团队的诚信取决于是否有合法、诚实、有技能的人。你值得更好的，我知道我们确实值得。

这是另一个 dev 战争故事的结局。shakycode
