# 为什么我们必须“问为什么”

> 原文:[https://dev.to/briannipper/why-we-have-to-ask-why-2ph0](https://dev.to/briannipper/why-we-have-to-ask-why-2ph0)

[![](../Images/d742db4cb01aec18f1c2c26da1ee9bd7.png)](http://2.bp.blogspot.com/-_KRAH1G8iQA/TzLKOmaiy6I/AAAAAAAAGXU/6pWMx77t8Us/s1600/WhyImage.gif) 
我总是很有兴趣去寻找那些似乎是“聪明”人共有的性格特征。当我使用“聪明”这个词时，我并不意味着他们拥有一些别人无法获得的知识，也不意味着他们比普通人更聪明(尽管我怀疑如果我们失去了智商，那可能是真的)。明确地说，我通常不把自己归入“聪明”一类，我倾向于认为我只是碰巧有一个好的记忆力，这给人以聪明的印象，并且容易问这个问题**为什么，**，我观察到这是“聪明”人的一个特点。

当人们被动地接受提供给他们的信息而不问为什么时，结果往往是接受信息的人对主题只有粗略的了解。在某些情况下，我完全同意“正确的方法”是只获得粗略的概述，以便能够快速地投入到任何任务中，首先获得所需的信息。然而，这种粗略的理解似乎经常成为大多数人的默认水平，即使在信息的使用变得更加重要之后。换句话说，当一个人获得了完成给定任务的“最少”信息后，就没有进一步的努力去更好地理解为什么“事情是这样运作的”。

在一家大型企业(比如我目前所在的企业)，不问为什么的最终结果是，人们在工作中做完全不必要的任务，因为默认的理解是“按照我之前的人做的方式去做”。我承认，就让某人学习一系列任务所需的精神能量而言，这可能更容易；然而，我想说的是，任何一个必须在几个月甚至几年的时间里重复完成一项特定任务的人，都会从对试图完成的任务的更深入理解中受益。

对于一个开发者来说，提出为什么的问题是不可选择的。如果你是一名开发人员，而你一天至少一次没有问“为什么”，那么你很有可能不是一名优秀的开发人员。人们很容易陷入这样一个陷阱，认为获得一个好的需求文档意味着“为什么”的问题已经被问过并回答了，你只需要实现需求的技术方面；然而，这种情况几乎从未出现过(也许除了最基本的特性)。

在开发过程的每一个阶段，所有参与方都将受益于更深入地理解在开发过程的每一步提出为什么的问题。事实上，甚至有一种叫做 [5 个为什么](http://en.wikipedia.org/wiki/5_Whys)的“方法”,它展示了连续问 5 个问题(至少 5 次)将如何有助于发现根本原因。值得注意的是，正如杰夫·阿特伍德(因[的恐怖代码](http://www.codinghorror.com/blog/)而出名)所引用的，如果你问为什么很多次答案都是“因为地球上有人”，这很好地表明你可能潜得太深了。

因此，如果你是一名开发人员，不要忘记问为什么，事实上，在生活中，记住问为什么可能是一个好主意，只是为了让事情变得有趣。
