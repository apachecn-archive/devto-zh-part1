# 你可以对开源做出的 6 个非代码贡献

> 原文：<https://dev.to/captainsafia/6-non-code-contributions-you-can-make-to-open-source-1akm>

*最初发表于[OpenSource.com](https://opensource.com/life/16/1/how-contribute-open-source-without-writing-single-line-code)。*

贡献开源！这会在你的简历上看起来很棒！这是令人满意的工作！

在你的职业生涯中，你可能无数次听到人们做出这些陈述，或者类似的陈述。他们没有错——为开源做贡献在许多方面都是一种有回报的努力——但是，当软件工程师建议其他软件工程师为开源做贡献时，他们通常指的是代码贡献。这是一个合理的假设，但现实是，有很多机会可以在不编写任何代码的情况下为开源做出贡献。

怎么会？让我们浏览一下为开源做贡献的一些非代码机会。

## 传道

对开源的非代码贡献通常包括代表项目进行宣传。如果您喜欢最新的 JavaScript 绘图库，并使用它来满足您所有的数据可视化需求，请考虑在技术讲座中分享您的专业知识。这是提高你自己的声誉和吸引更多用户的好方法。

## 报告错误

更多的用户意味着更多的错误报告。更多的错误报告意味着更多的错误修复。更多的错误修正意味着更好的软件。没错！你现在已经间接地，但是有意义地，在没有写一行代码的情况下对软件的改进做出了贡献。

## 导师

有时候那些错误报告可能有点…相关信息很少。项目的核心开发人员需要花很长时间与错误报告的作者合作，才能完全理解问题的范围。这些宝贵的时间可以用来开发项目。这就是你进来的地方！指导第一次编写 bug 报告的作者编写一份好的 bug 报告是一个有价值的细致入微的过程，可以为任何开源项目的核心团队省去许多麻烦。这可能涉及到你写一点代码，但理想情况下，你会在整个过程中指导另一个开发人员。

## 写

现在，如果你不喜欢公开演讲，不喜欢 bug(我不能责怪你)，你可以以开源的名义写文字，而不是代码。关于特定项目的信息丰富的博客帖子是有用的，并再次吸引更多的用户到项目中来(以及随之而来的所有好处)。如果博客帖子对你来说太难了，考虑在邮件列表、StackOverflow 或 Twitter 上回答关于该技术的问题。这是一个很好的方法，不仅可以发展你自己的技术知识，还可以有助于收集关于技术的信息。

## 举办一次聚会

如果你像我一样是一个外向且执着的项目经理，你可能会考虑在你所在的城市围绕特定的开源工具举办研讨会或聚会。这给了你一个围绕项目建立非数字社区的机会。这些社区对于那些负担不起一直在线的人(是的，他们存在，是的，他们很重要)和那些喜欢在与其他用户就软件进行互动时用虚拟形象来面对的人来说是很有价值的。

## 提高安全性

最后，在一些开源项目中经常被忽略的一点是安全性。如果你有网络安全或安全测试的经验，考虑为项目的改进贡献你的技能。寻找并提供安全漏洞的修复是改进软件和项目用户体验的直接方法。

## 结论

我从来不喜欢开源这个术语，因为它迫使开发人员在字节、位和 80 字符宽的行的狭窄范围内思考。开源远不止这些。它是关于开放的知识，开放的分享，开放的成长，开放的学习，开放的辩论，以及不断向前推进。大多数伟大的软件都不是在计算机前创建的，你没有理由限制自己通过文本编辑器和键盘为开源软件做贡献的能力。
