让我们继续看一个用于存储每一集《星际迷航》细节的数据库模式。如果您不熟悉数据库，那么模式就是数据库结构的设计。

我们有三个表，插曲，插曲作者和作家。更多细节请看我的上一篇文章。有人建议，对这种结构稍加改变，就可以容纳更多的创意人员。

让我们将 Writer 重命名为 Credit，将 EpisodeWriter 重命名为 EpisodeCredit。现在，任何参与一集的创作人员都可以存储在信用表中。让我们更改 EpisodeCredit 并添加一个名为 CreditType 的额外列。CreditType 只是一个文本字段，存储创意人在该集扮演的角色，可以是导演、编剧、演员、科学顾问等。

如果有人想重新创建我在这里描述的数据库，我将 SQL 保存在一个单独的页面上，可以在这里找到。

我们现在能够在剧集表中存储与剧集相关的信息，并在信用表中存储任何有创意的人的信息。我们还可以向数据库中添加什么？一个可以用来记录一集最后看的时间的表怎么样？我可能很奇怪，但有时我想看《星际迷航》中我喜欢的一集，但我已经很久没看了。

最后一个被监视的表非常简单，只有一个 datetime 字段和一个 episodeId。如果你想跟踪你的朋友们看了哪些剧集，这可以进一步扩展为一个 userId 字段。

另一个想法可能是用某些主题或话题来标记剧集，如克林贡语剧集或生命的意义剧集或柯克与电脑对话至死剧集。同样，这是一个相当简单的表，包含 TagName 和 EpisodeId。
