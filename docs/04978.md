# 为什么我需要一个测试用例？

> 原文：<https://dev.to/gokatz/why-the-hell-i-need-a-test-case>

## 一个晴朗的日子🌅

我是一名前端开发人员，主要专注于使用 [ember.js](https://emberjs.com/) 和相关生态系统的开发。在我开发生涯的初始阶段，我非常专注于构建东西，而不是测试那些部分。新手开发人员的典型心态！你相信这种情况会持续一年吗？是的，这是痛苦的事实。一个晴朗的日子，我的团队决定至少对所有模块进行验收测试。最终，是我开始了旅程。那天我感到非常不幸，没有体验到测试用例的潜力。

## 初尝用试例

带着那种深深的悲哀，我开始热衷于进入生态系统开始。因为 ember.js 有一个内置的测试环境，所以选择是显而易见的。我们使用 ember-cli-mirage 进行数据模拟(哈..足够的技术堆栈😉).有了一些代码参考，我设法编写了我的第一个验收测试来测试特定的页面是否被呈现。幸运的是，我在一堆尝试中成功了。

尽管如此，我还是没有动力采用 [TDD](https://en.wikipedia.org/wiki/Test-driven_development) 。我认为这是我的任务之一，而不是一种编写代码的方式。随着时间的推移，回购充满了大量的验收测试案例，日子过得很好！

## 第二个晴天🌅🌅

大约 8 个月后，在另一个晴朗的日子里，我开始了特性开发。通常，我在欢呼中成长！我手动测试了这个特性，它的工作情况和预期的一样。最后，我为该特性编写了一个测试用例，其外观如下👉 😏。我运行了特定的案例——测试案例在 dev 中通过。然后呢？我把我的修改推给了审查。超级爽！

此后事情开始发生。我的分支的测试用例构建也失败了，因为我甚至没有修改一个字母(在应用程序或测试中)😱。我立即重新运行了测试构建，因为我怀疑运行程序，因为我对我的测试用例非常有信心(是的，它在我的开发中运行得很好，对吗？).没有运气！又失败了。然后我决定调试失败的案例。

【至此，你可能已经想通了！]

是的，我的代码有问题。我对一些常用文件进行了修改，以实现一些功能。那是问题产生的地方。我的应用程序中的几乎所有模块都共享这些公共文件。我的特性(模块)发生了惊人的变化，但不幸的是，并不是所有的模块都发生了变化😑。其他一些模块被破坏了，就像

[![broken gif](img/f9513fc3893429f894f8cf1509791427.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--eRBCaKWu--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/5uorey6txcvde39zmn43.gif)

感谢那个曾经的人。我修改了代码来支持所有的模块。涌入我脑海的问题是，*如果我们没有覆盖所有模块的测试用例会怎么样？*，*如果错误代码投入生产会怎样？影响可能比我想象的更糟。*

你知道，那天我爱上了测试用例❤.是的，测试用例是好的😇。所以，EOD，

“我到底为什么需要一个测试用例？”
*- bcoz，我爱我的产品，不希望我的客户体验到我产品中的任何一件垃圾。*
