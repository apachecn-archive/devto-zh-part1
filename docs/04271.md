# 感觉驱动开发

> 原文:[https://dev.to/johnpaulada/sense-driven-development-443](https://dev.to/johnpaulada/sense-driven-development-443)

作为一名开发人员，教给我很多我现在所知道的东西的人之一是我们前 TestLab 团队领导 [Czar Pino](https://medium.com/@czarpino) 。有时当他在做决定或解释某事时，他会问，*“这有意义吗？”*。
我相信这是一种思考问题的好方法。当我们开发软件或实际做任何事情时，我们应该总是问自己，*“这有意义吗？”*。当我们做决定时，它们应该总是明智的。

> 明智的:(指声明或行动过程)根据智慧或谨慎选择的；可能会有好处。(Google.com，2017)

## “咄，那很明显”

[![no shit](img/45c5837499a37791b09e2abe674a227c.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s---Kwp6eCQ--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/5opcy7i7cc06zt0d8vce.gif)

你可能会认为这是显而易见的，但是我们经常做一些没有意义或者根本没有意义的决定。这都是因为我们的大脑和我们所谓的[认知偏差](https://www.wikiwand.com/en/Cognitive_bias)，顾名思义，这是一种我们思考方式的偏差。一个例子是[确认偏差](https://www.wikiwand.com/en/Confirmation_bias)，其中我们倾向于寻找证据来确认我们喜欢的或我们已经知道的。我强烈推荐看看这个[列出的认知偏见](https://www.scribd.com/doc/30548590/Cognitive-Biases-A-Visual-Study-Guide)作为参考。

为什么会这样？这是因为我们的大脑是高效的。听起来违反直觉？我知道，对吧？它通过走捷径变得有效率，而这些捷径导致了这些认知偏差。在过去，当我们还是食肉的穴居动物时，这些捷径是非常重要的——事实上，我现在仍然是(lol JK)——因为当我们即将被一群剑齿虎猫群殴并撕成碎片时，我们必须做出迅速的决定。这些大脑捷径在当时是相关的。现在，没那么多了。这是丹尼尔·卡内曼在他的《思维快与慢*一书中理论的“**思维快**或“**系统 1** ”部分。如今，我们需要学会慢慢思考——更精于计算，花时间做决定。这样，我们就有更好的机会避免认知偏差。另一种减少认知偏差的方法是让值得信任的人来评估我们和我们的决定。我们都有对他人的认知偏见，就像基本的归因错误，但至少有两个头脑而不是一个，你可以彼此坦诚。
另一种“降低认知的方式”是问这样一个问题，“这有意义吗？”。*

## 连锁反应

这个问题是思考我们思维过程的第一步。这听起来很元，确实如此，但这让它变得很棒。这个问题引发了很多其他问题，比如:

*   为什么这说不通？(缺点)
*   为什么有意义？(优点)
*   我为什么要做那个决定？(这种想法背后的原因和条件)
*   我是如何做出那个决定的？(导致做出该决定的想法)
*   是否矫枉过正？(过早优化)
*   我真的需要它吗？(简约主义)
*   有没有更好的办法？(更好的可能性)
*   这个决定的成本是什么？(嗯，成本)
*   其他人明白我刚才说的吗？(解释清楚)
*   一个 5 岁的孩子能理解吗？(如果不能，说明你对题目不够了解)

还有其他类似的。这个问题会引发一连串的问题，帮助你确认或否定那个决定——看看它是否真的是“*根据智慧或谨慎*做出的”。

让我们试着把这个放在上下文中。

## 场景:构建您的技术堆栈

(这是固执己见所以请不要杀我。)

想象一下，你是一家开发音乐应用的小公司。你需要能够在 2 个月内建立你的原型。您的技术主管决定您将使用以下堆栈:

1.  API:走上的图形 SQL
2.  网页前端:**魔族&替身**
3.  移动:**颤动**
4.  数据库: **MySQL**

你的团队知道什么:

1.  休息
2.  PHP ( **Laravel** )
3.  Javascript ( **Node.js，Meteor，React & Redux**
4.  **MongoDB**
5.  **MySQL** 的实现

你开始被你在这里看到的一切所淹没。现在是问这个问题的好时机，“这有意义吗？”。让我们从头到尾走一遍，找出答案。

### API

> **使用 GraphQL 超过 REST 有意义吗**？

嗯，它有助于节省带宽，而且对于 API 的消费者来说比 REST 更容易使用。

> **为什么在 REST 上使用 GraphQL 没有意义？**

团队知道休息。他们可能熟悉 GraphQL，但不知道如何实现它。

> **用 Go over PHP 或者 Node 有意义吗**？

Go 比 PHP 或 Node 快，因为它是底层的，对并发有很大的支持。

> **为什么**使用 Go over PHP 或 Node 没有意义？

你只有两个月的时间来制作原型。你的开发人员没有一个懂 Go 或者 C 的(也是低级的)。与开发时间相比，您不知道性能的提升是否值得。

### Web 前端

> **用 Inferno 有意义吗**？

嗯，你的团队知道 React 和 Inferno，没有太大的区别，所以开发时间不会有太大的不同。除了地狱通常比反应快。问题是:你真的需要那样的速度吗？你真的需要挤出最后一滴速度吗？答案取决于你。

> React 16 怎么样？

这是另一件需要考虑的事情。React 显然因为新的光纤架构而变得更快。这取决于你。如果性能提升真的很重要，也许你可以检查 Inferno 与 React 16 相比是否在速度上有显著优势。如果没有，也许您应该坚持使用 React，因为开发人员已经有了使用它的经验。

> **用 Redux 有意义吗**？

你的团队已经知道 Redux 了。使用 Redux。

> **为什么使用 Redux 没有意义？**

它会是一个巨大的应用吗？它会有很多活动部件吗？如果是的话，使用 Redux 是个不错的选择。此外，如果你的团队对**函数式编程**感兴趣，Redux 非常适合这种模式。如果它不会是一个巨大的应用程序，或者如果你确信你团队的每个成员都会编写干净的代码，你可以使用 MobX。如果你更喜欢**面向对象范式**而不是**函数范式**，MobX 也是一个不错的选择。更好的是，根本不要使用状态管理库。对于原型，也许你可以尝试 MobX，但想想未来:应用程序会比现在复杂得多吗？如果没有，也许你可以继续使用 MobX 或者干脆放弃它。如果会更复杂，从一开始就使用 Redux。

### 移动

> **用颤振有意义吗**？

不，不是的。你没时间学飞镖。用 **React Native** 或 **Cordova** 代替，因为你的团队已经知道 React。

### 数据库

> **用 MySQL 有意义吗**？

你的团队了解 MySQL。这将使你和你的队友更容易开发你的原型，因为你已经习惯了。

> **为什么用 MySQL 没有意义**？

你的应用将来可能会有大数据。如果是这样的话，也许 MongoDB 更适合。

## 德·博诺的六顶思考帽

爱德华·德·波诺博士创造了一个由六种不同类型的思维组成的系统，他称之为六顶思考帽。
思考的帽子如下:

> **白帽子**要求**已知或需要的信息**。“事实，只是事实。”
> 黄色的**帽子**象征**光明和乐观**。在这顶帽子下，你探索积极的一面，探索价值和益处。
> **黑帽子**是判断——魔鬼的代言人，或者为什么有些事情可能行不通。发现困难和危险；**哪里可能出问题**。可能是最强大和最有用的帽子，但如果过度使用就会出现问题。
> 红色帽子**象征着感觉、预感和直觉。当使用这顶帽子时，你可以表达情绪和感受，分享恐惧、喜欢、不喜欢、爱和恨。
> **绿帽**注重创意；可能性、替代方案和新想法。这是一个表达新概念和新观点的机会。
> **蓝帽子**用于管理思考过程。是**控制机制**确保了六大思维准则得到遵守。**

例如，你可以从戴上白帽子开始，思考一些确凿的事实和严格的要求。然后你可以戴上绿帽子，疯狂地想出新的点子。之后，你可以“戴上”黄帽子，探索某个想法的最佳部分和“最好的”，然后“戴上”黑帽子，思考一切可能出错的地方。

我认为这是一种非常有效的系统思考方式，因为人们倾向于只戴一顶或两顶思考帽。这个系统给了我们打破常规思维模式的机会。

## 结论

做决定时，你可以尝试以下方法:

*   总是考虑你正在做的决定
*   确保他们是明智的决定
*   确保你做了调查
*   确保问自己相关的问题
*   确保问你的队友相关的问题
*   总是检查其他的观点(尝试其他的思考帽)
*   问问自己，“这有意义吗？”

就是这样！我希望这能帮助你做出更好的决定。别担心，每个人都会做出错误的决定。我也不例外。我们可以更好地做出好的决定。
