# 关于 O 型圈和软件腐蚀的备忘录

> 原文：<https://dev.to/janvanryswyck/memo-on-o-ring-and-software-erosion-1jig>

迄今为止，我读过的最有趣的文件之一是罗杰·布瓦乔利关于 O 型圈腐蚀的备忘录。他写的这份备忘录最初的目标读者是 1985 年莫顿·瑟科尔的管理人员，大约是挑战者号灾难发生前的六个月。

我发现[整个故事](https://en.wikipedia.org/wiki/Roger_Boisjoly#Challenger_disaster)的惊人之处在于它与软件工程领域的相似之处。我们软件开发人员可以很好地理解这个故事。我个人也不止一次遇到这种情况。见鬼，我们中的一些人总是处于类似的情况。

罗杰·布瓦乔利试图阻止航天飞机的发射，可惜没有成功。在他的特殊情况下，航天飞机的失败将会夺去机上宇航员的生命。但是我们正在制作的软件呢？在大多数情况下，软件故障不会导致人命损失。这将花费公司一大笔钱。最坏的情况是，它的声誉将和钱一起被冲进下水道。

但是，当我们在开发的软件中遇到可靠性问题时，我们应该怎么做呢？首先，我们对这些问题的严重性和影响进行评估，并相应地解决它们。但是，如果我们要花几天、一周、两周甚至几个月的时间来解决问题呢？嗯，然后我们通常转向管理人员，用我们最好的非技术术语解释正在发生的事情，并向他们询问我们需要的时间(也就是预算)。但是棘手的部分来了。如果他们不想听呢？如果他们不把我们当回事呢？如果他们不相信我们的发现是正确的呢？如果他们只是忽视我们，想知道为什么我们仍然有精力在这个死亡行军之后或期间来和他们说话呢？如果他们倾听我们的请求，全心全意地同意我们所说的一切，把我们送回走廊，然后除了更多的业务需求和功能要求之外，什么也不给我们，那该怎么办？

现在我们来谈谈这个百万美元的问题。当我们没有时间做正确的事情时，我们软件开发人员会做什么？那么，我们可以试着在业余时间解决问题，牺牲掉我们与家人和朋友在一起的宝贵时间，甚至牺牲掉接下来几天、几周、几个月甚至几年的睡眠时间？我不认为这是非常可持续的。或者我们可以简单地忽略管理，停止添加新的特性和业务能力，并开始以我们自己的名义修复东西，就像某种软件开发团队叛变一样？我猜这会导致一个或多个开发人员被解雇。或者，我们可以在社交媒体上发起一场运动，宣传软件开发人员在这家或那家公司是如何受到不公正对待的。有人会在乎吗？这可能会导致一些人被解雇。

那么，作为专业软件开发人员，我们应该做些什么才能在这种情况下把事情做好呢？也许软件工艺运动的人们可以起草一个带有具体行动的脚本，来指导软件开发人员，当他们发现自己处于与 Roger Boisjoly 相同的情况时？

我很想知道，如果只有 10%的软件开发者遵循这样一个宣言中描述的行动，我们的行业会发生什么。这难道不是有用的建议吗？
