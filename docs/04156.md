# 如何充分利用拉动式请求

> 原文：<https://dev.to/kylegalbraith/how-to-make-the-most-out-of-pull-requests-90>

拉取请求对开发团队来说至关重要。它们提供了一种机制来检查一段代码以及它在更大的上下文中的位置。它们是提供反馈的关键检查点。

你是否打算为一个开源项目做贡献。第一次进入开发队伍。或者你是一个经验丰富的程序员，想要改进过程。充分利用拉式请求将永远改变你的开发过程。

### 没有一个人拥有代码

为了使拉式请求有效，团队中的每个人都必须在门口检查他们的自我。开发人员很容易沉迷于他们的代码。如果他们乐于接受反馈，这是可以的。

拉动请求是团队中其他人提供反馈的机会。为什么他们需要提供反馈？

*   验证代码是否符合团队定义的标准。
*   确认变更符合技术要求。
*   以了解未来维护的变化。*确保实现是可支持和可维护的。

这是三个不容置疑的原因。拉动式请求是团队提供反馈的机会。团队拥有并维护代码，而不是一个开发人员。如果不满足这个条件，就会破坏团队文化和拉动式请求流程。

### 如何做好被考核人

你已经检查了你的自我，并希望从你的团队得到反馈。您已经竭尽全力开发一项功能，并准备提出一个拉取请求。作为被评估人，您必须尽自己的职责，以充分利用拉动式请求流程，这一点非常重要。

*   在打开请购单之前，验证与代码通过相关的所有测试。
*   在打开 PR 之前，删除任何不必要的代码或注释。
*   尽量保持你的 PR 小，因为小代码块比大规模变更集更容易审查。
*   尽可能向评审者提供更多的背景信息。使用公关描述来解释你做了什么，为什么这么做，以及为什么你认为这是正确的解决方案。说出你所做的任何假设和你不确定的任何事情。

这些准则改进了拉取请求过程。它们为评审者提供了有效、简洁和清晰的反馈机会。最后一点是最重要的。评审者将会直接进入你的拉动式请求。他们会有一个心理环境的转换。认识到这一点，并提供尽可能多的关于代码的上下文。

### 如何成为一名优秀的审稿人

与成为一名优秀的评审者同等重要的是成为一名优秀的评审者。对于任何给定的开发团队来说，这都是一个重要的角色。评审员正在确保另一个开发人员的代码符合团队协议。这些协议可以是团队协议、技术需求或特性请求。

重要的是要认识到，一个人正在向评审者寻求反馈。如果您发现自己是拉式请求的审核者，请记住以下几点。

*   不要做一个混蛋。
*   始终验证所有测试是否通过。
*   切勿盲目批准拉动式请求。验证这些变化，并给予你全部的关注。
*   理解编写代码的上下文。
*   如果有不清楚的地方就提问。

一个好的拉请求应该给评审者信心。如果没有，那也没关系，因为现在是捕捉这些东西的时候了。如果作为评审者，你对公关没有信心，那么就提供建设性的反馈。

*   解释缺少什么以及如何改进。
*   帮助开发人员理解为什么这些东西是必要的。
*   记住不要做一个混蛋，相反要乐于助人。

如果他们是新开发人员，那么你可能需要做的不仅仅是解释问题。也许你需要用一个例子来帮助他们。花时间帮助开发人员理解为什么这个拉请求还没有准备好。通常，评论者不会这样做。它浪费时间，挫败开发者，并在公关过程中引入摩擦。

帮助别人。

### 结论

拉请求代表反馈的检查点。根据您正在处理的项目，拉取请求流程会有所不同。但是反馈过程总是以某种形式存在。

总会有一个开发人员需要其他人的反馈。这些指导方针讨论的是拉式请求，但它们适用于一般的反馈。作为一个需要反馈的人，解释你在哪里，你如何到达那里，以及你正在假设或不确定的事情。然后，作为提供反馈的人，不要做一个混蛋，把你的全部注意力放在问题上，并帮助其他人。
