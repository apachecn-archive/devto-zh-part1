# 你不一定是对的

> 原文:[https://dev.to/craigbuchek/you-don-t-have-to-be-right-l9h](https://dev.to/craigbuchek/you-don-t-have-to-be-right-l9h)

不久前，我在一次求职面试中被问到“你如何让你的队友相信你是对的？”我用可能是最令人惊讶的答案回答道:“我没有。”

我花了很多年才意识到正确并不十分重要。尤其是当有不止一个正确答案的时候——通常都是这样。团队合作更重要。被尊重和尊重他人更重要。

那次面试几周后，我和问我这个问题的人结对编程。我们都是高级开发人员，所以我们都有很多经验和基于这些经验的观点。我们正在编写一个 shell 脚本，我们都有一个关于如何编写代码的想法。我让他继续他的想法。这是一个非常好的主意；可能比我的好。但是我们配合得非常好。我会让他完成一些代码，我会想办法让它变得更好。然后他会想办法让我的代码更好。我们最终得到的比我最初的想法好得多，他最初的想法也是如此。我对此进行了评论，他表示同意。我们离开结对会议时，为完成了一些罕见的事情而感到兴奋——两个非常有经验的程序员提出了比我们想象中更好的代码。

我喜欢把写代码想象成一条通向目的地的道路。目标是好的代码，做它应该做的，可读的(揭示意图)，简洁的，分解良好的，以便将来容易改变。但是有很多途径可以到达目的地。尤其是在结对编程时，随着想法的分享，代码会变得更好，让你离目标越来越近。有时不同的路径以相同的代码结束；有时它们会以不同的代码结束。但是如果代码很好地完成了它的工作(根据那些标准)，你最终得到什么代码并不重要。

回想原来的问题，我觉得答案稍微细致一点。(我很确定我接着做了解释，但我不记得细节了。)在很多情况下，这真的无关紧要。如果有不同的路径通向同一个目的地，并且它们都大致相等，那么我的规则有效。即使你尝试的第一条路(一条“错误的”路)不成功，它也能起作用，只要尝试一条不同的路足够容易。我发现大多数日常编码都符合这些条件。

但是在有些地方，第一次出错的代价很高。在这些情况下，在开始之前花一些时间思考和讨论替代方案是值得的。软件架构是我想到的最重要的一个。一般来说，权衡每种选择的利弊，应用以前的经验效果最好。

所以下次你想说服别人你是对的时候，抑制住冲动。相反，让他们告诉你他们在想什么。它们可能会让你大吃一惊。更重要的是，你可能会让自己大吃一惊。
