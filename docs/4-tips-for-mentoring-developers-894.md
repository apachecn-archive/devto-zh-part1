# 指导开发人员的 4 个技巧

> 原文：<https://dev.to/d_ir/4-tips-for-mentoring-developers-894>

我工作的一个重要部分是指导初级开发人员。做导师就像做导游。你必须向人们展示所有可以看到的东西，让他们以合适的速度前进。但是要注意:走得太快，他们不会把所有东西都吸收进去；走得太慢，他们会感到厌烦。

这里有一些对我有用的想法。

### 1。不要对代码给出过于具体的反馈

技术指导的很大一部分是审查代码。不幸的是，当有些人听到“请检查这段代码”时，他们会回复“给我发送一个拉取请求，这样我就可以做评论了”。这导致了逐行吹毛求疵，这是像 GitHub 的 pull requests 这样的代码审查工具所鼓励的审查风格。

相比之下，口头的、面对面的反馈，我们更有可能不那么详细，而是专注于更大的问题。更不用说，通过语言交流，我们正在进行双向对话，这无疑将为每个参与者带来更多的见解。

评论风格有什么问题？嗯，我认为设计感来自于探索，而不是遵循既定的规则。最近，我的一个学员对这一行 Java 代码提出了一个问题:

```
new EventProcessor().process(event); 
```

哪个才是正确的风格？我不想说。我更愿意让学员自己去发现这一点。

设计感需要练习。写 1000 个循环，1000 个函数，你最终会明白的。戴上导师的帽子，我关心的是代码是否有效，是否“足够干净”，而不是它是否完美，或者看起来是否和我写的一模一样。

是的，知道如何写出完美的代码是需要学习的重要技能，但不需要成为每个练习的教学点。

### 2。在挑战不足和挑战过度之间游走

如果你的工作场所有一个导师或学徒计划，你可能会为你的学员安排练习。为了达到最佳的学习效果，你设置的每个练习都应该比前一个稍微有挑战性。

实际上，这意味着你指导的每个人都应该得到一份适合他们的工作计划。教学大纲是有帮助的，但它应该包含足够的变化机会，你可以确保每个学员都得到适当的照顾。离开滑雪道应该成为常态，而不是例外。

我通过思考我职业生涯中编写的所有算法、程序和库来做到这一点，并确定哪些对我的学员有用。作为导师，你应该确保你有大量不同的练习可供选择，并且一直在寻找更多的练习。

### 3。评估技能至关重要

当开发人员处于职业生涯的最开始时，他们很少了解他们在某一段时间内能够实现什么。然而，在某种程度上，他们会想要感觉到对自己技能的控制。他们想确定他们能在一定的时间内交付软件。为了获得这种控制感，他们需要知道如何评估他们的工作。

如果做错了，开发人员可能会承担过多的工作，并持续超负荷工作。过度工作是一个非常容易陷入的陷阱。初级开发人员学会以可持续的速度工作是很重要的。良好的评估技能可以做到这一点。

### 4。学会倾听

作为导师，我们最重要的技能是倾听的能力。编写代码是我们工作中最简单的部分，也很容易教。倾听他人，了解他们的需求，想出如何帮助他们...嗯，那很难。当我和我的学员在一起时，我的注意力几乎总是集中在学员的情绪上。学习破译他们在说什么，并弄清楚他们真正在说什么。他们有活力吗？他们在挣扎吗？他们快乐、沮丧、焦虑、疲惫吗？他们现在需要什么来帮助他们成功？

我相信我对导师的想法会继续发展，我也很想听听你的想法，所以请保持联系！
