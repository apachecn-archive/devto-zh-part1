# 开源:契约的另一面

> 原文:[https://dev . to/captainsafia/open-source-the other-side-the-contract-49c](https://dev.to/captainsafia/open-source-the-other-side-of-the-contract-49c)

我在推特上发布了很多关于开源维护者和贡献者可以采取的行动，以使他们的开源项目更加包容、友好和健康。

但是维护开源项目的健康和包容性并不是贡献者和维护者的任务——它也落在用户身上。

我觉得有时候关于用户在维护一个健康的开源生态系统中的责任的对话是不存在的或者是消极的。维护人员经常对用户不恰当的要求或没有意识到管理一个开源项目的困难感到沮丧(理所当然)。用户经常对他们认为的开发人员的不作为感到沮丧(理所当然)。

这是一个典型的人的问题，根源在于愿景的错位和用户与构建者之间意图的误解。

在这篇文章中，我想分享一些开源**用户**在积极参与开源项目时需要知道的事情。

开源是关于授权的。如果你曾经在一个开源项目上要求一个特性，但被维护者告知由于 X 原因它不会被包含在内——不要生气/心烦/小气。维护者在项目上比你有更多的技术专长，可能有更好的理由拒绝特性请求。如果你真的生气了，利用这种愤怒来有效地解决问题。开源允许您创建自己的 fork 或 pull 请求来讨论您希望修复的问题。当有机会利用这种力量时(比如当你的功能请求被拒绝时)，抓住它！用它来成长为一名开发人员、设计师和人类。在任何情况下都不要用它来贬低项目维护者/贡献者。他们是人，有自己的梦想、愿景和执念——随他们去吧。

在导航开源生态系统的过程中，您可能会遇到不恰当地怀有敌意的维护者和贡献者。在这种情况下，你能做的最好的事情就是退出。你是否在问题跟踪器或留言板上问了一个问题，却得到了不友好的回应？离开对话，到其他地方寻求支持。您是否收到了对您打开的拉取请求的敌意回应？关闭您的拉动式请求，他们不值得您付出努力。开源涉及到很多说“不”的问题，但是说不有正确的方式，也有错误的方式。不要和不知道正确说不的人交往。

如果你对你使用的项目特别有热情，但是没有技术技能来做贡献，那就代表这个项目去传道。维护者和贡献者可能太忙于开发软件而没有时间在会议上发言——但是你可以。

你也可以把你的钱放在你所说的地方。一些项目讨论通过 PayPal、Patreon、OpenCollective 或其他平台进行捐赠。我当然不是第一个这么说的人——但是如果你关心一个工具，就给出你认为适合它的东西。

开源是一条双行道。最好的开源项目有善良热情的维护者/贡献者和理解/耐心的用户。无论你是维护者、贡献者、用户，还是以上所有人——记住你所做的一切都是对开源生态系统的贡献，所以要确保它是积极的。
