# 变得愚蠢

> 原文：<https://dev.to/hmemcpy/becoming-foolish-8l6>

*这篇文章最初发表在[我的博客](https://hmemcpy.com/2017/10/becoming-foolish/)上。*

安迪·亨特和迪夫·托马斯所著的《实用程序员:从熟练工到大师 T3》一书建议，作为开发人员，我们应该“每年至少学习一门新语言”(第。14)

当我最近问一屋子的开发人员，今年是否有人学了一门新语言时，只有很少的人举手。一年前的今天，我会坐在观众席上，手保持向下。

十多年来，我一直是一名快乐的 C#开发人员，在此之前的某个时候，我开始了职业生涯。NET 2.0 发布了。在我的职业生涯中，我有幸在一个非常专业的领域工作——为其他开发人员构建工具，主要是以 Visual Studio 本身的扩展形式。我的工作经常要求我比编译器更好地理解代码，事实上，我经常不得不看一眼引擎盖下的东西来理解编译器、MSIL、JIT，有时还有本地代码在做什么，所有这些都远在像 Roslyn T1 这样的工具出现之前。我知道任何地方都没有记录的运行时的实现细节。我知道大多数的极限情况和语言陷阱。但总的来说，我的语言 C#是一把非常强大的锤子，能够处理任何种类的钉子。

这让我几乎忽略了其他任何与我日常工作无关的语言。每当 F#阵营的人吹嘘他们缺少空值、模式匹配或者不需要分号或大括号时，我通常会耸耸肩。在我的职业生涯中，我花了数年时间学习使用 ReSharper 和 IntelliJ IDEA 等生产力工具，以至于只需一次按键就能搞定所有的样板文件。我根本不在乎！拥有一些好的特性从来都不是我转换编程语言的足够好的理由——当我所精通的语言也可以做到这一点时，我为什么还要费心呢？为什么不同的*语法*会对我写软件的方式产生影响？毕竟，这一切不都是为了同一件事吗？

因为当你真正开始思考的时候，(几乎)所有的编程语言本质上都是这样的:

```
public static void main(String[] args)
{
    ...
} 
```

在我们经历过的大多数编程语言中，我们所知道的世界*在`main`函数的开括号和闭括号的范围内生存和死亡*，这是我们应用程序的入口点。我们程序中发生的一切，我们程序与之交互的宇宙，都发生在这两个括号之间。根据它的定义，我们的程序返回一个*退出代码*，指定它是否成功，其他的一切都是副作用。

这意味着，所有的工作都是在`main`方法的两个括号之间完成的，这就是驱动**我们所知道的关于这个模型的一切**——每一个设计模式、每一个实践、每一个规程、每一个工具、库或框架——都是为了让我们管理这个存在于这两个括号之间的世界模型而创建的。我们有几十年的知识，知道如何在几乎任何语言中正确地做到这一点。

对我来说，这是核心，是所有软件构建的基础！不管使用何种编程语言，这都是终极真理。

就这样，直到**哈斯克尔毁了我的一切**！

我就不赘述了——我在之前的[帖子](https://dev.to/2017/05/from-net-to-scala-and-beyond-a-journey-to-functional-programming/)中无意中做到了这一点。但是在观看一个看似不相关的主题的演讲时，演讲者设法在里面偷偷放了一些 Haskell！直到那时，我还没有真正看过任何 Haskell 代码——我总是认为它无关紧要——它就在那里，直直地盯着我。

```
main :: IO () -- reads as 'main, having the return type of IO ()' 
```

我看到的解释是，Haskell 实际上并不在其`main`函数内部做任何工作。相反，程序以一种声明的方式“描述”当你的程序执行时将要发生的事情，这个“描述”将被返回到 Haskell 运行时，所描述的事情将会发生——没有可能的*副作用*或意外。它颠倒了整个执行的想法——工作，效果的实际执行不是在`main`的内*完成的——而是在*外的*完成的！*

第一次看到这一点，试图让我接受它，在我的头脑中产生了一个 *bug* 。这与我所知道的相冲突——所有的语言都是一样的，区别仅仅在于语法！但是在这里，我看到了一些不同的工作方式，不符合我的世界观！我努力去理解它，又看了几次，直到有一天，它终于明白了！

在某一瞬间，我突然明白了 Haskell 的想法，明白了它为什么会这样。一切都很顺利，我终于“理解了单子”！对我来说，这感觉就像尼奥终于接受了[没有勺子](https://www.youtube.com/watch?v=uAXtO5dMqEI)的事实，只有这样他才能弯曲它。

我感谢安德里亚·马格诺斯基的精彩主题演讲，[邀请所有人参加派对](https://www.youtube.com/watch?v=WBu43Tj0zOY)，这让我意识到，我所经历的不仅仅有一个名字，事实上，它是这一现象的教科书定义:我所经历的被称为**范式转变**。

托马斯·库恩(Thomas Kuhn)在他 1962 年的著作《科学革命的结构》(The Structure of Scientific Revolutions)中创造的术语“范式转变”(paradigm shift)描述了一个过程，如果一个既定的模式不再符合世界观，它就会被拒绝，同时被另一个模式取代。库恩说:

> 拒绝一种范式的决定总是同时也是接受另一种范式的决定，而导致这一决定的判断涉及到两种范式与自然的比较以及它们之间的比较。托马斯·库恩，《科学革命的结构:50 周年纪念版》(第。78)

我有两个相互冲突的想法，当最后另一个想法开始有意义时，我的大脑接受了它，并立即切换到那个新的想法-切换是瞬间的！我能够从不同的角度看问题，理解一个全新的世界模型，这正是库恩对“范式转变”的定义。

但是发生了一些别的事情——我意识到我对软件的了解是多么的少，有一整个世界对我来说是未知的！直到那一刻我所知道的，我在一个狭窄领域的所有专业知识只是冰山一角。这种认识消除了我的一个障碍，它阻止我尝试学习新语言和尝试新事物。以前我会认为这些事情无关紧要，而现在我突然对以前从未感兴趣的事情产生了兴趣。仅去年一年，我就使用了 Haskell、Clojure 和 Rust。我看过 Erlang，Elm 和 PureScript。我正在通过 [SICP](https://mitpress.mit.edu/sicp/full-text/book/book.html) 努力学习[范畴理论](https://github.com/hmemcpy/milewski-ctfp-pdf)。我这么做是因为这么长时间以来，我第一次真正好奇这些语言/范式能*教给我什么！我真的不记得上次我对学习如此兴奋是什么时候了。真诚地为自己能够说“我不是什么都知道”而自豪令人难以置信的解放！*

Bret Victor 精彩且极具娱乐性的主题演讲“编程的未来”也呼应了这一观点，他在 1973 年假装自己是一名软件开发人员，并对未来 40 年的编程进行了预测。那个演讲的最后 5 分钟有一些我听过的最好的建议，我希望我能早点明白。他说，

> ...你必须对自己说，“我不知道我在做什么。我们，作为一个领域，不知道自己在做什么。我们不知道编程是什么，我们不知道计算是什么，我们甚至不知道计算机是什么。”一旦你真正理解了这一点，一旦你真正相信了这一点，那么你就自由了，你可以想任何事情。
> **Bret Victor** - [编程的未来](https://vimeo.com/71278954)

*保持饥饿。保持愚蠢。*自由自在！
