# 软件开发中有缺陷的一致性目标

> 原文:[https://dev . to/Ben/the-defected-goal-of-uniformity-in-software-development](https://dev.to/ben/the-flawed-goal-of-uniformity-in-software-development)

有时候，我会完成令人难以置信的大量高效编码工作。有些日子甚至不近。我猜想八二法则非常适用于我自己以及我所观察到的开发人员和组织的生产力。尽管如此，我们的商业模式期望一些统一输入的概念，这与软件开发等创造性工作的本质相抵触。我们人为地强制实行源于流水线作业的平均工作日。我们分配同质的冲刺周期，似乎没有考虑到在一个人的心流中自然产生的根本不一致，不允许个人问题或动力的冲击有回旋的余地。

当我们被期望产生一致的输出时，它可能在光谱的两端都是消极的。当事情不顺利时，工人们被迫坐在座位上，出于对一致性的期望，他们不太可能在深夜进行有灵感的跑步，因为担心建立新的规范。

虽然这些程序有一些历史目的，但我觉得它们是低效和不合理的。我们仍然处于前钱球时代，甚至“创新”技术部门似乎也对“破坏”公认的最佳实践不感兴趣。

我几乎没有结构化管理的经验，所以我天生天真，但我确信，不断变化的行业不会根据手头的问题来设计工作程序，这对它们不利。当变化可能是革命性的时，它们往往是渐进的。当然，有一些权衡使得这些革命者不切实际。做任何与众不同的事情都会招致别人的嘲笑，否则你就不会遭受这种嘲笑。一个组织承担以不同方式做事的风险。因此，虽然目前的情况是低效的，但任何超出正常范围的事情都可能发生在这个范围内的任何地方。如果你只是做其他人都做的事情，那么让新员工跟上新实践的速度也是有问题的。这就是为什么所有 32 支球队使用相同的踢法。

在一个不断发展的行业(如软件开发)的从业者要求经验丰富的高薪的环境中，以吸引特定类型的熟练从业者的方式区分组织的过程是值得的。我相信这对业务成果和开发人员的幸福都有好处。从长远来看，这些将齐头并进。尽管存在潜在的权衡，但比我们更努力地思考“我们一直做的方式”是值得的。
