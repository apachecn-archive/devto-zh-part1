# 源代码管理失败

> 原文:[https://dev.to/funkysi1701/source-control-fail-dgn](https://dev.to/funkysi1701/source-control-fail-dgn)

每个开发人员都使用源代码控制，这是一个很好的工具，可以跟踪代码的变化，看看谁做了什么。

不管我怎么搞砸，我对它的使用。我在创建新特性时相当自律，我的所有更改都将提交到源代码控制中，当我高兴时，我会将这些更改部署到活动系统中。但是一旦有 bug，特别是客户在寻找补丁的时候，我会尽可能快地将补丁部署到实时系统上。

乍一看，我所描述的并没有错，但是下次我部署一个新特性时会发生什么呢？是的，客户抱怨的 bug 在新特性中得到了部署，因为补丁从未提交到源代码控制中。客户很生气，因为他尖叫的那个 bug 又回来了。

您对系统**所做的每一个更改都必须**提交到源代码控制中。如果不是这样，变化看起来就像从未存在过。我在源代码控制领域工作了 5 年多，为什么我总是一次又一次地犯这个新手的错误。

Troy Hunt 有一篇关于使用源代码控制的 10 条戒律的博客文章。他的第二条戒律是“如果它不在源代码控制中，它就不存在”他更多地谈到你写的代码没有保存到源代码控制中，但是原则对于我的例子和他的例子是一样的。

有很多方法可以从源代码控制中自动部署，但是大多数情况下，您不希望因为修复了一个打字错误而重新构建您的实时数据库。将需要实现额外的步骤，并且您仍然有可能想要绕过这些步骤来修复紧急问题。解决这个问题的唯一方法是，您和您团队中的每个人都要遵守纪律，并且只首先提交源代码控制，并且只在此之后进行现场部署(无论是自动的还是手动的)
