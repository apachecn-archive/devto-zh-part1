# 时间估计的权衡

> 原文:[https://dev.to/pbeekums/tradeoffs-of-time-estimates](https://dev.to/pbeekums/tradeoffs-of-time-estimates)

我喜欢构建软件时的时间估计。这不是一个流行的观点，但是时间估计有一些有价值的用途，只要它们不被滥用。

时间是任何人或任何公司拥有的最宝贵的资源。我喜欢确保我的时间用得其所。我希望我的时间投资能获得最大的回报。在软件中，这意味着致力于为所花时间提供最大价值的特性。

要计算功能提供的确切金额是不可能的，所以让我们使用一种称为效用的任意价值度量。

特征 A 将提供 10 的效用。

功能 B 将提供 20 的效用。

因此，功能 B 将提供更多的价值，这意味着我们应该首先处理它，对吗？

如果特性 B 要花 2 周时间，特性 A 要花 2 小时，那就不会了。计算任何东西的收益都需要分子和分母。效用是我们的分子。时间是我们的分母。当考虑到时间时，我们意识到，考虑到我们需要投入多少时间，特性 A 更有价值。

在期限合理的情况下，时间估计也很有用。会议不会等待软件发布。我开发了一个与电视节目发布相关的软件。电视节目和电影不会坐等软件发布。如果你需要与税收相关的软件，国税局肯定不会等着软件发布。

评估在这里很有用，因为盲目的优先级列表并不总是反映正确的值。假设特性 A、B 和 C 被认为是最重要的特性。这意味着它们将在特性 d 之前被处理。但是，如果特性 A、B 和 C 的价值只有在它们都完成时才能实现，那该怎么办呢？如果它们单独使用几乎毫无用处呢？时间估计会让你知道在截止日期前及时完成这三件事是否现实。如果不是，那么这三个都需要被砍掉。

那么为什么开发者讨厌时间估计呢？不管怎样，许多开发人员会自己估算时间。几分钟就能修改一个错别字。几个小时或几天来修复一个坏的数据库查询。仅仅传达这些内部评估有什么问题？

问题在于处理时间估计的方式。它们被称为“有原因的估计”。它们不被称为“精确的”是有原因的。虽然开发人员知道一项任务需要多长时间，但他们很可能是错的。迈克尔·沃尔夫的这篇[文章是我见过的关于为什么的最好的类比。](https://www.quora.com/Why-are-software-development-task-estimations-regularly-off-by-a-factor-of-2-3/answer/Michael-Wolfe)

如果你在看一张美国地图，在旧金山和洛杉矶之间画一条线，它的距离将是 x。但是这条线是在一个高层次上，不会考虑增加距离的实际路径。地图越详细，揭示的曲折就越多。每增加一层细节都会增加距离。只有当你真正踏上旅程时，你才能得到所需距离的精确测量值。

软件开发也是如此。完美地确定一件事要花多长时间的唯一方法是实际建造它。在这一点上，你不再创建一个时间估计。

开发人员总是会得到错误的准确时间估计，但是他们通常会得到正确的工作量。需要几分钟解决的事情可能需要 3 分钟或 10 分钟，但不会需要一天。需要几天的事情不会需要几个月。当然也有例外，但这应该是普遍适用的。幸运的是，有效的项目管理只需要这么大的规模就可以完成。获得时间投资的大致回报比盲目工作要好。

在项目管理之外使用时间估计是事情开始变糟的地方。我见过许多喜欢将评估的准确性与绩效挂钩的经理。在这一点上，开发人员不再仅仅考虑对项目有什么好处。这是他们开始根据如何获得回报来做决定的时候。

当与绩效挂钩时，时间估计会产生一些不正当的动机。工作质量变得没有花费的时间重要。通常，团队会讨论这种权衡，以决定是质量更重要还是时间更重要。当时间估计与绩效挂钩时，这种对话就不太可能出现。这个决定不会被带到团队中来。

最终结果是时间估计将开始变得不那么有价值。缺乏沟通使得项目难以管理。这降低了从时间估计中获得的好处。度量是评估开发人员的一个糟糕的方法，所以无论如何，当试图评估性能时，没有获得任何价值。不管这样做的诱惑有多大，时间估计都不应该用于绩效评估。
