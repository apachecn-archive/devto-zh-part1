# 我为什么喜欢 Maven

> 原文:[https://dev.to/grahamcox82/why-i-like-maven](https://dev.to/grahamcox82/why-i-like-maven)

我刚开始用 Java 的时候，Maven 还不存在。我指的是 Maven 1.0 Beta 2——Apache Maven 发布历史页面上最早的条目——还没有发布。Ant 1.1 已经在几个月前发布了，但我也从未使用过它。当时，它是用手或者 Make 来执行命令的。

快进到我第一次在 Uni 之外使用 Java 时，Maven 2.0.4 已经存在，我参与的第一个项目就使用了它。这真的很痛苦。您必须编写大量复杂的 XML，并在 Maven 存储库中定位您的依赖项——或者使用正确的元数据文件手动将它们上传到您自己的 web 服务器，以及各种复杂的东西，只是为了构建一些东西。你甚至不能自由地把你的文件放在你想放的地方——它决定了它们应该放在哪里。这与我以前习惯的任何事情都截然不同，这很糟糕。但我坚持下来了，我渐渐接受了它，并习惯了它的特质。

再往前，我们现在有了过多的选择。Maven 仍在那里，变得越来越强大，但我们已经有了许多改进版本的 Ant，以及 Gradle、SBT、Leiningen，可能还有其他人。然而，这些天我个人更喜欢 Maven。

这有两个原因:

最明显的是它们的配置方式。给 Ant 一个要执行的指令列表。给了格拉德、SBT、莱宁根等用实际代码编写的脚本来配置构建系统。Maven 得到了一个描述构建的简单文档。XML 没有什么神奇之处。它没有做任何聪明的事情。它没有内置的编码支持来根据自己的目的扭曲它。只是一份文件。因此，所有 Maven 配置的工作方式都是一样的。你可以获取任何人的 pom.xml 文件，阅读它并知道它要做什么。您不必考虑这个聪明的 Groovy 或 Scala 会对最终结果产生什么影响。

然而，这对我来说不是最大的。最大的是成熟。Maven 现在是每个人都必须遵守的。因此，如果您想做一些不寻常的事情，Maven 可能会以更少的努力和更多的例子来支持它。当然，您可以让其他人做任何事情，因为他们是可脚本化的。你可以用它们做任何事情。但有时要实现它需要付出巨大的努力。

举个例子，我经常用这个例子，因为它太荒谬了。当我第一次使用 Gradle 时，没有对“provided”范围的支持。没有简单的方法来构建一个编译时依赖于 Jar 文件的应用程序，它不应该与 Jar 文件捆绑在一起——比如 Servlets API。有很多方法可以实现这一点，通过在构建脚本中编写十几行 Groovy 代码，使用文档记录很少的特性，而 API 在每个版本中都会发生变化，但是这样做既痛苦又脆弱。

可能性是，我最终将不得不搬到格雷尔，我会习惯它，它会很好。如今，很多项目都采用 Gradle 而不是 Maven，因为公平地说，它在使简单的构建变得非常简单方面做得非常好。但我永远会给梅文留一个位置。

1)有一点需要注意的是，这里有太多的选项，几乎可以肯定人们会使用不同的选项子集。
