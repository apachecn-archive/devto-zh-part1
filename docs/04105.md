冒险和犯错是通往伟大的学习之路的一部分。快速而频繁的失败是成功的标志。然而，避免犯同样的错误不止一次应该是你优先考虑的事情。

亲身经历是最有力的学习方式。但是，从别人的经历中获得的知识可以让你走上成功的捷径，并省去巨大的痛苦。特别是，意识到其他人犯的常见错误可以防止陷入可避免的陷阱。

让我举一个类似的例子。我对木工很感兴趣，当我买了台锯后，我阅读了所有我能找到的关于台锯事故以及如何预防的资料。表锯错误可以花费你的手指，你的手，甚至你的生命。他人的不良经历可以防止严重伤害。在你学会如何使用新工具之前，你不必遭受失去一条肢体的痛苦。

当软件工程师犯下大错时，他们不会受到身体上的伤害。然而，他们可能会错过多年的职业发展机会，因为他们没有意识到自己做错了什么。软件开发中的许多错误并没有直接转化为明显的学习机会。它们会减缓人们的职业发展，让他们无法发挥最大潜力。

# [](#9-mistakes-to-avoid-at-all-costs)不惜一切代价避免的 9 个错误。

作为一名软件工程师，如果你是这样的话，你就犯了一个错误:

# [](#1-not-testing-your-work)1 -没有测试你的工作。

当你写代码时，你必须测试它。在我继续之前，让我给测试相关的一些术语下定义；它将对本主题的其余部分有所帮助。

> ### [T2】测试相关定义](#testing-related-definitions)
> 
> *   **[Happy Road]** : a series of operations, data flows and user actions, which execute the business logic of the code according to the design.
> *   **[unpleasant path]** : by
>     *   Any deviation from the pleasant path caused by factors such as errors.
>     *   Unexpected input.
>     *   Or the system is poorly configured.
>     *   Missing or exhausted resources (memory, disk space, network, etc ...).
>     *   Hardware failure.
>     *   Unexpected scale/concurrency conditions of.
> *   **Formal test:** -A detailed test completed by someone who has not written the code. It may be testers, QA engineers, other developers or suppliers. Formal testing considers many things, including specifications (if any), documents written by programmers, and knowledge of users and business environment.
> *   **Development Test** : This is the test that developers prepare for the formal test. The development must include:
>     *   **Manual test** . Apply business logic like an end user.
>     *   **Unit test** . Create repeatable tests for code units.
>     *   **Automatic test.** End-to-end testing the code of the system or part of the system.

记住这一点，仅仅因为你的代码编译没有错误，或者仅仅因为快乐路径似乎工作，这并不意味着工作已经准备好进行正式测试。

### [](#to-avoid-this-blunder-you-must)为了避免这种失策，你必须:

1.  为所有快乐的路径执行开发测试。
2.  对你能想到的所有容易测试的不愉快路径进行开发测试。
3.  记录下所有你能想到的不容易测试的不愉快路径。与正式的测试团队一起评估和讨论这些，以决定谁应该去做以及如何去做。

# [](#2-hiding-your-mistakes)2 -隐藏自己的错误。

在大多数情况下，隐藏错误比你试图隐藏的错误更糟糕。你应该始终保持透明，知道哪里出了问题，它是如何发生的，你正在做什么来修复它，以及你将做什么来防止将来发生类似的事情。

我从未见过有人因为犯了并承认的错误而被解雇。然而，我见过有人因为试图隐瞒错误而被解雇。隐瞒一个错误会毁掉你的信誉；夺回它会非常困难。

为了避免这种错误，当你犯错误的时候大声点。让可能受影响的人知道并立即公开承认全部责任。你应该感受到你隐藏的东西的重量，在它压垮你之前，你需要释放它。

# [](#3-feeling-a-strong-sense-of-individual-code-ownership)3 -感受强烈的个人代码拥有感。

我在避免个人代码所有权的 12 个理由中写过这个话题。有时候，个人代码所有权是您工作的组织的工程文化的一部分。其他时候，这是你因为自我或骄傲而犯下的错误。能避免就避免。如果你不这样做，它会回来困扰你。

为了避免这个陷阱，不要保护你的代码。把它看作是你建造的供别人行走的人行道。

# [](#4-documenting-the-code-for-yourself-and-not-for-others)4 -为自己而不是为别人记录代码。

当你记录你的代码时，你应该为那些几个月甚至几年后从事这项工作的开发人员做。由于您不拥有代码，文档应该有针对性地向其他人解释您做了什么以及为什么做。

如果你的文档只有你自己能理解，那么其他人就不能有效地处理代码，而你将成为事实上的所有者。这将你引向一条个人代码所有权的道路，结果陷入了一个自制的职业监狱。

此外，糟糕的文档是糟糕代码最明显的标志之一。可能很难判断一段代码有多糟糕，但是很容易判断它的文档有多糟糕。当你的工作中到处都是糟糕的文档时，你就会和糟糕的代码联系在一起。

匆忙构建的原型系统——尤其是在创业的早期阶段——通常都有糟糕的文档。在这种环境下编写的代码并不意味着可以持续很长时间。它通常被写为概念的证明。不幸的是，原型代码往往会无限期地存在。出于这个原因，我建议编写所有的代码，就好像它会被许多开发人员年复一年地接触一样。大多数时候是这样的。

为了避免这种错误，在编写所有代码注释时，要记住这样一个形象:一个开发人员几年后会试图修改它。给开发者一张脸和一个名字。我的名字叫博尔博，他有一头黑发，方脸，戴着眼镜，有一种玩世不恭的幽默感。我甚至画了它。别问了。

# [](#5-fighting-the-development-process)5 -战斗发展过程。

像 Scrum 和看板这样的开发过程被设计成允许团队扩展。它们促进了项目的协作和共享所有权，确保完成了足够的计划，并且计划的执行得到了协调。

对抗正式过程的开发人员会制造摩擦，这会消耗能量并减缓团队的可伸缩性。这是一个有问题的趋势，倾向于将反对者降级为单独的倡议。这种孤立对于软件工程师来说可能是职业杀手。他或她会被认为没有能力在团队中合作和工作。

不同的组织选择采用不同的正式流程。如果你对这件事有强烈的意见，帮助你做出选择并把选择的过程塑造成可行的东西是很重要的。如果您因为喜欢牛仔风格的编码方法而反对它，您可能会考虑一个没有团队可伸缩性愿望的较小的组织。

为了避免这个错误，将开发过程的需求和优势内在化。如果有些事情对你不起作用，帮助解决它。不要指望去牛仔式的无流程环境。

# 6——做了 90%的工作，还没做完就转做其他事情。

如果你雇了一个油漆工来粉刷你房子的内墙，你会期待一份完工的工作。在你签支票的时候，墙壁应该已经粉刷好了，房间已经打扫干净，可以使用了。如果你得到的是一个装饰上仍有蓝色胶带的房间，你不会满意的。

已经完成 90%的代码还没有完成。一些开发人员喜欢原型概念的证明，他们很擅长这个。然而，让“原型制作”成为一种习惯是一条不归路。当涉及到将一个可交付产品带入可交付状态时，他们经常不能完成重要的收尾工作和润色工作。

未完成的工作不能发布。如果做了大部分工作的人没有完成，其他人需要完成。这种行为转化为团队什么也做不了。

解决办法总是完成你的工作。不要等着别人从你停下的地方继续。把它看作是帮助一个蹒跚学步的孩子穿过繁忙的街道。你不想让孩子在没有牵着你的手的情况下穿过最危险的部分，对吗？

# [](#7-not-checkingin-your-work)7 -没有签到你的工作。

一些开发人员没有签入他们正在工作的代码的习惯。这可能看起来很奇怪，但在以下情况下尤其常见:

*   实验或早期项目。
*   构建脚本。
*   基础设施代码。
*   测试代码。
*   配置。
*   部署脚本。
*   原本要运行一次的脚本。

所有代码都必须尽可能快、尽可能频繁地签入。如果您使用 GIT，分支会使它变得轻而易举。您应该抱着未签入的文件不存在的心态工作。

# [](#8-not-understanding-the-value-of-estimations)8 -不了解估计值。

软件组织受制于经济法则。我在一篇名为[软件开发的经济](https://www.coderhood.com/software-development-economy/)的文章中写过这个话题。特别是，组织需要能够决定他们是否应该投资于产品、功能或增强的开发。为了做出决定，他们需要对成本和结果的价值进行估计。没有这样的估计，就不可能进行这样的评估，决策最终只能在真空中做出。

开发人员有时会认为不可能估计完成一个项目所需的工作量。我承认这可能很难，但是随着实践和经验的积累，估计会变得越来越准确。与不愿意提高评估技能的开发人员相比，愿意提高评估技能的开发人员在职业生涯中走得更远。

避免这一错误的解决方案是将评估的商业价值内在化，并将其作为职业生涯的优先事项，从而变得更好。

# [](#9-not-sharing-knowledge-with-others-for-fear-of-becoming-unimportant-or-irrelevant)9 -不与他人分享知识，因为害怕变得不重要或无关紧要。

我在一篇名为[职业建议:让自己失业](https://www.coderhood.com/career-advice-work-yourself-out-of-your-job/)的文章和[避免个人代码所有权的 12 个理由](https://www.coderhood.com/12-reasons-avoid-individual-code-ownership/)中讨论了这个话题的各个方面。

把知识藏在身边，不让你的队友知道是一个非常有害的习惯。谬论是成为知识的守护者提供了工作保障；事实上，这让你很难获得新的责任。

晋升通常与更大的影响范围相关联。如果你把知识隐藏起来，你会被认为是一个不能授权和影响他人或在团队中工作的人。你将被视为一个工蜂，拥有一个关键的位置，但不能移动到其他事情或扩展到更高级的角色。一旦你在公司的任期开始与你带来的价值不一致，你会注意到别人对你的看法发生了变化。你掌握的知识最终会过时，更有活力的工程师最终会重建你一直保护的东西，你将不再像过去那样被视为有价值。

解决方法简单但不容易。分享你所知道的。你可以通过写文档、培训他人、委派一些工作、主持午宴等方式做到这一点。试着不要长时间持有开启知识宝库的钥匙。
