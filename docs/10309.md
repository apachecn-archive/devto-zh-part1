# 头疼的服务:在接受服务依赖之前要考虑的事情

> 原文:[https://dev . to/Ben/heather-as-a-service-在采用外部软件服务之前要考虑的事情](https://dev.to/ben/headache-as-a-service-things-to-consider-before-taking-on-external-software-services)

当你能找到一个很棒的“软件即服务”(SAAS)产品来做你需要做的事情，这是一件好事。这些产品通常定价合理，因此与其花费开发人员的时间，不如付费购买。如果是一个单独的项目，这些服务通常有小规模的免费层。

然而，成本总是存在的，而且经常会发生。作为一些便利的交换，您的代码现在有了一些依赖项，这些依赖项很难在您的测试环境中隐藏，并使您的代码库更难跨开发环境进行管理。根据我的经验，当试图让新人加入项目并让他们准备好开始编写代码时，外部服务会带来一些最大的挫折。就软件设计而言，您还面临着服务无法为关键用例提供支持的风险，而您可以轻松地扩展自己的代码来实现这一点。你不能就这样放弃你订阅的专有服务。

当您可以选择依赖服务和外部库时，软件开发在很大程度上变成了风险管理。服务的大部分价值是公司承担的基础设施、安全、支持或其他任务。他们承担的每一项任务都可能降低你的一些个人风险，增加其他风险。

总而言之，通常更可取的做法是采用合理数量的自托管依赖项，并且只在您获得明确的特殊价值时才添加服务到您的代码库中，这种价值本身并不适合自托管。也就是说，接受提升你的产品的创新服务，或者提供基本基础设施的服务，而不仅仅是云中基于便利的软件 API。从长远来看，这些往往会带来不便。

如果你接受外部服务，马上写好包装器。包装器提供了一个接口，它允许你维护内部命名约定，交换不同的服务，创建抽象，维护未来的可扩展性，以及合理地测试你的代码。

我有一套特殊的经历，所以我的想法可能不会放之四海而皆准。如果你不同意或有任何附加建议，让我知道！
