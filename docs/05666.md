# 技术问题首先是人的问题

> 原文:[https://dev . to/pbeekums/technology-problems-is-people-problems-first](https://dev.to/pbeekums/technology-problems-are-people-problems-first)

我是一名软件开发人员。当我看到一个技术问题时，我首先想到的总是技术解决方案。这是显而易见的事情。

[![](../Images/167bc44f5fa388e805f1b019d9c6020a.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--1XO1IgG6--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://blog.professorbeekums.com/img/2017/8/messycables.jpg)

然而，要很好地解决一个问题，通常需要首先了解问题的起因。“以前的开发者太差了，我太爱打球了，这是我年轻时经常得到的答案。那个具体的答案往往是不正确的。然而，它确实提出了一个观点，即技术问题不是突然出现的。它们是由人创造的。任何技术问题的根本原因都是人的问题，而这些是最难解决的问题。

最明显的人的问题是技能。没有构建软件的专业知识会导致很多问题。我参加过很多团队，他们主要是后端开发人员，在前端代码中混日子。但是我们应该做些什么呢？直到我们雇了专家来教我们才停止工作？我从没见过这种雇佣时间少于 6 个月的。等待的时间太长了。

数据库也是技能问题的另一个很好的例子。在产品推出之前，任何数据库都适用于任何功能。大规模是指事物开始迅速分崩离析。如果团队中没有人处理过您的特定大规模数据库，您会怎么做？你是否会停止开发软件，直到你能雇佣到非常困难和昂贵的人？

混乱的前端代码和崩溃的数据库是令人讨厌的技术问题，但它们的根本原因是人的问题。有专业人士吗？如果没有，团队中的某个人如何获得专业知识，需要多长时间？没有专业知识前进有什么坏处，不前进有什么坏处？团队是否意识到他们遗漏了一项重要的技能？

有时候技能是有的，但问题在于态度和决策。比如有些开发者真的很讨厌改变。他们坚持可能曾经是标准的做法。他们停留在那些熟悉的实践的舒适区。我认识一个开发人员，他通过纯粹的意志力保持整个工程组织使用 SVN。SVN 曾经是大多数开发人员存储代码的方式。

今天，可能有很大一部分开发人员从未听说过它。那是因为 GIT 要好得多。整合多个开发人员的工作现在只需要几分钟，而不是几个小时。SVN 是次等技术产品，但我们的问题不在于技术。那就是一个人正在阻止进步的发生。

这是一个明显的例子，但还有很多微妙的例子。我个人对在生产环境中使用 Docker 持怀疑态度。我知道有人谈论把它放在适当的位置，但我不知道有谁已经衡量它。如果出现问题，我担心是否能够解决。

我缺乏技术方面的专业知识，但我也没有花太多时间来获得这方面的专业知识，因为我不确定这是否值得我花时间。我相信热爱 SVN 的开发者对 GIT 也有同样的想法。

我是不是又老又暴躁？我的谨慎是正确的吗？只有时间能证明一切。但我可能会也可能不会与人相处，我不知道事情会以哪种方式结束。我所能做的就是保持一个开放的心态，如果 Docker 成为一个明显更好的解决方案，那么我怀疑 Docker 和跟风可能是错误的。

当然，也有一些开发人员有相反的问题，他们总是抓住最新最闪亮的东西不放。他们在技术真正准备好被广泛使用之前就采用了它。如果技术有很多缺陷，这可能会导致问题。如果事实证明新技术从根本上来说是有缺陷的，并且用它来找出它的缺陷，这将是一个更大的问题。

还有很多其他例子可以说明人的问题导致了技术问题。一些开发人员喜欢过度设计解决方案，制造不必要的复杂性。有些人喜欢把东西拼凑在一起，并添加大量的技术债务。一些开发者喜欢自己构建一切。一些开发人员没有经过任何努力就选择了第三方解决方案。所有这些事情都有极端的例子，但它们也都有微妙的例子，很难知道开发人员站在哪一边。

最后一种人的问题与经理有关，它表现在很多方面。有些经理更关心的是当事情出错时不被指责，而不是确保事情顺利。有些经理喜欢把自己的领域当成自己的领地，而不是和公司的其他人一起工作。有些经理把更多的时间等同于完成更多的工作。

这些人的问题产生了各种各样的技术问题，但是我见过的最具破坏性的人的问题是那些不理解软件开发如何工作的经理。我看到这些发生，要么是因为经理已经忘记了作为一名开发人员是什么样子，要么是有人在作为开发人员打下坚实的基础之前就成为了经理。

经理的工作是确保他们的开发人员被激励去做好工作。如果不了解开发人员如何做好工作，就不可能产生好的激励。这导致许多实践在理论上听起来很好，但在实践中却分崩离析。

例如，在规划项目时，评估是必不可少的。一些经理认为应该奖励那些有更准确估计的开发人员。但是请注意“估计”这个词。不是“需要的确切时间”。

软件开发中经常会出现问题。开发人员很可能会在实现过程中想到更好的方法。您希望鼓励开发人员提出这个问题，以便团队可以决定它是否值得。如果经理将评估的准确性纳入绩效评估，激励措施就是确保评估的准确性。这意味着对新的可能性保持沉默，并专注于确保工作在预计的时间内完成。结果是更多的技术债务。

当面临技术问题的冲击时，有一种直接解决它们的诱惑。如果出现时间敏感的危机(例如，所有服务器都停机)，这是有意义的。当事情不那么紧急的时候，更有意义的做法是后退一步，思考潜在的原因。这些根本原因很少是技术性的。
