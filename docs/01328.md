这篇文章试图提炼我今年在 Weedmaps 努力成长自己和工程组织所学到的东西。有些可能是显而易见的，但花些时间反思和关注关键的学习内容是有好处的，因为学习最终是我们作为工程师的工作。

*   没有自动化的过程总是会被工程团队拒绝或忽略。如果你想改变过程，创造新的过程，自动化它，狗粮它，然后向人们展示如何和为什么它更好。

*   新工程师是获得外界对我们认为是佳能的过程和编程技术的看法的最好机会。倾听他们，用他们新鲜的眼光重新审视最佳实践。大约 90 天后，他们将会吸收公司文化，也就更难看出效率低下。

*   清晰是发布解决业务需求的有效代码的关键。敏捷并不意味着写代码不清晰。这种清晰应该在你执行的票据中表现出来。如果没有，你不应该做这项工作，直到你有清晰。没有目的的速度是无效的。

*   清晰从首席执行官开始，并通过团队向下聚焦。如果没有清晰明了的有效沟通，最终的代码将会偏离目标。然而，清晰并不属于领导层。如果有，提供给同行。在屋顶上大声喊出来，直到你的团队达成共识

*   清晰和意图在代码和团队动态中很重要。强类型系统非常清楚地传达意图。现代的强类型语言会让你成为一名更好的工程师，而且在大多数情况下，会减少交付生产的 bug 数量。Rust、C#、Swift、Kotlin 和 ReasonML 都是拥有优秀类型系统的现代语言的典范。如果你不能使用这些，那么使用螺栓型系统，它们也很棒(透析器，流式，打字)

*   异步系统更复杂，除非绝对必要，否则应该避免使用。一种常见的面向服务架构模式是使用异步消息队列。避免使用它们，直到你有充分的理由需要它们(ETL，事件源)。在遇到性能问题之前，最好使用同步通信范例(HTTP)。

*   你的公共 API 会被滥用、误用和误解。记录并交流您的公共接口，以确保它们被正确使用并且它们的效果被理解。这不仅限于 HTTP APIs，还包括库、模块和类。

*   面向对象编程 OOP 提供了像类和模块这样的结构，有助于你的代码模块化。使用它们！然而，要避免深度嵌套的继承(不止一个层次)，多重继承(混合，关注)。几乎在所有情况下都支持功能或对象组合。

*   在你的代码中要有防御性。始终假设您的依赖项会失败，并关注您的应用程序在这些场景中的行为。

*   专注于对你的团队的成功有倍增效应的工作。乘法工作包括使你的团队(和你未来的自己)更快、更有效的抽象、工具和文档。

*   如果没有一个有动力的人的支持，计划、工具和过程变化将会死亡。永远不要低估一个有动力的工程师改变整个公司的力量。坚持下去，直到你想看到的改变完成。

*   web 性能工程是一种思维模式，没有单一的灵丹妙药。要有一个高绩效的网站，需要反复的衡量和提炼。这项工作永远做不完，或者做得不够好。它需要适应环境的变化。

*   代码质量来自上游实践的组合。试图在评审时分析代码的质量通常是太少了，太晚了。使用上面提到的方法(API 设计、类和模块设计、强类型)在质量通过同行评审之前提高质量。

*   工程师应该了解你的代码是如何进入生产的。最终，将代码交付到生产环境是您的责任，并且您的工作直到它进入生产环境才算完成。使用工具、文档和人员来理解。了解这一点将会改变你编写代码的方式，并为改进代码提供见解。

*   自助服务部署(和回滚)、可靠的监控、流程编排、功能标志和可靠的弃用模型是面向服务的架构中的关键。没有他们，增加的复杂性将会压垮团队。
