# 反应——一年后

我喜欢学习，不断提高我的技能，就像我遇到的大多数事情一样，它们需要一段时间才能理解。我相信大多数人都一样。

我们接触到一个想法，在 Twitter 上或通过同事看到它，然后我们思考它，并将其与我们之前的想法进行比较。一些想法比其他的更快的被学习。

React 是我在脑海中和一些 GitHub 项目中思考了一段时间的想法之一，但在过去的一年左右的时间里，我一直在探索专业地构建 ui 意味着什么，我不仅仅指为此获得报酬。

### 为什么我需要⚛️做出反应

当我构建第一个真正复杂的 UI 时——这是一个地图应用程序——我没有框架，没有编码标准，没有 transpiler，甚至没有构建步骤——我有 jQuery_。_

我清楚地记得，在被告知客户对我完成的工作非常满意之后，我坐在那里*(当然非常重要)*，但不知何故，我仍然觉得这个应用程序不是我最好的作品。随着几个月的过去，维护和新特性被要求，从项目中挤出新生命变得更加困难。

我仍然在 JavaScript 中摸索，并感到不安，因为我认为必须有更好的方法来做我正在做的事情。也许它只是需要能够分割文件和创建导入/导出模块。也许我可以使用 Lodash 这样的库来帮助完成一些繁重的工作。

我现在会使用一大堆工具和技术来使应用程序完全不同，但也许最大的变化是简单地从执行一组步骤，一个接一个地执行并期待某个结果，变成声明当应用程序处于某个状态时 UI 应该是什么样子。

我知道你在想什么，我刚刚描述了反应，但我相信它比那更大。你不需要开始用这种方式思考——它让这个想法流行起来。

当我构建地图应用程序时，总是按照一定的顺序执行一组特定的步骤来产生结果(而且错误百出！).这从根本上说是构建 ui 的错误方式。它假设我们确切地知道用户将采取哪些步骤。按照一套特定的步骤，你不得不再造多少个 bug？全部吗？

你可能会说也许我的代码是狗屎，也许确实如此——但是我认为去掉这些步骤并用“这就是它应该有的样子和行为”来代替它们会使 JavaScript 应用程序更容易理解和维护。

### 你不需要做出反应

这不是我现在写的花哨的 ES6 代码，而是我考虑将复杂的逻辑分解成小的离散函数的方式，这些函数有合理的输入和可测试的输出。这些不是 React 发明的概念。

在过去的一年里，我是 React 的忠实用户，并且阅读了关于函数式编程原则的书籍——这两者在 JavaScript 社区中都非常流行。我有机会用 React 开发不同类型的应用程序，一些小组件，有时是整个用户流，我可以诚实地说，它改变了我对编程的总体看法。

它完全颠覆了前端开发人员的思维模式，甚至导致了疲劳，但如果你像我一样，责任并不止于此。

### 如何让碎片吻合

为了让这一切真正深入人心，我选择以这种方式思考我所构建的一切。

当我创建一个函数时，它应该描述输出，只要看看函数。我不是指文档块*(尽管你仍然应该写它们)*而是确保你的函数:

*   接受有限数量的参数，可选或不可选
*   基于这些输入返回输出
*   不需要来自功能之外的任何东西，也不影响外界
*   通过函数和参数的命名提供含义。

我很想让 React 继续存在一段时间，但我也很高兴下一个工具或框架能让每个人都大吃一惊，并以不同的方式看待事物。如果人们不开发工具和框架，不在网上发布他们的想法，就没有进步。

在一天结束时，像你一样，我生活在现实世界中，事情必须在商定的时间内完成，但我不认为这是对应用程序或 React 组件结构进行任何较少考虑的借口。

### 代码是给人类阅读的

不是你写代码的方式，而是别人怎么读。

最终，其他人必须阅读你的代码，无论是代码审查还是仅仅修改文件。所以善待其他开发人员，当事情变得有点糟糕时，添加代码注释。

当您提交代码时，您应该能够确信其他人能够捡起它并理解它。这就是我认为我们应该在代码质量方面努力的方向。

> 任何傻瓜都能写出计算机能理解的代码。优秀的程序员编写人类能够理解的代码。"–Martin Fowler，重构:改进现有代码的设计。
