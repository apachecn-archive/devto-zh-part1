# 我们是如何从 Bootstrap 成长起来的

> 原文：<https://dev.to/cwardzala/how-we-grew-out-of-bootstrap-4b3n>

当我们开始重新设计 Cheezburger 网络时，我们面临着构建前端框架的艰巨任务，不仅要满足开发人员的需求，还要满足我们庞大的网站网络的需求。经过多次会议、激烈的讨论和 DIY 尝试，我们最终选择了[twitter] [Bootstrap](http://getbootstrap.com) 。它是“It”框架，刚刚达到 2.0 里程碑，为什么不呢？

我们不知道如何把 bootstrap 做成 Cheezburger。我们最初试图通过覆盖我们自己的 chzboot 库中的样式来“扩展”Bootstrap。这被证明是不太理想的，因为我们要为所有想要扩展的东西复制代码。最终，我们将我们的更改与 Bootstrap core 合并，并放弃了任何升级到新版本的途径。

我们自己的伪 Bootstrap 分支一直运行良好。然而，随着我们设计的改变和成熟，我们已经开始从 Bootstrap 核心组件中迁移出来，构建我们自己的组件。当我们需要快速的 UI 处理时，有 Bootstrap 作为后盾是很好的，但是现在我们有了自己的风格指南和关于使用什么组件以及如何使用它们的文档。如果某样东西不在风格指南中，那么我们需要构建它，并将其添加到以后可以重用的东西的桶中。

在维护现有的基于引导的组件的同时开发我们自己的组件变得很困难。在 Bootstrap 提供的东西和我们站点的需求之间找到平衡点，是有效地驱使我们不再继续使用 Bootstrap 作为我们的核心基础的一件事。我们已经讨论过从我们的代码中完全删除任何引导程序并重写组件，但这是一个目前不符合逻辑的任务。最终，随着我们代码的不断成熟，引导程序将不再扮演重要的角色。

最终，我们脱离了 bootstrap，不是因为它是一个糟糕的框架，而是因为我们长大了，我们开始需要不在其核心的东西。我们做了太多的改变，继续保持与核心库的连接已经没有意义了。我们保留了 bootstrap 使用的许多命名和结构约定，但是我们不依赖它的 UI 元素库。

我没有提到的一件事是，当我们决定使用 Bootstrap 而不是创建我们自己的产品时，我极力主张使用 Bootstrap，因为我害怕“重新发明轮子”。通过这次经历，我开始相信你不应该害怕重新发明轮子，你应该害怕在需要方轮子的时候使用圆轮子。

我不想为 [@b_sted](http://twitter.com/b_sted) 、 [@marthakelly](http://twitter.com/marthakelly) 或 [@trademark](http://twitter.com/trademark) 代言，但这是我在重新设计时最大的恐惧。
