曾几何时，休息是一场革命。一个新的世界的前提是肥皂问题将被洗掉。(我在玩文字游戏时走上了一条很滑的路，不是吗？)REST 现在是一个事实上的标准，开始失去势头(或者可能已经失去了很多)。对于我们中的一些人来说，它仍然在我们的脑海中占据着特殊的位置，那就是我们的初恋。在这段旅程中，我们发现了有效的互操作性。对于其他人来说，从开发开始，它代表了一种不再属于先锋的标准，一种来自过去的东西，一种遗产，一种他们不想投入时间的东西——它就在那里。

是的，除了 REST 和 JSON 中的 web 服务，现在还有很多其他的解决方案。甚至有多种 REST 理解，我已经知道通过写这篇文章，我可能会引发一场关于什么是真正的 REST 架构的书呆子战争。人们会开始拿出罗伊·菲尔丁 2000 年的论文作为证据。老实说，我也为这种想法努力过，但那个时代已经过去了。在这个行业工作了这么多年后，我认为自己只是一个开发人员，一个普通的开发人员。所以，我不会假装我写的是事实。我会写下我在过去的几年里，在实践中，在工作中，在外面所做的和所见的。

**休息 101**

如果我需要用一句话向我的亲戚解释 REST，我会说，“这是让应用程序一起说话的一种方式”，不多也不少。如果我想更进一步(知道我在这个过程中丢失了它们)，我会补充说，“这是一种在多个应用程序之间构建对话的方式。”就像乐队指挥期望第一小提琴在他的左边，大提琴在他的右边；作为一名开发人员，您希望 REST 架构能够遵循一些指导方针来快速找到您想要的东西。

当你写一个应用程序时，你的目标是解决你的用户的问题，无论他们是感到无聊(所以你写游戏)，不知所措(所以你写提醒，日历，待办事项)，有创造力(嗯，你有这个想法)，等等。有时候，应用程序本身也有自己的问题需要解决(例如，从一个地址获取 GPS 坐标)。有些时候，应用程序能够解决其他应用程序的问题(例如，您编写了一个语法检查程序、一个单词计数程序、一个字典或一本词典)。因此，在某种程度上，您可能希望您的应用程序向另一个应用程序请求服务，或者您可能会诱使另一个应用程序从您的应用程序获得答案。有许多方法可以让您在两个不同的(并且距离很远的)应用程序之间进行通信，甚至有专门介绍这方面的书籍。休息是其中之一。

REST 是技术和标准的结合。当我们说休息时，我们通常会想到以下几点:

*   HTTP，
*   JSON，
*   还有…就是这样！

可能会更糟，对吗？

所以从根本上说，这就是休息。

**HTTP，这个东西大家都用**

当我告诉我的朋友(嗯，那些不从事 IT 工作的人)他们几乎每天都在使用一个 HTTP 客户端，并且他们甚至有一个最喜欢的客户端时，他们往往难以置信。

"是的，当然，我更喜欢这个，不管你说的是哪个客户."当他们意识到自己确实有偏好时，他们通常会感到震惊，因为一个普通的网络浏览器*是一个 HTTP 客户端。当我向他们展示他们可以查阅页面背后的代码时，让他们感到惊讶(甚至更惊讶)的是幕后发生的事情。这个互联网浏览器是一个 HTTP 客户端，它的唯一目的是将 HTML、CSS 和 Javascript 转换成漂亮的页面(或者如果我是设计者，就只是页面)。*

因此，在某种程度上，每个人都知道 web 服务—您的机器(客户端)使用 HTTP(协议)与另一台机器(服务器)通信，因此服务器发送显示在您的浏览器窗口(您的应用程序的功能)中的 HTML(响应)。

是的，REST 使用与您的浏览器完全相同的协议。但是在我们谈论这些区别之前，让我们先来谈谈反应本身…

JSON，一个人类可读的消息

当计算机交换信息时，它们不期望人类阅读它们的原始对话。这种假设有其优势，因为人脑无法理解二进制文件。令人惊讶的是，这种神奇的非人工智能，即我们的大脑，如果不是用字符链(多么原始)来表达，就无法解释信息。但是人类，这些顽固的生物，有时会发现能够阅读交换是很方便的，JSON 是一种人类(甚至初学者)能够理解的文本格式。这也是最容易阅读的书之一。

假设我可以访问位于 */service/reservations* 的 web 服务，在那里我可以看到我最喜欢的餐馆的预订情况。

响应应该包含预订日期、时间和数量。这正是典型的 JSON 响应中的内容。
