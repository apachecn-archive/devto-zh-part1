## 自举

19 世纪，实用主义和符号学之父查尔斯·桑德斯·皮尔士正在为心灵哲学中一些非常深奥的问题而苦苦挣扎。其中两个问题是:一个有缺陷的/有限的大脑如何让自己变得更聪明？当一种语言还没有表达抽象概念的词汇时，它是如何发展起来的呢？？

皮尔斯对这两个问题的回答是，我们可以自举。例如，如果我们有基本的读写能力，我们可以阅读超出我们现有水平的书籍，并学习新的词汇和概念。我们用这些新词汇去读更多的书，无穷无尽。同样，给定足够多的具体图标和索引(很容易学习)，我们可以用它们定义抽象概念，并指定任意符号。因此，我们可以发明新词，向他人解释它们，并用它们来抽象地说话。

我们从一组有限的功能开始，并在此基础上构建一组扩展的功能。我称之为*自举心态*。我喜欢 Lisp 的原因之一是它是自举思维的一种表现。这并不是 Lisp 独有的，但那是我[第一次看到它在编程中的应用](https://dev.to/ericnormand/the-idea-of-lisp)的地方。

简单总结一下上一篇链接文章，Lisp 从一些很容易用机器码定义的原语开始。你用这些原语构造一个解释器。现在你有了一个可执行的 lambda 演算。这个过程的杠杆作用是惊人的。

让我换一种说法。假设你可以用 10，000 行机器代码写一些算法。太好了！您也可以用 1000 行机器码编写一个小的 Lisp 实现。然后你可以用 Lisp 的 10 行代码写出你的算法。给定一个足够大的项目，你应该构建自己的工具。工具投资将获得数倍的回报。

你也可以在 [FORTH](https://github.com/AlexandreAbreu/jonesforth/blob/master/jonesforth.f) 中看到这种现象(构建一小组单词和一种定义新单词的方法)和 Smalltalk(构建一小组消息解释器和一种定义新解释器的方法)。

在所有这些例子中，我们都有一个通过使用词汇表本身来扩展的“词汇表”。一个很好的例子和解释是[发展一门语言](https://www.youtube.com/watch?v=_ahvzDzKdB0)。盖伊·斯蒂尔探索了如何创造一种不受我们现有理解限制的语言的想法。这门语言的未来潜力是无限的。

在我看来，**图灵机是自举引擎**。一旦你有了图灵完备性，你就可以解释任何语言。区别在于表现力。哪些引导步骤会让你发现你的问题微不足道？

在所有这些例子中，我们发现表现力有了巨大的飞跃。我们希望构建足够强大和足够健壮的东西，以便很少(如果有的话)处理较低的级别。我可以整天写 Clojure 代码，从来不考虑 JVM 字节码，更不用说实际运行的机器码了。你最后一次调试代码的与非门是什么时候？

下面是我对自举的两个要求。我称他们为:

1.  **杠杆**:表现力的飞跃。
2.  **稳健**:很少和下级打交道。

如果你没有这两样东西，它可能会非常有用，但它不是自举。

## 网络

这是我最初的担忧:我们能在网络上自举吗？我相信我们非常需要它。我们每年在自己发明的最愚蠢的问题上花费数十亿美元。我厌倦了在如此低层次工作的单调乏味。是的，网络是世界上一股巨大的经济力量，但是它产生的垃圾工作数量惊人。

我举一个我说的“垃圾工作”的例子。我们拥有非常强大的图形设计工具，如 Adobe Illustrator 和 Sketch。我们可以自由地画画，把东西做得和人类一样漂亮。即使我们将我们的工作限制在只有在网络上可行的范围内，我们也要手动将这个设计编译到尖括号中。这是一项专业技能，通过会议、书籍、行业杂志和培训，能够将一个设计变成你的浏览器可以使用的东西。我尊重做这件事的人，因为这很难。但是我不尊重人们浪费时间学习让你的设计在浏览器中工作的魔法咒语的需要。事实上，我们没有看到一个伟大的工具，直接输出 HTML + CSS 是一个坏兆头。

让我更深入地研究一下，因为我不想给人留下错误的印象。让东西在浏览器中工作是很重要的。但实际上让它工作比发现它应该如何工作要多得多，这是设计的实际工作。我们不能专注于重要的事情:把像素变成我们人性的表达。为什么不呢？因为我们无法让它正确渲染。我们讨论的是不重要的细节，比如在 CSS 中选择元素的最佳方式。你应该增加一个新的类吗？还是使用嵌套选择器？没有好的答案，因为它们都很烂，所以我们为之争吵，称之为“哲学”。或者我们等待委员会在标准中加入更多的东西，这仍然不能解决问题。

让我换一种方式来说:**人们用比普通 PHP 页面**更少的代码行编写了定制的布局引擎。在与浏览器的斗争中，我们 lob 包装 div 和 CSS 规则。

我并不是在抱怨编写 HTML、CSS 和 JavaScript 是一种拖累。机器码也是一种累赘。但我可以通过自举来转义机器码。我要说的问题是:我不相信我们知道如何从网络中引导。如果可以，我们会让它更有活力，注入我们的价值观。

既然我已经彻底失望了，并且已经充分定义了问题和我要找的东西，我想如果我回顾一下我在搜索中找到的东西，对志同道合的搜索者来说会很有用。

## 明显无解

这些项目很棒，但不是我想要的。

### Flexbox

Flexbox 很棒，但这不是我最终停止编写 CSS 的最后一件事，因为我已经对它进行了抽象。相反，它只是让使用 CSS 的生活更加舒适。

### [自举](http://getbootstrap.com/)

是的，多么贴切的名字！Bootstrap 承诺:使用他们的 HTML，CSS 就可以工作了。由于分析是相同的，我还将在这个标题下加入所有类似的框架。

这不是我想要的全部解决方案。这有点像给某人一顶帐篷来解决他的住房抵押贷款问题。您仍然在处理 HTML、URL 和所有其他内容。不过，对于快速的网络项目，我还是会求助于它。

### [轨道上的红宝石](http://rubyonrails.org/)

记得第一次看到[15 分钟视频](https://www.youtube.com/watch?v=Gzj723LkRJY)建博客的时候。哇！我太兴奋了。那时候，15 分钟是闻所未闻的。

但是 10 年后，我无意中听到 Rails 程序员抱怨什么:如何编写他们的模板，如何编写他们的 CSS，如何设置他们的路由，等等。换句话说，他们正在使用强大的工具让网络更加舒适。他们没有逃脱。这里再扔点其他知名框架，包括[基座](https://github.com/pedestal/pedestal)、[路明斯](http://www.luminusweb.net/)、[亚达](https://github.com/juxt/yada)。

### [花园](https://github.com/noprompt/garden)

已经有几个人用关于 Garden 的 tweets 回复了我的搜索，Garden 是一个用 Clojure 生成 CSS 的库。虽然它确实让您能够使用强大的逻辑来输出 CSS(高杠杆)，但我看不出它让您能够从 CSS 中引导出来(低健壮性)。例如，你能给我一个使用 Garden 居中 div 的通用解决方案吗？您仍然在处理 CSS 的复杂语义。

## 满怀希望的局部解决方案

### [霍普伦](https://hoplon.io/)

Hoplon 是一个 ClojureScript web 框架。它做了几件有趣的事情。

1.  它将 DOM 元素变成函数，这样它们就可以像常规函数一样在代码中使用。
2.  它让你的元素像电子表格一样对变化做出反应。
3.  它使您的客户机-服务器通信看起来像函数调用。

原来这三个功能加在一起帮助很大。它提供了一个很好的基础。让我们看看这个项目将走向何方。

### [霍普隆 UI](https://github.com/hoplon/ui)

我正在关注这个项目。前提很简单:如果你只是将所有的 div 包装在另外两个 div 中(无论如何你可能已经用`wrapper`、`container`和`inner`div 做了很多)，你可以用一点 JavaScript“解决”布局。这些包装 div 可以通过多几行代码自动生成。事实证明，这比你手写的 div 要少！我还不太明白，但这有点令人兴奋。我很自豪这直接来自 ClojureScript 社区。

### [网格样式表](http://gridstylesheets.org/)

使用食火鸡约束求解器求解布局。这是一个自定义布局引擎，比 CSS 更具表现力。它有一个声明性的语言来建立约束。它肯定有自举思维。您仍然生活在 HTML + CSS 中，但是它的行为更像您所认为的那样。

### [花粉](http://docs.racket-lang.org/pollen/)

这个项目让我肃然起敬。它制作了[实用印刷术](http://practicaltypography.com/)，一本漂亮的在线书籍。这些页面看起来像是由经验丰富的专业网页设计师手工调整的 HTML。作者使用输出到 HTML + CSS 的 Racket(一种 Lisp)编写了自己的创作格式。它肯定有自举思维。它强调了编写自己的工具的价值，也许问题在于我们没有剥离足够多的堆栈层。

### [做出反应](https://facebook.github.io/react/)

我在这里提到 React 是因为它的理念和潜力。它设法找到了一个高杠杆的抽象来引导容易出错的 DOM 操作。这是网络的一个严重问题。同样的抽象也被用在“本地”平台上，比如手机和台式机。有了 React on the web，你仍然生活在 HTML 和 CSS 的土地上，但这比手动设置`innerHTML`要好得多。

### [从安息到 CQRS 与克洛居尔、卡夫卡、&同归于尽](https://www.youtube.com/watch?v=qDNPQo9UmJA)

这只是 2015 年在 Clojure/conj 的一次演讲。他说的是他正在使用的堆栈，但其中嵌入了一个重要的引导程序。他在网上启动了一个通讯协议。他定义了一个接受 POSTs 的端点和一个服务器发送事件端点。POST 端点用于发送命令(捕获用户的意图),而服务器发送事件端点用于来自服务器的更新。就是这样。不再摆弄网址和“路线”。*这个*是真正的自举。高杠杆和高稳健性。

## 结论、一线希望和恳求

自举是一个美丽的现象。除了我们目前有限的能力，我们可以超越我们目前的限制。这是我们人性的精彩表达。然而，我们是否在我们的网络应用上投资了人性？或者我们在括号里辛苦工作，修复我们的 CSS，选择如何命名我们的包装 div，解决我们制造的问题？

自举的目的是一劳永逸地解决这个问题，这样我们就可以专注于重要的事情。我们可以走很长一段路，或者我们可以“用想法、情感、幽默和温暖来填充(我们的创作)。”(引自[扭转预期下降的趋势](http://unitscale.com/mb/reversing-the-tide/)，Matthew Butterick，错字柏林 2012。)为了世界，也为了我们自己，我们比以往任何时候都更需要这样做。

我们生活在一个有趣的时代。我已经等了很久，想看看 Clojure(或者任何 Lisp，真的)将如何超越 web。它有这么多伟大的工具！社区理解这种自举思维。然而，我还没有看到自举的发生。我们能实现吗？我们能剥开这层皮，建造一个表达人性的引擎吗？我们想要什么样的表现力？我们的想象力是唯一的限制。
